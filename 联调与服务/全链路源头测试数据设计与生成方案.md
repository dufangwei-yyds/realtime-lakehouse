## 一、文档目标

这份文档用于输出“一期全链路联调”第一版源头测试数据，目标是：

1. 统一准备可贯穿 `ODS -> DIM -> DWD -> DWS -> ADS` 的源头数据批次
2. 让两大核心场景都能被一次性验证：
   - 全渠道实时库存可视与智能调拨
   - 爆款趋势识别与快速追单
3. 在主链路可跑通的基础上，顺带覆盖部分异常场景：
   - Schema 漂移
   - 迟到数据
   - 重复数据

本版坚持两个原则：

- 主链路优先：先保证链路可跑通、结果可落地、看板和接口可出数
- 异常增强适度：只放入少量异常样本，不在首轮联调中引入过多噪音

## 三、测试数据批次设计

### 3.1 批次信息

| 项目             | 值                                                                   |
| ---------------- | -------------------------------------------------------------------- |
| 批次编号         | `BATCH_20260403_V1`                                                |
| 业务日期         | `2026-04-03`                                                       |
| 主窗口           | `2026-04-03 10:00:00 ~ 2026-04-03 11:00:00`                        |
| 补充窗口         | `2026-04-03 11:00:00 ~ 2026-04-03 12:00:00`                        |
| 迟到数据回补时间 | `2026-04-03 12:30:00` 写入，事件时间回落到 `2026-04-03 10:xx:xx` |

### 3.2 设计原则

- 所有事实尽量围绕同一批 SKU 展开，确保可以在 DWD 层关联 `dim_sku_wide`
- 订单、库存、物流、社媒、行为围绕同一时段发生，方便核对分钟级和小时级结果
- 至少准备一条“健康库存”样本和一条“缺货风险”样本
- 至少准备一条“爆款升温”样本和一条“普通波动”样本

## 十、推荐执行顺序

### 10.1 主链路联调顺序

1. 写入 ERP 维度主数据目标：跑通 `cdc_m_* -> ods_m_* -> dim_sku_wide`
2. 写入 ERP 订单和库存事实目标：跑通 `ods_order_item_cdc`、`ods_inventory_txn_cdc`
3. 发送 App 行为流目标：跑通 `ods_kafka_app_click -> ods_app_click_analyze -> dwd_user_behavior_detail_rt`
4. 发送物流流和社媒流
   目标：跑通 `ods_api_logistics_track(_analyze)`、`ods_social_mention(_analyze)`
5. 启动 / 观察 DWD 作业
   目标：形成 6 张 DWD 明细表
6. 启动 / 观察 DWS 作业
   目标：形成库存主题、爆款主题聚合结果
7. 执行 ADS 刷新 SQL
   目标：形成 Dashboard 表、API 表
8. 最后再发送增强样本
   目标：验证 Schema 漂移、迟到、重复处理能力

### 10.2 联调建议

- 第一轮只跑主链路样本，不混入增强样本
- 每完成一层就做抽样核对，不要等到 ADS 才回头排查
- 若发现口径异常，优先回查上一层，而不是直接改最下游结果

## 十二、建议的抽样校验 SQL

### 12.1 校验 DIM 宽表

```sql
SELECT sku_id, spu_code, sku_name, brand_name, season, product_type, color_code, size_code
FROM dim.dim_sku_wide
ORDER BY sku_id;
```

### 12.2 校验订单与库存 ODS

```sql
SELECT order_id, line_no, sku_id, qty, sale_amount, order_status, channel_code
FROM ods.ods_order_item_cdc
ORDER BY order_id, line_no;

SELECT txn_id, sku_id, store_id, warehouse_id, biz_type, qty, stock_status, biz_time
FROM ods.ods_inventory_txn_cdc
ORDER BY biz_time, txn_id;
```

### 12.3 校验爆款源事件

```sql
SELECT sku_id, behavior, event_time
FROM dwd.dwd_user_behavior_detail_rt
ORDER BY event_time;

SELECT sku_id, like_cnt, fav_cnt, comment_cnt, event_time
FROM dwd.dwd_social_mention_detail_rt
ORDER BY event_time;
```

### 12.4 校验库存主题与爆款主题

```sql
SELECT stat_minute, store_id, warehouse_id, sku_id, atp_qty, available_qty, locked_qty, in_transit_qty
FROM dws.dws_inventory_atp_rt
ORDER BY stat_minute, sku_id;

SELECT stat_hour, sku_id, sale_amount_1h, order_cnt_1h, view_cnt_1h, click_cnt_1h, add_cart_cnt_1h, social_hot_1h, hot_score
FROM dws.dws_hot_sku_feature_rt
ORDER BY stat_hour, hot_score DESC;
```

### 12.5 校验 ADS 结果

```sql
SELECT stat_minute, sku_id, atp_qty, demand_qty, shortage_risk_flag, dispatch_priority_score, inventory_health_level, alert_reason
FROM ads.ads_inventory_dispatch_dashboard
ORDER BY stat_minute, dispatch_priority_score DESC;

SELECT stat_hour, sku_id, hot_score, hot_level, trend_label, rank_no
FROM ads.ads_hot_sku_dashboard
ORDER BY stat_hour, rank_no;
```

## 十三、第一版结论

这版“全链路源头测试数据”已经具备以下价值：

1. 能支撑从源头到 ADS 的一次性联调
2. 两大核心场景都有明确数据支撑
3. 主链路与异常链路已做分层设计，便于分步验证
4. 后续可以继续在此基础上扩展：
   - 更大批量压测数据
   - 多门店、多仓、多渠道样本
   - 退货、换货、逆向物流样本
   - 更复杂的社媒爆点与营销活动样本

下一步建议直接进入：

1. 先按本方案执行主链路样本
2. 跑通一次 `ODS -> DIM -> DWD -> DWS -> ADS`
3. 记录每层核对结果
4. 再补发增强样本做健壮性验证
