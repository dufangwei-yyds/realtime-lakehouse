### 6.2 库存数量业务逻辑说明

#### 6.2.1 基础口径

- 货品总量 = 销售数量 + 在库数量 + 在途数量 + 生产未回数量
- 库存总量 = 在库数量 + 在途数量

#### 6.2.2 在库数量

- 基于库存流动进出账计算实时库存在库数量。
- 计算时需减去买断数量。
- 参考逻辑：
  - `FA_STORAGE_FTP.C_STORE_ID` 关联 `C_STORE.ID`
  - `C_STORE.C_CUSTOMER_ID` 关联 `C_CUSINV_STATUS.C_CUSTOMER_ID`
  - `FA_STORAGE_FTP.M_PRODUCT_ID` 关联 `C_CUSINV_STATUS.M_PRODUCT_ID`
  - 若 `C_CUSINV_STATUS.CHGDATE`（转换时间）不为空，则视为买断；转换时间之后库存数量记为 `0`
  - 使用 `FA_STORAGE_FTP.STATE * FA_STORAGE_FTP.QTY` 求和得到在库数量

#### 6.2.3 在途数量

根据销售单、销售退货单、调拨单的出入库状态判断是否为在途状态。一般规则为：出库已提交、入库未提交时，记为在途。

1. 销售单 `M_SALE`
   - 判断区间：`OUTTIME` 至入库完成时间之间
   - 数量字段：`M_RETAILITEM.QTYOUT`
   - 关联关系：
     - `M_SALEITEM.M_RETAIL_ID` 关联 `M_SALE.ID`
     - `M_SALEITEM.M_PRODUCT_ID` 关联 `M_PRODUCT.ID`
     - `M_SALE.C_DEST_ID` 关联 `C_STORE.ID`
   - 买断逻辑判断同上

2. 销售退货单 `M_RET_SALE`
   - 判断区间：`OUTTIME` 至入库完成时间之间
   - 数量字段：`M_RET_SALEITEM.QTYOUT`
   - 关联关系：
     - `M_RET_SALEITEM.M_RET_SALE_ID` 关联 `M_RET_SALE.ID`
     - `M_RET_SALEITEM.M_PRODUCT_ID` 关联 `M_PRODUCT.ID`
     - `M_RET_SALE.C_STORE_ID` 关联 `C_STORE.ID`
   - 买断逻辑判断同上

3. 调拨单 `M_TRANSFER`
   - 判断区间：`OUTTIME` 至入库完成时间之间
   - 数量字段：`M_TRANSFERITEM.QTYOUT`
   - 关联关系：
     - `M_TRANSFERITEM.M_TRANSFER_ID` 关联 `M_TRANSFER.ID`
     - `M_TRANSFERITEM.M_PRODUCT_ID` 关联 `M_PRODUCT.ID`
     - `M_TRANSFER.C_DEST_ID` 关联 `C_STORE.ID`
   - 买断逻辑判断同上

#### 6.2.4 生产未回数量

- 取生产计划单未回数量。
- 状态范围：提交未结案状态。
- 涉及表：
  - `Y_REQUIRE_PLAN_ITEM`
  - `Y_REQUIRE_PLAN`
- 关联关系：
  - `Y_REQUIRE_PLAN_ITEM.Y_REQUIRE_PLAN_ID` 关联 `Y_REQUIRE_PLAN.ID`

### 11.2 今日遇到的问题与解决方案

| 序号 | 问题现象                                                                                                     | 根因分析                                                                            | 解决方案                                                                                                                                  | 最终结论                                                                   |
| ---- | ------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| 1    | Flink 窗口聚合报错：`The window function requires the timecol is a time attribute type, but is TIMESTAMP(3)` | DWD 表中的 `event_time` 仅为普通 `TIMESTAMP(3)`，不是带水位的事件时间属性           | 在 DWS 过渡阶段新增 `_wm` 版 DWD 表；随后正式将 DWD 核心表统一改为建表即带 `WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND` | DWS 事件时间窗口聚合必须基于带 `WATERMARK` 的 DWD 表                       |
| 2    | `dwd_order_trade_detail_rt_wm` 建表后执行 `INSERT INTO ... SELECT * ...` 报字段数不匹配                      | `_wm` 表结构按早期假设写法设计，和当前真实 DWD 订单明细 schema 不一致               | 按实际已验证的 DWD 订单表字段重建 `_wm` 表，使用真实字段：`order_id、line_no、user_id、series、sale_price、order_time、pay_time` 等       | 所有过渡表、下游表都必须以当前真实 DWD schema 为准，不能再沿用早期草稿结构 |
| 3    | `dwd_hot_sku_event_detail_rt_wm` 与源表不一致                                                                | 早期版本误写了 `store_id / ingest_time`，遗漏了 `user_id / note_id / partition_hh`  | 参照当前真实的 `dwd_hot_sku_event_detail_rt` 重建，主键、分区、`sequence.field` 全部按源表对齐                                            | 爆款事件归一化表属于多来源统一事件表，字段和分区必须严格对齐真实实现       |
| 4    | Flink 执行 `INSERT INTO doris_dws_inventory_atp_rt` 报错：`Failed to get response from Doris`                | Doris Sink 使用了错误的 `fenodes` 端口，文档早期写成 `9030`                         | 将所有 Doris Sink 的 `fenodes` 统一改为 Doris FE HTTP 端口 `8030`，并补充连通性排查清单                                                   | Flink Doris Connector 写入优先使用 `8030`，不是 MySQL 查询端口 `9030`      |
| 5    | Doris 创建物化视图报错：`replication num should be less than the number of available backends`               | 当前测试环境仅有 1 个可用 BE，而 MV 默认副本数为 3                                  | 在物化视图 `PROPERTIES` 中显式指定 `"replication_num" = "1"`                                                                              | 单 BE 测试环境下，表和 MV 都应按 1 副本编写                                |

### 11.3 本次沉淀出的开发规范

- DWD 正式版建表时直接补齐 `WATERMARK`，不要依赖长期存在的 `_wm` 过渡表
- DWS 开发前先校验 DWD 实际 schema，再编写 Doris Sink、窗口 SQL 和下游聚合模型
- Doris Sink 连接参数要区分清楚查询端口与写入端口
- 测试环境的 Doris 副本数配置要和可用 BE 数量保持一致
- Flink SQL Client 中尽量减少“窗口聚合临时视图 + 二次 Join”的拆分写法
- 所有修正结果都要同步回文档，保持“方案文档 = 可执行版本”

- 看板型 ADS 与接口型 ADS 必须拆分，避免一个结果集同时承担多种消费职责
- Superset 只描述如何消费 ADS 结果，不展开搭建过程
- 接口设计要同步考虑分页、排序、筛选、解释字段与统一返回结构
- 高频驾驶舱和榜单页面适合在 ADS 之上继续叠加 Doris 异步物化视图

   - 可回退
   - 不影响主链路
   - 有明确效果判断标准

### 18.7 二期启动前补充建议项

结合 [Fluss预研/Fluss二期试点建议.md](/d:/workspace/realtime-lakehouse/Fluss预研/Fluss二期试点建议.md) 与淘宝闪购 `Flink + Fluss` 生产实践带来的启发，当前建议在二期正式启动前，把 `Fluss` 的后续推进进一步收敛为以下明确建议项：

1. 明确 `Fluss` 在二期中的定位  
   继续保持“补充型试点能力”定位，不作为一期主链路替代方案。

2. 优先试点“实时库存状态层”  
   这是当前最推荐的切入点，因为：
   - 与现有项目贴合度高
   - `PrimaryKey Table` 价值最容易体现
   - 结果最易解释、最适合可回退试点

3. 第二优先试点“行为流富化层”  
   重点验证：
   - `Log Table + PrimaryKey Table + Lookup Join`
   - 是否能降低富化链路复杂度
   - 是否适合作为后续实时特征层前置

4. 若二期前置接入补齐，再评估“日志实时层”  
   前提是：
   - 埋点服务
   - `nginx` 日志
   - API 数据
   - 爬虫数据
   接入体系在二期逐步补齐。  
   届时可重点验证：
   - `Log Table`
   - 列裁剪
   - 自动分层

5. 二期试点必须满足四个条件  
   - 可回退
   - 不影响主链路
   - 有量化评价指标
   - 试点范围足够小且场景足够明确

6. 二期试点评价时重点看这些维度  
   - 建模复杂度
   - SQL / 作业复杂度
   - Flink State 压力变化
   - 运维复杂度
   - 是否真正减少胶水层

当前建议结论：

- `Fluss` 后续推进不宜“大而全”
- 应坚持“小步快跑、局部突破、可回退、可量化”的二期试点节奏
