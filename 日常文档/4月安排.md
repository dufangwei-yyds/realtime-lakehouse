1. `Fluss` 小范围试点
方向：
实时状态层、行为流富化层

2. `Doris AI` 小范围增强
方向：
社媒舆情增强、商品知识增强、相似商品语义检索

3. 继续完善真实生产数据质量与业务口径
方向：
补齐一期遗留的业务补充需求和测试数据问题

4. 进一步推动“数据服务 + 智能分析”融合
方向：
面向业务运营、商品管理、客服服务的智能增强场景

5. 扩展实时数据集成体系
方向：
在一期 `Flink CDC + Kafka` 验证基础上，继续补齐更完整的数据接入前置体系，例如：

- 埋点数据模型设计
- Python 埋点数据服务接入 Kafka
- `nginx` 服务日志接入
- API 接口数据接入
- 爬虫数据接入

6. 深化治理、监控与作业优化
方向：

- 持续完善实时湖仓数据治理体系
- 深化运维监控、质量告警、巡检和运行保障
- 持续优化现有 `Flink SQL` 作业的稳定性、性能与资源使用

7. 拓展平台与生态预研
方向：

- `DolphinScheduler`
- `DataHub` 元数据服务
- 结合 2026 年热门方向继续做技术预研与生态接入探索，例如 `OpenClaw` 等

8. 来自淘宝闪购 `Flink + Fluss` 生产实践的启发
- `Fluss` 二期试点优先从“实时状态层”和“行为流富化层”切入，而不是直接替换一期 `Paimon` 主链路
- 若二期补齐埋点服务、`nginx` 日志、API 数据、爬虫数据接入，则可以重点评估 `Log Table + 列裁剪 + 自动分层` 的价值
- 若二期继续加强多源行为宽表或状态宽表，可以重点观察 `Partial Update` 的收益
- 若未来实时归因、转化链路、长周期关联变重，可以进一步评估类似 `Delta Join` 的适配价值

### 基于 AI 能力重构实时湖仓的探索

#### 总体思路

AI 不应被孤立为“报表问答”能力，而应重构实时湖仓的三个核心环节：

- 智能建模
- 智能治理
- 智能分析与运营闭环

####  AI 落地方向

##### 方向一：智能建模助手

- 自动解析源表结构、生成 ODS/DWD 建模草案
- 生成 Flink SQL 模板、字段映射建议、维度归类建议
- 识别同义字段、重复表、潜在主外键关系

##### 方向二：智能数据治理

- 自动识别异常波动、缺失字段、漂移指标
- 基于规则 + 模型做数据质量根因分析
- 为字段补充释义、标签、血缘说明

##### 方向三：智能经营分析

- 面向 Doris 4.x 验证自然语言问数能力
- 构建“爆款原因解释”“库存异常解释”“履约异常解释”能力
- 支持运营人员通过问答方式获取趋势洞察和建议

##### 方向四：智能调度与运维

- 对 Flink 作业异常自动归因
- 对 Kafka 积压、状态膨胀、写入失败给出优化建议
- 自动生成巡检日报与发布风险提示

#### AI 建设原则

- 先辅助，再自动化
- 先规则增强，再模型增强
- 高风险动作必须有人审
- 所有 AI 输出都要可追溯、可解释、可回滚
