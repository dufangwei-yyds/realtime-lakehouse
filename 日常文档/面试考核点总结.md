一、Flink 核心考察点（实时计算核心）
1. 基础原理（必问，考察功底）
编程模型：DataStream 与 Table API 的区别？何时选择哪种 API？（如服装订单实时清洗用 DataStream，聚合统计用 Table API）
状态管理：KeyedState/OperatorState 的适用场景？状态后端（Memory/RocksDB）选型依据？（如大促高并发场景必选 RocksDB）
容错机制：Checkpoint 与 Savepoint 的核心差异？Exactly-Once 语义如何实现？（结合 Chandy-Lamport 算法、两阶段提交）
时间语义：EventTime/ProcessingTime/IngestionTime 的区别？Watermark 生成策略（周期性 / 断点续传）？迟到数据如何处理？（如直播订单延迟数据的侧输出流处理）
窗口机制：滚动 / 滑动 / 会话窗口的适用场景？（如 5 分钟直播 GMV 统计用滚动窗口，30 分钟用户互动统计用滑动窗口）
2. 业务落地（结合服装行业）
如何用 Flink 实现服装直播的「实时 GMV 统计」？需考虑哪些细节？（如退款订单冲减、优惠券分摊、跨直播间合并）
服装库存实时预警场景：如何通过 Flink 监控库存低于安全阈值的商品（含尺码 / 颜色维度）？如何避免重复告警？
订单履约跟踪：如何处理分布式事务（如订单支付、库存扣减、物流创建的原子性）？是否用过 Flink XA 事务或 TCC 模式？
3. 性能优化（核心区分点）
并行度优化：如何根据 Kafka 分区数、CPU 核心数设置 Flink 并行度？（如 Kafka 主题 8 分区，Flink 源算子并行度设为 8）
状态优化：RocksDB 调优（write buffer 大小、压缩算法、前缀编码）？状态过期策略如何设置？（如服装历史订单状态保留 30 天）
反压处理：Flink 反压的检测方式（WebUI 指标、Metrics）？如何定位反压源头？（如聚合算子 bottleneck，可拆分算子或增加并行度）
算子优化：算子链（Chain）的开启 / 关闭场景？Broadcast State 适用场景（如服装商品维度表广播）？
4. 故障排查
Checkpoint 失败：常见原因（资源不足、状态过大、依赖外部系统超时）？如何排查？（查看 Checkpoint 日志、分析 barriers 对齐情况）
数据延迟：如何通过 Flink WebUI 定位延迟节点？（如 Watermark 滞后、算子处理耗时过长）
状态膨胀：如何检测状态大小？如何优化？（如拆分状态、使用增量 Checkpoint、状态 TTL）
数据不一致：Exactly-Once 语义失效的场景？（如外部系统不支持幂等写入，需结合 Paimon 的 Upsert 机制）
5. 进阶拓展

与其他计算引擎对比：Flink vs Spark Streaming（延迟、吞吐量、容错）？为何在服装直播场景选择 Flink？

3. 性能优化（面试官重点追问）
执行计划优化：
如何用 EXPLAIN PLAN FOR [SQL] 分析执行计划？重点看是否「谓词下推」（如过滤条件推到 Kafka/CDC 源端）、「列裁剪」（只读取需要的字段）。
反例：未下推的过滤会导致全量数据拉取后再过滤，服装大促时会严重拖慢速度。
窗口优化：
窗口类型选择：滚动窗口（5 分钟 GMV）、滑动窗口（30 分钟销量）、会话窗口（用户直播间停留）的性能差异？
迟到数据优化：ALLOWED_LATENESS 设置（如 TUMBLE(pay_time, INTERVAL '5' MINUTE) WITH ALLOWED_LATENESS INTERVAL '1' MINUTE），避免窗口长期占用状态。
Join 优化：
广播 Join：服装直播间维度表（数据量小）优先用广播 Join（/*+ BROADCAST(p) */），避免 Shuffle 开销。
Temporal Join 优化：维度表设置 TTL，避免状态无限膨胀（如商品维度表保留 7 天变更记录）。
函数优化：
优先使用内置函数（如 DATE_FORMAT），自定义 UDF/UDAF 需注意：避免频繁创建对象（如在 eval 方法外初始化）、UDTF 减少数据膨胀。
示例：服装尺码转换 UDF（如 S/M/L → 165/170/175），需保证线程安全。
状态优化：
显式设置状态 TTL：SET table.exec.state.ttl = '86400 s'（订单数据保留 1 天），避免状态膨胀。
状态后端适配：Flink SQL 默认使用 Job 配置的状态后端，大状态场景需指定 RocksDB。
4. 故障排查（实战能力体现）
语法 / 执行错误：
常见问题 1：水印定义错误（如 WATERMARK FOR ts AS ts，无延迟会导致迟到数据直接丢弃）→ 修正为 ts - INTERVAL 'n' SECOND。
常见问题 2：主键冲突（写入 Paimon/Doris 时，同一主键多次写入）→ 确认 write-mode = 'upsert'，或调整聚合逻辑。
常见问题 3：类型不匹配（如 Kafka 字符串数字 → Flink DECIMAL）→ 用 CAST 显式转换（CAST(pay_amount AS DECIMAL(10,2))）。
数据延迟 / 积压：
排查步骤：① 看 Flink WebUI 的「Watermark 滞后」→ 确认源数据时间是否乱序；② 看算子处理耗时 → 优化 Join / 聚合逻辑；③ 看并行度 → 调整为与 Kafka 分区数匹配。
示例：服装大促时订单 SQL 作业延迟，排查发现是窗口聚合算子并行度不足，调整为 Kafka 分区数（8）后延迟从 30s 降至 5s。
数据不一致：
原因 1：Exactly-Once 未生效 → 确认 Sink 表开启幂等 / 事务（如 Paimon 两阶段提交）。
原因 2：状态恢复异常 → 用 Savepoint 恢复作业，检查状态数据完整性。
资源不足：
表现：TaskManager OOM → 优化状态 TTL、减少窗口数据量，或增加 TaskManager 内存。

2. 服装行业架构选型核心考察点
核心问题 1：你在服装直播实时湖仓项目中，选择了哪种架构？为什么？（需结合业务需求阐述）
参考回答：选择「湖仓一体架构」，理由：① 服装直播需实时 GMV 统计（延迟 < 5 秒），同时需支持历史订单回溯（如 7 天内退款冲减）；② Paimon 支持明细数据的增量写入 + 版本回溯，Doris 支持多维度聚合查询（如按品牌 + 尺码统计销量）；③ 流批一体开发（Flink SQL 同时处理实时流 + 离线批数据），降低开发成本。
核心问题 2：服装行业大促（如双 11 直播）时，架构如何应对流量峰值？（考察架构弹性）
关键思路：① 数据接入层：Kafka 分区扩容（峰值是日常 3-5 倍）；② 计算层：Flink 作业动态扩并行度（结合 K8s 弹性伸缩）；③ 存储层：Paimon 冷热分离（热数据存 SSD，冷数据存 HDFS）；④ 服务层：Doris 物化视图预计算，避免大促时全表扫描。
核心问题 3：如何平衡「实时性」与「数据一致性」？（架构设计的核心矛盾）
解决方案：① 写入层：Flink + Paimon 两阶段提交（Exactly-Once 语义）；② 读取层：Doris 通过 Paimon 外部表读取时，指定「快照版本」，避免读取未提交数据；③ 维度表同步：Flink CDC 实时同步商品 / 用户维度表，Temporal Table Join 保证关联一致性。
核心问题 4：架构如何支持「多业务场景」？（如直播 GMV 统计、库存预警、订单履约跟踪）
分层设计思路：① 数据接入层：按业务域拆分 Kafka 主题（order_realtime、inventory_change、live_interaction）；② 计算层：Flink 作业按场景拆分（独立作业处理 GMV、库存、履约），避免相互影响；③ 存储层：Paimon 按表类型存储（订单明细 - Append 表、库存快照 - Upsert 表）；④ 服务层：Doris 按场景创建视图（GMV 视图、库存视图），支持业务按需查询。

2. 核心选型争议与面试高频追问
追问 1：为什么不用 Hudi/Iceberg 替代 Paimon？（考察组件深度认知）
回答思路：① 集成性：Paimon 是 Flink 生态原生项目，写入性能比 Hudi/Iceberg 高 30%+（服装大促时更适配）；② 运维成本：Paimon 元数据管理更简洁，无需额外依赖 Hive Metastore；③ 功能适配：Paimon 的 Upsert 模式更高效（库存实时更新场景），而 Hudi 更适合 CDC 增量同步场景。
追问 2：Doris 与 ClickHouse 怎么选？（考察 OLAP 场景适配）
回答思路：① 多表 Join：服装行业需关联订单、商品、直播间多表，Doris Join 性能优于 ClickHouse；② 数据更新：Doris 支持实时 Upsert（GMV 统计冲减），ClickHouse 更新需通过 ReplacingMergeTree，延迟高；③ 并发支撑：运营团队多用户同时查询报表，Doris 的 FE/BE 架构更适合高并发，ClickHouse 更适合单表高吞吐查询。
追问 3：数据采集为什么不用 Debezium+Kafka Connect？（考察采集组件对比）
回答思路：① 架构复杂度：Flink CDC 无需依赖 Kafka Connect，减少组件层级（MySQL→Flink CDC→Kafka）；② 全量同步：Flink CDC 支持无锁全量同步，Debezium 全量同步需锁表（服装订单表千万级数据，锁表会影响业务）；③ 计算集成：Flink CDC 可直接在 Flink 作业中解析变更数据（如商品 SCD2 处理），无需额外消费 Kafka 消息。
3. 选型风险规避（面试加分项）
组件兼容性：避免选择小众组件（如放弃 Pulsar 选 Kafka），减少生态适配成本；
运维成本：优先选择社区活跃、文档完善的组件（如 Flink/Doris/Kafka），避免自研组件；
业务扩展性：预留组件扩容接口（如 Kafka 分区预分配、Doris BE 节点横向扩展），应对服装品牌业务增长。
十一、集群规划与数据量预估（核心考察「实战落地能力」）
1. 数据量预估（面试必问：如何量化业务规模？）
（1）预估逻辑（以中型服装品牌为例）
核心维度：日活用户、订单量、峰值 QPS、数据保留周期、单条数据大小；
服装行业典型数据：
业务指标	日常规模	大促峰值（双 11 / 直播专场）	单条数据大小	数据保留周期
日订单量	50 万笔（含支付 / 退款 / 取消）	200 万笔（4 倍日常）	512B / 条	90 天
库存变更记录	100 万条 / 日（尺码 / 颜色维度）	500 万条 / 日（5 倍日常）	256B / 条	30 天
直播互动数据（点击 / 观看）	500 万条 / 日（单直播间峰值 10 万 QPS）	2000 万条 / 日（4 倍日常）	128B / 条	7 天
商品维度表数据	10 万条（含尺码 / 颜色子维度，共 50 万条）	15 万条（新增商品）	1KB / 条	永久保留
（2）存储容量预估（核心公式：总容量 = 日数据量 × 保留天数 × 冗余系数）
订单明细（Paimon 存储）：
日常日存储：50 万条 × 512B = 256MB → 90 天 × 256MB = 23GB → 冗余系数（3 倍，含副本 + 合并文件）→ 69GB；
大促日存储：200 万条 × 512B = 1GB → 90 天 × 1GB = 90GB → 冗余系数 3 → 270GB；
总存储：≈ 300GB（预留 20% 冗余 → 360GB）。
库存变更数据（Paimon 存储）：
日常：100 万条 × 256B = 256MB → 30 天 × 256MB = 7.68GB → 冗余 3 倍 → 23GB；
大促：500 万条 × 256B = 1.25GB → 30 天 × 1.25GB = 37.5GB → 冗余 3 倍 → 112.5GB；
总存储：≈ 130GB（预留 20% → 156GB）。
Kafka 消息存储（保留 7 天）：
峰值日消息量：2000 万（互动）+ 200 万（订单）+ 500 万（库存）= 2700 万条 → 2700 万 × 平均 300B = 810MB → 7 天 × 810MB = 5.67GB → 副本 3 倍 → 17GB；
总存储：≈ 20GB（预留 50% → 30GB）。
Doris 存储（聚合数据）：
直播 GMV 统计（5 分钟窗口）：200 万订单 / 日 → 200 万 / (24×12) ≈ 7000 个窗口 → 每个窗口 1KB → 7000KB ≈ 7MB / 日 → 90 天 × 7MB = 630MB；
多维度聚合表（品牌 / 尺码 / 直播间）：约 50GB（预留冗余 → 80GB）。
（3）峰值 QPS 预估
订单支付峰值 QPS：日常 500 QPS → 大促 2000 QPS（4 倍）；
库存变更峰值 QPS：日常 1000 QPS → 大促 5000 QPS（5 倍）；
直播互动峰值 QPS：单直播间 10 万 QPS（需按直播间拆分 Kafka 主题）；
Doris 查询峰值 QPS：运营报表查询 → 日常 100 QPS → 大促 300 QPS。
2. 集群规划（基于数据量预估的资源配置）
（1）Kafka 集群规划（消息总线）
节点配置：3 台物理机（或云服务器），每台 8 核 16GB 内存，1TB SSD（存储消息）；
核心参数：
分区数：订单主题（order_realtime）→ 16 个分区（峰值 2000 QPS ÷ 单分区 150 QPS ≈ 13，向上取整 16）；
副本数：3 副本（保证数据可靠性）；
日志保留：7 天（互动数据）、15 天（订单 / 库存数据）；
优化配置：开启 Snappy 压缩，batch.size=16KB，linger.ms=5ms（平衡延迟与吞吐量）。
（2）Flink 集群规划（实时计算）
部署模式：K8s 集群（弹性伸缩，应对大促峰值）；
集群规模：
JobManager：2 台（主从备份），每台 4 核 8GB 内存；
TaskManager：日常 6 台，大促弹性扩容至 12 台，每台 8 核 16GB 内存（其中堆内存 8GB，RocksDB 状态内存 4GB）；
核心参数：
并行度：订单处理作业 → 16 并行度（与 Kafka 分区数一致）；
Checkpoint：增量 Checkpoint，间隔 5 分钟，超时 30 分钟；
状态后端：RocksDB（开启前缀编码 + LZ4 压缩）；
作业配置：每个业务作业独立部署（GMV 统计、库存预警、订单关联），避免相互影响。
（3）Paimon 存储规划（数据湖）
存储介质：热数据（近 7 天）存 SSD（1TB），冷数据（7 天以上）存 HDFS（10TB 集群）；
核心参数：
分桶数：订单表 → 32 个分桶（按 order_id 哈希），库存表 → 16 个分桶（按 product_id+size 哈希）；
合并策略：Minor Merge 触发条件（10 个 Delta 文件），Major Merge（每日凌晨 1 点定时执行）；
版本保留：订单表保留 30 个版本，库存表保留 10 个版本。
（4）Doris 集群规划（OLAP 服务）
集群规模：
FE 节点：3 台（1 主 2 从），每台 8 核 16GB 内存；
BE 节点：6 台，每台 16 核 32GB 内存，2TB SSD（存储聚合数据 + 物化视图）；
核心参数：
分桶数：聚合表按 product_id 分桶（16 个分桶）；
物化视图：直播 GMV 表创建 3 个物化视图（按直播间、品牌、尺码维度预计算）；
查询优化：开启谓词下推、列裁剪，parallel_fragment_exec_instance_num=8（与 BE 核心数匹配）。
3. 面试高频追问与加分项
追问 1：大促时集群资源不足，如何应急处理？（考察风险应对）
回答思路：① 降级策略：关闭非核心作业（如直播互动数据统计），优先保障订单 / 库存核心流程；② 限流措施：Kafka 生产者限流（避免消息积压过多），Doris 查询限流（限制非运营用户查询）；③ 资源调度：K8s 集群将其他业务资源临时调度给实时湖仓作业。
追问 2：如何验证集群规划的合理性？（考察实战复盘能力）
回答思路：① 压测验证：用 JMeter 模拟大促峰值 QPS，测试 Kafka 吞吐量、Flink 处理延迟、Doris 查询响应时间；② 监控指标：观察集群 CPU 利用率（日常 <70%，大促 < 90%）、内存使用率（<80%）、磁盘 IO（<80MB/s）；③ 扩容阈值：当 Kafka 消息积压超过 100 万条、Flink Checkpoint 超时率> 10%，触发自动扩容。
加分项：
成本优化：冷数据迁移至对象存储（如 OSS/S3），降低 SSD 存储成本；
资源复用：Flink 作业共享 TaskManager 资源池，提高资源利用率；
容灾规划：跨可用区部署 Kafka/Doris 集群，避免单点故障（如华东仓与华南仓数据同步）。
总结：三大模块面试准备核心
架构选型：紧扣「服装行业业务需求」（实时性、多维度、峰值波动），说明架构分层逻辑，对比不同架构的优劣，体现「业务驱动架构」思维；
技术栈选型：每个组件都要讲清「选型理由 + 场景适配 + 组件对比」，避免只说 “选什么”，不说 “为什么”，体现「组件深度认知」；
集群规划：先做「数据量量化预估」，再基于预估结果推导资源配置，给出具体的计算逻辑和参数示例，体现「实战落地能力」。
