# Fluss预研规划

> 文档定位：服装行业实时湖仓项目二阶段预研规划文档  
> 适用阶段：一期项目结项后，进入 `Fluss` 轻量预研与 PoC 规划阶段  
> 关联项目：服装行业实时湖仓构建项目  
> 当前定位：不替代一期主生产链路，作为旁路 PoC 与演进选项进行验证

---

## 一、预研背景

一期项目已完成 `ODS -> DIM -> DWD -> DWS -> ADS` 全链路建设与联调验证，主生产技术路线已经稳定在：

- `Flink CDC / Kafka / Flink SQL & DataStream API / Paimon / Doris 4.x / Superset`

在主链路完成阶段性闭环后，项目进入“轻量预研与能力扩展”阶段。  
`Apache Fluss` 在本阶段的角色不是直接替代现有架构，而是作为新一代实时分析存储组件进行专项评估，判断其是否适合作为二期演进选项。

结合现有项目资料与官方定位，`Fluss` 适合重点观察以下能力：

- 面向实时分析与 AI 场景的流式存储能力
- 对实时明细写入、主键更新、低延迟读取的支撑能力
- 与 `Apache Flink` 的集成效率
- 作为实时数据层时，对当前 `Kafka + Paimon` 链路的补充价值

---

## 二、预研目标

本轮 `Fluss` 预研目标分为 4 类。

### 2.1 架构定位目标

- 明确 `Fluss` 在当前实时湖仓中的建议定位
- 判断其更适合作为“补充型实时数据层”还是“可演进替代选项”
- 明确它与 `Kafka`、`Paimon`、`Doris` 的边界关系

### 2.2 技术能力目标

- 验证 `Fluss` 对 `Append/Log` 类型数据的承载方式
- 验证 `Fluss` 对 `Primary Key` 更新流的承载方式
- 验证低延迟读取、实时订阅、Lookup 类访问的潜力
- 验证与 `Flink` 集成时的开发复杂度与稳定性

### 2.3 场景适配目标

- 评估其是否适合接入行为流实时特征场景
- 评估其是否适合承担热点库存/订单实时状态层
- 评估其是否适合作为面向 AI/特征服务的中间实时层

### 2.4 演进决策目标

- 给出“是否建议进入二期继续 PoC / 小范围试点 / 暂不推进”的结论
- 给出进入二期前必须补齐的技术验证项与风险项

---

## 三、与现有主链路的关系

### 3.1 当前主生产链路

- 数据接入：`Flink CDC + Kafka`
- 湖仓主存储：`Paimon`
- 明细加工：`Flink SQL / DataStream API`
- 查询与服务：`Doris 4.x`
- BI 展示：`Superset`

### 3.2 Fluss 在本阶段的定位

当前建议将 `Fluss` 视为：

- `Kafka + Paimon` 之间的“新型实时数据层候选”
- 面向“低延迟读写、实时特征、主键流式更新”场景的专项候选组件
- 面向二期扩展的技术储备，而不是一期主链路替换方案

### 3.3 当前不建议做的事情

- 不建议直接替换一期已经跑通的 `Paimon` 主明细链路
- 不建议直接承担一期正式 SLA 场景
- 不建议同时大规模改造 `ODS / DWD / DWS / ADS` 多层结构
- 不建议在预研初期就引入过多业务方依赖

---

## 四、官方能力与项目相关性判断

根据 `Fluss` 官方文档，其定位是“面向实时分析与 AI 的流式存储（streaming storage）”，支持：

- `Log Table`
- `PrimaryKey Table`
- 流式读写
- 低延迟实时读取
- 与 `Apache Flink` 集成

结合本项目场景，相关性判断如下：


| 能力方向             | 与项目的相关性 | 当前判断                   |
| ---------------- | ------- | ---------------------- |
| Log Table        | 高       | 可用于事件流、行为流、物流轨迹类数据     |
| PrimaryKey Table | 高       | 可用于实时状态、实时宽表、实时主键快照类场景 |
| 低延迟读取            | 高       | 可用于实时调拨与实时特征场景探索       |
| Flink 集成         | 高       | 是本项目最关键的技术前提           |
| AI 场景数据层         | 中高      | 后续可与 `Doris AI` 实验形成联动 |
| 大规模替代 Paimon     | 低       | 当前阶段不建议直接推进            |

## 五、预研范围

### 5.1 本轮纳入预研的范围

- `Fluss` 核心概念与组件角色梳理
- `Log Table / PrimaryKey Table` 使用方式梳理
- 与 `Flink` 的最小链路联通验证规划
- 与现有库存、行为、爆款场景的适配性分析
- 与当前 `Kafka + Paimon` 链路的对比项设计

### 5.2 本轮暂不纳入的范围

- 大规模性能压测
- 多机高可用生产部署
- 主生产任务迁移
- 长周期稳定性验证
- 面向业务正式交付的产品化方案

---

## 六、优先预研的 4 个问题

### 6.1 Fluss 最适合替代或补充当前链路的哪一段

重点判断：

- 是更接近 `Kafka` 的实时消息承载角色
- 还是更接近实时状态存储/实时特征层
- 还是更适合作为 `Flink` 的低延迟上游/下游表存储

### 6.2 Fluss 是否适合承担“主键更新流”的中间层

重点判断：

- 对订单状态、库存状态、商品实时状态等主键流数据是否友好
- 与当前 `Paimon latest / history / append` 模型相比，是否更适合某些“实时状态”类场景

### 6.3 Fluss 是否能提升“低延迟读取 + 实时 Join”体验

重点判断：

- 是否适合做高频 lookup
- 是否适合做实时服务层前置状态存储
- 是否适合做行为流、爆款流的特征暂存层

### 6.4 Fluss 与现有技术栈结合后会增加哪些复杂度

重点判断：

- 新组件引入后的部署复杂度
- 运维复杂度
- 监控体系改造成本
- 数据治理与血缘治理复杂度

---

## 七、建议的 PoC 场景

本轮建议优先选择 3 个轻量场景做 PoC 设计。

### 7.1 场景一：行为流实时特征层

候选数据：

- App 点击/浏览行为流
- 商品社媒热度事件流

验证目标：

- 验证 `Log Table` 写入与流式读取能力
- 验证行为明细进入后，是否适合快速沉淀实时特征
- 对比当前 `Kafka -> Flink -> Paimon` 的复杂度与时延

### 7.2 场景二：实时库存状态层

候选数据：

- 库存流水事实
- SKU / 店仓粒度库存状态

验证目标：

- 验证 `PrimaryKey Table` 对实时状态更新的适配性
- 验证是否可作为库存 ATP 场景中的中间状态层
- 观察实时更新、读取和幂等处理方式

### 7.3 场景三：爆款追单实时特征层

候选数据：

- 行为事件
- 社媒事件
- 订单明细事件

验证目标：

- 验证多事件流汇入时的实时聚合前置能力
- 评估其作为爆款追单特征实时层的可行性
- 为后续与 `Doris AI` 联动提供实时特征输入层思路

---

## 八、与现有架构的对比视角

### 8.1 建议重点对比维度


| 对比维度   | 当前主链路                   | Fluss 预研重点          |
| ------ | ----------------------- | ------------------- |
| 实时写入链路 | Kafka / Paimon          | 写入复杂度、时延、吞吐         |
| 主键更新支持 | Paimon latest / history | PrimaryKey Table 体验 |
| 实时读取   | Doris / Paimon / Flink  | 流式读、低延迟读、lookup     |
| 开发复杂度  | 已成熟                     | 需要评估学习成本与作业改造量      |
| 运维复杂度  | 已建立规范                   | 需要评估新增组件成本          |
| 数据治理接入 | 已有体系                    | 需要评估血缘、监控、SLA 衔接    |


### 8.2 预期判断方式

- 不只看单点性能
- 更看“是否值得引入到当前体系”
- 重点看“是否解决我们现有体系中某一段痛点”
- 如果只是能力相近但复杂度更高，则暂不建议引入

## 九、建议的验证路线

### 9.1 第一阶段：资料梳理与最小概念验证

目标：

- 看懂 `Fluss` 核心概念
- 明确组件部署与表模型
- 梳理与 `Flink` 集成的最小样例

输出物：

- `Fluss` 预研笔记
- 技术概念映射表
- 最小验证路线图

### 9.2 第二阶段：最小链路 PoC

目标：

- 跑通一个最小的 `Flink -> Fluss -> Flink/消费端` 示例
- 初步验证 `Log Table` 和 `PrimaryKey Table`

输出物：

- PoC 操作文档
- 最小场景验证记录

### 9.3 第三阶段：与现有场景结合验证

目标：

- 将库存状态或行为流特征中的一个小场景接入 PoC
- 对比当前实现与 `Fluss` 方案在复杂度、时延和可维护性上的差异

输出物：

- 场景对比报告
- 是否建议纳入二期试点的结论

## 十、关键评估指标

### 10.1 技术指标

- 写入时延
- 读取时延
- 主键更新正确性
- 重复数据处理表现
- 迟到数据处理适配性
- Flink 集成复杂度

### 10.2 工程指标

- 部署成本
- 运维监控可接入性
- 观测性成熟度
- 数据治理可衔接性
- 文档与社区成熟度

### 10.3 业务指标

- 是否有助于实时库存场景
- 是否有助于爆款追单特征场景
- 是否有助于未来 AI/特征服务场景

---

## 十一、风险与约束

### 11.1 当前风险

- 团队对 `Fluss` 的使用经验尚少
- 当前项目已经形成稳定主链路，不宜轻易替换
- 新组件引入会增加部署、监控、治理与运维复杂度
- 社区成熟度、生态支持度仍需持续观察

### 11.2 风险控制原则

- 坚持旁路 PoC，不动主链路
- 先做单场景小范围验证
- 先形成结论，再讨论二期引入
- 与现有治理监控体系同步考虑，不单看“能不能跑”

---

## 十二、与项目二期的结合点

若本轮 `Fluss` 预研结果积极，二期可优先考虑以下结合方向：

- 行为流实时特征层
- 爆款追单特征层
- 实时库存状态层
- AI/向量检索前的实时特征输入层

若本轮结果一般，则建议保持：

- 主生产链路继续以 `Kafka + Flink + Paimon + Doris` 为主
- `Fluss` 继续作为观察型技术储备，不急于引入

## 十四、结论

`Fluss` 在本项目中的合理定位，是一期主链路稳定后的“旁路轻量预研组件”，而不是立即进入生产替换范围的核心组件。  
本轮工作的关键，是判断它是否值得在本项目二期中承担某一段明确、有价值、可控风险的实时数据层角色。
