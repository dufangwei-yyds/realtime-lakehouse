# Fluss二期试点建议

> 文档定位：服装行业实时湖仓项目 `Fluss` 二期试点建议  
> 目标：基于一期项目成果、`Fluss` 三阶段预研结论，以及淘宝闪购 `Flink + Fluss` 生产实践启发，形成面向二期的小范围试点建议

---

## 一、结论先行

我的明确建议是：

**建议在二期开展 `Fluss` 小范围试点，但不建议直接将其升级为主生产链路替代方案。**

原因很清楚：

- 我们自己的三阶段预研已经证明 `Fluss` 可用
- 与一期当前技术栈对照后，尚未体现压倒性替代优势
- 但在实时状态层、行为流富化层、日志实时层方向，已经出现了值得继续验证的价值信号
- 淘宝闪购的生产实践进一步说明：`Fluss` 真正适合的，是“精准打击局部痛点”，而不是“一上来全量替换”

---

## 二、试点目标

二期 `Fluss` 试点的目标，不是“证明它先进”，而是回答 3 个工程问题：

1. 它是否能在某段链路里明显降低复杂度
2. 它是否能在某段链路里减少 Flink State 压力或胶水层作业
3. 它是否值得在该局部场景继续扩大范围

---

## 三、建议试点原则

### 3.1 只做局部，不动主链路

二期试点必须坚持：

- 不替换一期主生产链路
- 不影响现有 SLA
- 保持可回退

### 3.2 选痛点最明确的场景

只有以下类型场景，才值得优先试点：

- 当前 Flink State 压力明显
- 当前 Join / 宽表拼接链路复杂
- 当前日志链路 I/O 浪费明显
- 当前同步搬运作业较多、维护成本高

### 3.3 必须可量化评估

每个试点都要提前定义：

- 时延目标
- 资源目标
- 复杂度目标
- 运维代价目标

否则试点容易停留在“能跑”而不是“值得继续”。

---

## 四、建议优先试点的 3 个方向

### 4.1 方向一：实时库存状态层

这是我最优先推荐的方向。

原因：

- 与我们现有项目高度贴合
- `PrimaryKey Table` 在这类场景中最自然
- 结果最容易解释
- 最适合做小范围可回退试点

建议试点内容：

- 选取部分 `sku_id + store_id` 的实时库存状态
- 用 `Fluss PrimaryKey Table` 承接状态更新
- 与当前 `Paimon latest / DWD / DWS` 状态表达方式做对照

重点验证：

- 建模是否更直接
- 状态更新是否更清晰
- 查询与读取语义是否更贴近实时状态层

### 4.2 方向二：行为流富化层

这是第二优先级。

原因：

- 我们前面的 `Lookup Join` 已验证通过
- 和爆款识别、用户行为增强分析天然相关
- 淘宝闪购的订单归因/访购率实践也从侧面说明了这类场景的价值

建议试点内容：

- 选择 1 条行为流
- 选择 1 张商品/状态主键表
- 用 `Log Table + PrimaryKey Table + Lookup Join` 做实时富化

重点验证：

- 是否能减少 Flink State 依赖
- 是否比当前链路更容易理解和维护
- 是否适合作为后续实时特征层前置

### 4.3 方向三：日志实时层

这是我认为二期非常值得补的新方向。

前提是二期把这些数据接进来：

- 埋点数据
- `nginx` 日志
- API 接口日志
- 爬虫日志或抓取结果

如果这些前置接入补齐，那么很适合试：

- `Log Table`
- 服务端列裁剪
- 自动分层到湖仓

重点验证：

- 是否能减少大宽日志的无效读取
- 是否能减少“监控链路 + 归档链路”的重复作业
- 是否能形成“一次写入、多端复用”的日志底座

---

## 五、当前不建议优先试点的方向

### 5.1 不建议直接试全链路替换

不建议：

- 用 `Fluss` 直接替掉 `Paimon`
- 重构整个 ODS/DWD/DWS 主链路
- 一上来就做面向全域的存储重构

原因：

- 成本高
- 风险大
- 当前证据还不支持

### 5.2 不建议优先试“复杂大宽表全量重构”

虽然 `Partial Update` 很吸引人，但对我们来说，现阶段更适合先：

- 做单场景状态层
- 做单场景富化层
- 再逐步观察是否值得扩展到复杂宽表

---

## 六、建议的试点路线

### 6.1 第一阶段：单场景试点

建议优先二选一：

1. 库存状态层
2. 行为流富化层

输出：

- 试点方案
- 测试数据方案
- 操作验证文档
- 试点记录
- 阶段结论

### 6.2 第二阶段：与现网链路并行对照

要求：

- 与现有 `Paimon + Flink` 链路并行运行
- 不替换现网
- 对比时延、复杂度、资源、稳定性

### 6.3 第三阶段：决定是否扩大范围

只有在以下条件满足时，才建议扩大：

- 局部场景收益明显
- 运维复杂度可接受
- 团队已掌握使用边界
- 没有新增明显不确定性

---

## 七、试点评价指标建议

建议每个试点至少评估以下维度：

| 维度 | 关注点 |
|---|---|
| 建模复杂度 | 表设计、DDL、主键与分桶设计是否清晰 |
| SQL 复杂度 | 写入、读取、join、状态表达是否更简洁 |
| 资源开销 | Flink State、作业数、CPU/内存消耗变化 |
| 查询体验 | 结果可解释性、读取模式限制是否可接受 |
| 运维成本 | 插件、依赖、监控、排障复杂度 |
| 迁移可行性 | 是否适合逐步扩大范围 |

---

## 八、与淘宝闪购实践的对应关系

结合淘宝闪购生产实践，我认为我们最适合借鉴的不是全部 5 个场景，而是其中的方法论和其中 3 个切入点：

1. `Delta Join` 的思路  
   对应我们的行为流富化、订单归因、长周期关联类场景

2. `Partial Update` 的思路  
   对应我们未来可能扩展的实时状态宽表、多流指标宽表

3. `Log Table + 列裁剪 + 自动分层` 的思路  
   对应我们二期准备补齐的埋点、日志、API、爬虫数据接入

因此，我建议：

- 不要机械复制他们的全部架构
- 要学习他们“先抓痛点，再局部替换”的方法

---

## 九、最终建议

我的最终建议可以概括为：

1. 二期建议开展 `Fluss` 小范围试点
2. 优先从“实时库存状态层”开始
3. 第二优先选择“行为流富化层”
4. 若二期前置接入体系补齐，再评估“日志实时层”
5. 继续把 `Fluss` 定位为 `Paimon` 的补充和增强，而不是替代

一句话总结：

**二期可以试，但要小步快跑、局部突破、可回退、可量化。**
