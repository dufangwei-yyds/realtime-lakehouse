# Fluss 测试数据方案

> 文档定位：`Fluss` 第二阶段最小链路 PoC 测试数据方案  
> 目标：为场景 A、场景 B 提供可直接支撑验证的最小测试数据设计  
> 原则：数据量适度放大，但仍保持结构简单、便于观察与排查

---

## 一、设计目标

本轮测试数据方案服务于两个 PoC 场景：

- 场景 A：行为流 + 商品状态补全
- 场景 B：库存状态实时表

测试数据设计需要同时满足：

- 能覆盖 `Log Table`
- 能覆盖 `PrimaryKey Table`
- 能覆盖 `Lookup Join`
- 能观察主键更新效果
- 能在本地实验环境中稳定运行

---

## 二、总体设计原则

1. 数据量比“纯最小样例”略大  
便于观察流式读写、lookup 与状态变化，但不追求压测级别规模。

2. 维持有限主键范围  
保证重复主键、状态更新、lookup 命中都能自然发生。

3. 先用 `datagen` 跑通  
后续如需更强可控性，再补一版静态 `VALUES` 或自定义 source。

---

## 三、场景 A 测试数据设计

### 3.1 行为流数据

用途：

- 写入 `Log Table`
- 作为 lookup join 左表

建议字段：

| 字段 | 说明 | 建议范围 |
|---|---|---|
| `event_id` | 事件唯一标识 | 1 到 5000 顺序增长 |
| `user_id` | 用户 ID | 10001 到 10100 |
| `sku_id` | 商品 SKU ID | 1001 到 1020 |
| `event_type` | 事件类型 | 随机字符串或后续替换为枚举 |
| `event_time` | 事件时间 | 当前时间附近自动生成 |

建议规模：

- `rows-per-second = 20`
- 单次观察 1 到 3 分钟
- 预期累计数据量：1200 到 3600 行

设计意图：

- 让 `sku_id` 命中范围和商品状态表重叠
- 保证 lookup join 有较高命中概率

### 3.2 商品状态数据

用途：

- 写入 `PrimaryKey Table`
- 作为 lookup join 右表

建议字段：

| 字段 | 说明 | 建议范围 |
|---|---|---|
| `sku_id` | 商品 SKU ID | 1001 到 1020 |
| `sku_name` | 商品名称 | 随机字符串 |
| `brand_name` | 品牌名 | 随机字符串 |
| `product_type` | 商品类型 | 随机字符串 |
| `available_flag` | 是否可售 | 0 到 1 |
| `status_time` | 状态更新时间 | 当前时间附近自动生成 |

建议规模：

- `rows-per-second = 5`
- 单次观察 1 到 3 分钟
- 由于 `sku_id` 范围有限，会自然形成主键覆盖与更新

设计意图：

- 控制 `sku_id` 总量较小
- 促使 `PrimaryKey Table` 出现重复主键更新
- 方便观察 lookup 补全稳定性

## 四、场景 B 测试数据设计

### 4.1 库存状态数据

用途：

- 写入 `PrimaryKey Table`
- 验证实时状态表与最新态读取

建议字段：

| 字段 | 说明 | 建议范围 |
|---|---|---|
| `sku_id` | 商品 SKU ID | 1001 到 1020 |
| `store_id` | 门店 ID | 1 到 5 |
| `available_qty` | 可售库存 | 0 到 300 |
| `locked_qty` | 锁定库存 | 0 到 60 |
| `in_transit_qty` | 在途库存 | 0 到 80 |
| `update_time` | 更新时间 | 当前时间附近自动生成 |

建议规模：

- `rows-per-second = 10`
- 单次观察 2 到 5 分钟
- 主键总组合数：约 `20 * 5 = 100`

设计意图：

- 主键空间有限，便于形成频繁状态更新
- 数据量足够让我们看到“最新态覆盖”的效果
- 贴近库存状态类实际业务语义

---

## 五、推荐的数据量级

| 场景 | 表 | 推荐 rows-per-second | 建议运行时长 | 预期效果 |
|---|---|---:|---:|---|
| 场景 A | `src_app_click` | 20 | 1 到 3 分钟 | 快速形成行为流样本 |
| 场景 A | `src_sku_status` | 5 | 1 到 3 分钟 | 自然形成主键状态覆盖 |
| 场景 B | `src_inventory_state` | 10 | 2 到 5 分钟 | 持续观察库存最新态 |

说明：

- 这个量级在本地 PoC 环境通常比较友好
- 如果机器资源较紧，可把 `rows-per-second` 统一减半
- 如果想更明显观察 lookup 与状态覆盖，可把运行时长稍微拉长

---

## 六、推荐的观察点

### 6.1 行为流观察点

- `event_id` 是否持续增长
- `sku_id` 是否稳定落在 1001 到 1020 范围
- `Log Table` 是否能持续读取

### 6.2 商品状态观察点

- 同一个 `sku_id` 是否多次出现更新
- lookup join 是否命中并补全字段

### 6.3 库存状态观察点

- `(sku_id, store_id)` 是否形成稳定主键组合
- 多次写入后是否体现“最新值”而不是重复堆积

---

## 七、可选增强数据方案

如果基础 PoC 跑通，后续可以增加一版“更可控”的静态样例数据，用于精确观察主键覆盖与 lookup 结果。

建议增强方向：

- 使用 `VALUES` 构造固定样例
- 固定 `event_type` 为 `click/view/cart/fav`
- 固定 `brand_name` 为 `TBE/LILY/JNBY`
- 固定 `product_type` 为 `T恤/连衣裙/夹克`
- 固定库存状态更新顺序，观察最新态覆盖是否符合预期

本轮第二阶段可先不做，先以 `datagen` 跑通链路为主。

---

## 八、当前结论

本轮测试数据方案建议优先采用：

- `datagen` 作为最小 source
- 有限主键空间
- 中等偏小数据量
- 足够支撑 `Log Table`、`PrimaryKey Table` 与 `Lookup Join` 的验证

这套方案的优点是：

- 开箱即用
- 本地实验成本低
- 便于快速进入第二阶段 PoC
