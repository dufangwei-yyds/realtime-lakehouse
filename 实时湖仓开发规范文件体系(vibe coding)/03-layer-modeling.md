## 二、ODS 层规范

### 2.1 核心职责

| 职责 | 说明 |
|-----|------|
| 贴源接入 | 保持与数据源一致的字段结构和语义 |
| 轻度清洗 | 过滤非法数据、统一空值处理 |
| 元数据注入 | 补充 _event_time、_ingest_time、_partition_dt |
| 原始保留 | 保留 JSON 原始内容，应对 Schema 漂移 |
| 快速入湖 | 延迟解析，不做重度计算 |

### 2.2 建模原则

**三大原则**

1. **保持原始 (Keep Original)**
   - 完整保留 Kafka 原始 JSON 内容
   - 不丢弃任何源系统字段

2. **标准化字段注入**
   - 每张 ODS 事实表必须注入元数据字段
   - _event_time：事件发生时间
   - _ingest_time：入湖时间
   - _partition_dt：日期分区

3. **分区存储**
   - 按 dt（天）+ hh（小时）二级分区
   - 解决大促期间数据膨胀问题

### 2.3 ODS 表类型

| 表类型 | 命名后缀 | 写入模式 | Merge Engine | 适用场景 |
|-------|---------|---------|--------------|---------|
| CDC 事实表 | - | 主键表 | deduplicate | 订单、库存变更 |
| 事件流事实表 | - | Append | - | 行为埋点、日志 |
| 维度追加表 | _append | Append | - | 不变化的维度 |
| 维度历史表 | _history | 主键表 | aggregate | 需要保留历史的维度 |
| 维度最新表 | _latest | 主键表 | deduplicate | 需要最新快照的维度 |
| 去重表 | _dedup | 主键表 | deduplicate | 应对 Kafka 重复 |

### 2.4 ODS 层禁止事项

? **以下行为严格禁止**

- 禁止在 ODS 层做重度业务聚合（如计算销售额汇总）
- 禁止跳过 ODS 直连 DIM 层
- 禁止丢弃源系统字段
- 禁止不注入元数据字段
- 禁止将 ODS 宽表化
- 禁止在 ODS 层直接服务消费侧

## 三、DIM 层规范

### 3.1 核心职责

| 职责 | 说明 |
|-----|------|
| 口径统一 | 同一维度在不同作业中使用口径一致 |
| 主键统一 | 维度主键类型和语义跨层一致 |
| 三态支持 | 支持 append / history / latest 三种形态 |
| 宽表退化 | 高频 Join 维度退化到宽表 |

### 3.2 建模原则

**维度宽表设计原则**

1. **主键唯一** - sku_id 唯一，跨层一致
2. **属性完整** - 包含所有下游常用的维度属性
3. **标准化** - 渠道编码、状态枚举等标准化处理
4. **退化适度** - 仅退化高频 Join 的维度属性

### 3.3 DIM 表类型

| 表类型 | 命名后缀 | 写入模式 | Merge Engine | 适用场景 |
|-------|---------|---------|--------------|---------|
| 维度宽表 | _wide | 主键表 | deduplicate | 高频 Join 场景 |
| 维度历史表 | _history | 主键表 | aggregate | SCD2 拉链 |
| 维度最新表 | _latest | 主键表 | deduplicate | 获取最新维度 |
| 维度追加表 | _append | Append | - | 不变化的维度 |

### 3.4 DIM 层禁止事项

? **以下行为严格禁止**

- 禁止维度主键跨作业不一致
- 禁止在 DIM 层做事实聚合
- 禁止维度属性重复出现在多个维度表
- 禁止在 DIM 层使用业务库原始编码不做标准化

## 四、DWD 层规范

### 4.1 核心职责

| 职责 | 说明 |
|-----|------|
| 事实打宽 | 以事实过程为核心，关联维度属性 |
| 主键统一 | 统一事实主键类型和语义 |
| 事件时间统一 | 统一使用业务事件时间作为事件时间 |
| 维度补齐 | 关联 DIM 层，获取维度属性 |
| 明细沉淀 | 保留明细数据，支持后续分析 |

### 4.2 建模原则

**事实表设计原则**

1. **一个事实过程一张表** - 订单支付、库存变动等独立业务过程分开建模
2. **主键唯一** - 明确主键，支撑幂等写入
3. **事件时间统一** - 统一使用业务事件时间
4. **维度关联** - 通过 FOR SYSTEM_TIME AS OF 语法关联维度
5. **重复防护** - 应对 Kafka 重复和 CDC 更新

### 4.3 DWD 表命名

```
dwd_{domain}_{process}_detail_rt
```

| 组成部分 | 说明 | 示例 |
|---------|------|------|
| domain | 业务域 | trade / inventory / behavior |
| process | 业务过程 | order / inventory / click |
| detail | 明细标识 | detail |
| rt | 实时标识 | rt |

### 4.4 DWD 层禁止事项

? **以下行为严格禁止**

- 禁止将多个业务过程混在一张"万能宽表"
- 禁止主键不明确或跨层不一致
- 禁止使用处理时间代替业务事件时间
- 禁止跳过维度关联直接输出
- 禁止在 DWD 层做重度汇总

## 五、DWS 层规范

### 5.1 核心职责

| 职责 | 说明 |
|-----|------|
| 公共粒度定义 | 先定义粒度，再定义指标 |
| 指标固化 | 核心指标统一在 DWS 固化 |
| 窗口策略 | 明确窗口类型、窗口大小、水位 |
| 延迟策略 | 明确延迟容忍阈值 |

### 5.2 建模原则

**汇总表设计原则**

1. **粒度优先** - 先定义汇总粒度（如 SKU+小时），再定义指标
2. **指标原子化** - 原子指标在 DWS 层定义，派生指标在 ADS 层组合
3. **窗口明确** - Session 窗口、滚动窗口、滑动窗口策略明确
4. **水位明确** - Watermark 延迟策略明确
5. **复用优先** - DWS 层指标可被多个 ADS 层消费

### 5.3 DWS 表命名

```
dws_{theme}_{grain}_rt
```

| 组成部分 | 说明 | 示例 |
|---------|------|------|
| theme | 主题 | inventory / sales / behavior |
| grain | 粒度 | sku_hh / store_dd / sku_dd |
| rt | 实时标识 | rt |

### 5.4 DWS 层禁止事项

? **以下行为严格禁止**

- 禁止粒度不清（如混合不同粒度的指标）
- 禁止派生指标在 DWS 层定义（应在 ADS 层组合）
- 禁止跳过 DWS 层直接在 ADS 层计算公共指标
- 禁止不同汇总粒度混在同一张表

## 六、ADS 层规范

### 6.1 核心职责

| 职责 | 说明 |
|-----|------|
| 面向消费 | 面向 Dashboard / API 分开设计 |
| 解释字段 | 提供指标解释、计算说明 |
| 建议字段 | 提供业务建议、阈值说明 |
| 不重复定义 | 不重复定义 DWS 层已固化的公共指标 |

### 6.2 建模原则

**应用表设计原则**

1. **消费导向** - 按 Dashboard 或 API 需求设计
2. **解释清晰** - 包含指标解释、计算口径说明
3. **字段完整** - 包含业务建议、趋势判断等辅助字段
4. **不重复** - 直接引用 DWS 层指标，不重复计算

### 6.3 ADS 表命名

```
ads_{theme}_{consumer}
```

| 组成部分 | 说明 | 示例 |
|---------|------|------|
| theme | 主题 | inventory / hot_sku / sales |
| consumer | 消费对象 | dashboard / api / report |

### 6.4 ADS 层禁止事项

? **以下行为严格禁止**

- 禁止重复定义 DWS 层已固化的公共指标
- 禁止不包含指标解释字段
- 禁止设计不符合消费场景的表结构

## 七、分层边界检查清单

### 7.1 ODS 层自检

- [ ] 是否做贴源接入？
- [ ] 是否注入元数据字段（_event_time、_ingest_time、_partition_dt）？
- [ ] 是否保留原始数据（JSON）？
- [ ] 是否未做业务聚合？
- [ ] 是否未跳过 ODS 直连 DIM？

### 7.2 DIM 层自检

- [ ] 主键是否统一？
- [ ] 是否支持 append / history / latest 三态？
- [ ] 是否未做事实聚合？
- [ ] 维度属性是否标准化？

### 7.3 DWD 层自检

- [ ] 主键是否明确且跨层一致？
- [ ] 事件时间是否统一使用业务时间？
- [ ] 是否关联 DIM 层获取维度属性？
- [ ] 是否一个事实过程一张表？
- [ ] 是否考虑了重复数据处理？

### 7.4 DWS 层自检

- [ ] 粒度是否明确定义？
- [ ] 指标是否在 DWS 层固化？
- [ ] 窗口策略是否明确？
- [ ] 水位延迟策略是否明确？

### 7.5 ADS 层自检

- [ ] 是否面向消费对象设计？
- [ ] 是否包含解释字段？
- [ ] 是否未重复定义 DWS 层指标？
- [ ] 是否符合 Dashboard / API 需求？
