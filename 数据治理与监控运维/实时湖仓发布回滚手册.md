## 一、文档目标

本手册用于统一实时湖仓一期的发布、回滚和补数动作，目标是：

1. 降低上线变更风险
2. 降低多层联动变更带来的连锁影响
3. 统一发布前检查、发布执行、发布后验证、回滚决策和补数记录
4. 为后续生产级日常维护提供标准操作依据

## 三、发布基本原则

### 3.1 通用原则

- 先测试环境验证，再进入正式发布窗口
- 先主链路、后增强链路
- 先上游、后下游
- 一次只处理一批清晰变更
- 每次发布都必须有对应回滚点

### 3.2 高风险变更识别

以下变更默认按高风险处理：

- 主键变更
- 粒度变更
- 分区策略变更
- 事件时间语义变更
- 维度 Join 键变更
- DWS 指标口径变更
- ADS 结果字段变更
- Doris 物化视图刷新策略变更

## 六、发布执行顺序建议

## 6.1 主链路标准顺序

如一次发布涉及多层，建议顺序为：

1. ODS
2. DIM
3. DWD
4. DWS
5. ADS
6. 消费侧对象

### 原则说明

- 上游先变更，避免下游先依赖不存在字段
- ADS 和消费侧永远放最后
- 物化视图刷新策略调整建议放在结果稳定后再执行

## 6.2 单层发布建议

| 层级 | 发布建议 |
|---|---|
| ODS | 优先保证源头接入和元数据字段兼容 |
| DIM | 优先核查 latest 输入和宽表唯一性 |
| DWD | 优先核查 Join 键、主键和事件时间 |
| DWS | 优先核查窗口、水位和 Doris Sink |
| ADS | 优先核查结果解释字段、Dashboard / API 消费 |

## 七、发布后验证手册

## 7.1 发布后 10 分钟内必查

1. 作业是否恢复为 `RUNNING`
2. Checkpoint 是否恢复连续成功
3. 目标表是否持续出数
4. 告警是否异常升高
5. Dashboard / API 是否可访问

## 7.2 发布后 30 分钟内重点验证

### 作业验证

- Flink 作业状态
- Backpressure
- Checkpoint 成功率
- Sink 成功率

### 结果验证

```sql
SELECT COUNT(*) FROM dwd.dwd_order_trade_detail_rt;
SELECT COUNT(*) FROM dws.dws_inventory_dispatch_rt;
SELECT COUNT(*) FROM ads.ads_hot_sku_reorder_api;
```

### 消费侧验证

- Superset Dashboard 打开正常
- API `/actuator/health` 返回 `UP`
- 关键接口抽样结果正确

## 7.3 发布后差异核对建议

如变更不涉及业务口径，建议核查：

- 行数是否异常波动
- 主键重复是否上升
- 关键维度字段空值率是否上升
- ADS 结果是否大面积变化

## 八、回滚手册

## 8.1 回滚触发条件

以下情况建议立即评估回滚：

- 主链路作业无法恢复
- 结果表无数据或严重偏离预期
- Checkpoint 连续失败
- 消费侧无法使用
- 短时间内无法准确定位问题根因

## 8.2 回滚原则

- 优先回滚到上一个稳定版本
- 不在问题未定位前做二次叠加发布
- 回滚后优先恢复主链路，再处理增强链路
- 回滚动作必须保留记录

## 8.3 常见回滚场景

| 场景 | 回滚建议 |
|---|---|
| Flink SQL 逻辑错误 | 回滚到上一个稳定 SQL 版本 |
| 表结构变更不兼容 | 回滚下游依赖，恢复旧结构 |
| DWS 指标口径错误 | 回滚聚合逻辑，保留旧结果供核查 |
| ADS 结果错误 | 回滚 ADS 刷新逻辑或临时降级消费 |
| API 消费异常 | 回滚查询 SQL 或临时切换旧接口结果 |

## 8.4 回滚后必查项

1. 作业是否恢复正常
2. 数据是否重新产出
3. 告警是否恢复
4. Dashboard / API 是否恢复可用
5. 回滚记录是否已归档

---

## 九、补数手册

## 9.1 补数原则

- 补数优先从最早可控层开始
- ODS 问题优先 ODS 重放
- DWD / DWS 问题优先上游重算
- ADS 问题避免直接人工修结果，优先从上游恢复

## 9.2 常见补数场景

| 场景 | 建议动作 |
|---|---|
| ODS 漏数 | 重放 Kafka / CDC |
| DIM 宽表缺维 | 修正 latest 输入后重建 DIM |
| DWD 补维失败 | 修正维表或 Join 后重跑 DWD |
| DWS 聚合异常 | 回溯时间窗口重新聚合 |
| ADS 刷新失败 | 在 DWS 正常后重刷 ADS |

## 9.3 补数前检查

- 确认补数范围
- 确认影响层级
- 确认是否会影响当前实时链路
- 确认补数窗口内是否已有错误结果
- 确认补数后验证 SQL

## 十、分层发布与回滚建议

## 10.1 ODS

发布重点：

- 源头表结构兼容
- Schema 漂移兼容
- JSON 解析兼容
- append / history / latest 一致性

回滚重点：

- 回到上一个稳定 Source / 解析逻辑
- 重点确认 latest 未回退

## 10.2 DIM

发布重点：

- `dim_sku_wide` 唯一性
- 关键字段补齐
- 维度口径稳定

回滚重点：

- 回滚宽表构建逻辑
- 重新核查 latest 输入完整性

## 10.3 DWD

发布重点：

- 主键幂等
- Lookup Join 字段类型一致
- 事件时间和水位定义一致

回滚重点：

- 回退到稳定版 DWD 逻辑
- 重点核查主键重复和维度补齐率

## 10.4 DWS

发布重点：

- 窗口口径
- 水位定义
- Doris Sink 连通性
- 聚合指标正确性

回滚重点：

- 回退到旧聚合逻辑
- 核查窗口结果与 DWD 是否重新对齐

## 10.5 ADS

发布重点：

- 结果表字段兼容
- 解释字段完整
- Dashboard / API 消费兼容

回滚重点：

- 临时切回旧结果表或旧接口逻辑
- 必要时暂停新字段消费

---

## 十一、风险分级建议

| 风险等级 | 典型场景 | 建议 |
|---|---|---|
| 高 | 主键、粒度、分区、口径变更 | 必须准备回滚方案 |
| 中 | 新增字段、优化 SQL、阈值调整 | 做好发布后观测 |
| 低 | 文档修订、非主链路监控调整 | 正常发布即可 |

## 十三、通过标准

一次发布 / 回滚 / 补数可判定为通过，通常应同时满足：

1. 主链路作业稳定运行
2. 目标表持续出数
3. 无新增 P1 未闭环告警
4. Dashboard / API 可正常使用
5. 发布、回滚或补数记录完整归档
