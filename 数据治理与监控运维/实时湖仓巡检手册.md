## 一、文档目标

本手册用于把实时湖仓一期的巡检动作沉淀成固定操作清单，支撑日常值守、发布前后核查、异常复盘和项目验收。

使用目标：

1. 固化每日、每周、每月巡检节奏
2. 统一巡检对象、巡检项、通过标准和处置动作
3. 降低故障发现滞后和巡检遗漏风险
4. 支撑值守交接和项目长期维护

---

## 二、巡检原则

### 2.1 基本原则

- 先服务、后数据
- 先主链路、后增强链路
- 先时效、后质量
- 先结果、后定位

### 2.2 巡检顺序

1. 基础服务巡检
2. Flink 作业巡检
3. ODS / DIM / DWD / DWS / ADS 分层巡检
4. 消费侧巡检
5. 告警与问题闭环巡检

---

## 三、巡检角色分工

| 角色 | 巡检重点 |
|---|---|
| 数据开发 | 分层数据结果、质量规则、作业逻辑、补数核查 |
| 运维 | 服务存活、资源、告警、容量、发布回滚 |
| BI / 接口支持 | ADS 结果、Dashboard、接口可用性 |
| 项目负责人 | 问题闭环、周月报、验收前总体检查 |

## 四、每日巡检手册

## 4.1 基础服务巡检

### 巡检目标

确认实时湖仓支撑组件全部可用，避免把基础故障误判为数据异常。

### 巡检项

| 巡检项 | 检查方式 | 通过标准 | 异常动作 |
|---|---|---|---|
| Flink Web UI | 页面访问 | 可访问 | 先恢复服务 |
| Kafka Broker | 端口 / Exporter | Broker 正常 | 查 Broker 状态 |
| HDFS | 页面 / 命令 | 可读写 | 优先恢复 HDFS |
| Doris FE / BE | SQL / 页面 | 全部正常 | 恢复 FE / BE |
| Superset | 页面访问 | 可登录 | 查服务或数据源 |
| Prometheus / Grafana | 页面访问 | 可查询指标 | 恢复监控链路 |

### 建议命令

```bash
ss -lntp | grep 8081
ss -lntp | grep 9092
ss -lntp | grep 8030
ss -lntp | grep 9030
ss -lntp | grep 8888
ss -lntp | grep 9090
```

---

## 4.2 Flink 作业巡检

### 巡检目标

确认 ODS、DIM、DWD、DWS 主链路作业都处于健康运行状态。

### 巡检项

| 巡检项 | 通过标准 | 重点关注 |
|---|---|---|
| 作业状态 | `RUNNING` | 是否有失败或重启 |
| Checkpoint | 连续成功 | 是否存在连续失败 |
| Backpressure | 无严重背压 | 是否为红色热点 |
| 延迟 | 在 SLA 范围内 | 是否出现明显抖动 |
| Sink 成功率 | 正常 | Doris / Paimon 是否写入失败 |

### 重点对象

- ODS 维度 SCD2 作业
- ODS 事件流解析作业
- DIM 宽表构建作业
- DWD 明细构建作业
- DWS 聚合写 Doris 作业

### PromQL 参考

```promql
flink_jobmanager_numRegisteredTaskManagers

avg_over_time(flink_jobmanager_job_checkpoint_duration[5m])

max_over_time(flink_taskmanager_job_task_busyTimeMsPerSecond[2m]) / 1000
```

### 异常动作

- 作业失败：先查最近一次错误日志和上次变更
- Checkpoint 失败：优先查状态后端、Sink 和网络
- 背压严重：优先查 Doris Sink、窗口聚合和 Join

---

## 4.3 ODS 层巡检

### 巡检目标

确认 ODS 数据接入稳定，最新镜像、历史数据、事件流解析和 SCD2 处理都正常。

### 必查对象

- `ods_m_product_append`
- `ods_m_product_history`
- `ods_m_product_latest`
- `ods_m_dim_latest`
- `ods_m_product_alias_latest`
- `ods_m_attributesetinstance_latest`
- `ods_m_attribute_latest`
- `ods_order_item_cdc`
- `ods_inventory_txn_cdc`
- `ods_api_logistics_track`
- `ods_api_logistics_track_analyze`
- `ods_social_mention`
- `ods_social_mention_analyze`
- `ods_app_click_analyze`

### 巡检 SQL

```sql
SELECT COUNT(*) FROM ods.ods_m_product_latest;
SELECT COUNT(*) FROM ods.ods_order_item_cdc;
SELECT COUNT(*) FROM ods.ods_inventory_txn_cdc;
SELECT COUNT(*) FROM ods.ods_api_logistics_track_analyze;
SELECT COUNT(*) FROM ods.ods_social_mention_analyze;
SELECT COUNT(*) FROM ods.ods_app_click_analyze;
```

### 质量抽查

```sql
SELECT *
FROM ods.ods_order_item_cdc
WHERE order_id IS NULL OR line_no IS NULL OR sku_id IS NULL;

SELECT json_parse_status, COUNT(*)
FROM ods.ods_social_mention_analyze
GROUP BY json_parse_status;
```

### SCD2 专项巡检

重点关注：

- append / history / latest 是否同步推进
- 最新镜像是否有数据回退
- 历史版本是否存在时间闭合异常
- SCD2 自定义业务指标是否正常上报

建议核查项：

```sql
SELECT COUNT(*) FROM ods.ods_m_product_append;
SELECT COUNT(*) FROM ods.ods_m_product_history;
SELECT COUNT(*) FROM ods.ods_m_product_latest;
```

## 4.4 DIM 层巡检

### 巡检目标

确认商品退化维度宽表稳定可用，能持续服务 DWD 补维。

### 必查对象

- `dim_sku_wide`

### 巡检 SQL

```sql
SELECT COUNT(*) FROM dim.dim_sku_wide;

SELECT sku_id, spu_code, sku_name, brand_name, season, product_type
FROM dim.dim_sku_wide
ORDER BY sku_id
LIMIT 20;
```

### 重复与缺失检查

```sql
SELECT sku_id, COUNT(*)
FROM dim.dim_sku_wide
GROUP BY sku_id
HAVING COUNT(*) > 1;
```

检查重点：

- `sku_id` 是否唯一
- 品牌、季节、品类是否有大面积空值
- 停用商品是否被正确标识

## 4.5 DWD 层巡检

### 巡检目标

确认核心明细宽表出数正常、补维成功、主键稳定。

### 必查对象

- `dwd_order_trade_detail_rt`
- `dwd_inventory_flow_detail_rt`
- `dwd_logistics_track_detail_rt`
- `dwd_user_behavior_detail_rt`
- `dwd_social_mention_detail_rt`
- `dwd_hot_sku_event_detail_rt`

### 巡检 SQL

```sql
SELECT COUNT(*) FROM dwd.dwd_order_trade_detail_rt;
SELECT COUNT(*) FROM dwd.dwd_inventory_flow_detail_rt;
SELECT COUNT(*) FROM dwd.dwd_hot_sku_event_detail_rt;
```

### 主键与补维检查

```sql
SELECT order_id, line_no, partition_dt, COUNT(*)
FROM dwd.dwd_order_trade_detail_rt
GROUP BY order_id, line_no, partition_dt
HAVING COUNT(*) > 1;

SELECT
  COUNT(*) AS total_cnt,
  SUM(CASE WHEN sku_name IS NOT NULL THEN 1 ELSE 0 END) AS filled_cnt
FROM dwd.dwd_order_trade_detail_rt;
```

## 4.6 DWS 层巡检

### 巡检目标

确认公共主题汇总按时产出，指标波动在可接受范围内。

### 必查对象

- `dws_inventory_atp_rt`
- `dws_inventory_dispatch_rt`
- `dws_hot_sku_feature_rt`
- `dws_hot_sku_rank_rt`

### 巡检 SQL

```sql
SELECT COUNT(*) FROM dws.dws_inventory_atp_rt;
SELECT COUNT(*) FROM dws.dws_inventory_dispatch_rt;
SELECT COUNT(*) FROM dws.dws_hot_sku_feature_rt;
SELECT COUNT(*) FROM dws.dws_hot_sku_rank_rt;
```

### 抽样核查

```sql
SELECT stat_minute, sku_id, atp_qty, available_qty, locked_qty, in_transit_qty
FROM dws.dws_inventory_atp_rt
ORDER BY stat_minute DESC
LIMIT 20;

SELECT stat_hour, sku_id, hot_score, order_cnt_1h, social_hot_1h
FROM dws.dws_hot_sku_feature_rt
ORDER BY stat_hour DESC, hot_score DESC
LIMIT 20;
```

## 4.7 ADS 层与消费侧巡检

### 巡检目标

确认面向看板和接口的结果集刷新稳定，对外消费正常。

### 必查对象

- `ads_inventory_dispatch_dashboard`
- `ads_inventory_dispatch_api`
- `ads_hot_sku_dashboard`
- `ads_hot_sku_reorder_api`
- Superset Dashboard
- Java API 服务

### 巡检 SQL

```sql
SELECT COUNT(*) FROM ads.ads_inventory_dispatch_dashboard;
SELECT COUNT(*) FROM ads.ads_inventory_dispatch_api;
SELECT COUNT(*) FROM ads.ads_hot_sku_dashboard;
SELECT COUNT(*) FROM ads.ads_hot_sku_reorder_api;
```

### 更新时间检查

```sql
SELECT MAX(update_time) FROM ads.ads_inventory_dispatch_dashboard;
SELECT MAX(update_time) FROM ads.ads_hot_sku_dashboard;
```

### 消费侧检查

- Superset 数据源连接正常
- Dashboard 可正常加载
- API 健康检查 `/actuator/health` 返回 `UP`
- 关键接口分页、排序、结果字段正常

## 五、每周巡检手册

## 5.1 资源与容量巡检

| 巡检项 | 重点 |
|---|---|
| HDFS 容量 | 是否接近阈值 |
| Paimon Snapshot | 是否增长异常 |
| 小文件数量 | 是否需要 Compact |
| Doris 存储 | 热表是否膨胀 |
| Flink 资源 | TM 数量、CPU、内存是否异常 |

## 5.2 规则与告警巡检

检查：

- 本周告警是否存在高频噪音
- 同类问题是否重复发生
- 是否有异常未纳入规则
- 告警阈值是否需要微调

## 5.3 文档一致性巡检

检查：

- 数据字典是否与现网表结构一致
- 指标口径是否变更
- 新增字段是否同步到文档
- 巡检和告警台账是否同步更新

## 八、异常巡检处置指引

## 8.1 ADS 无数据

排查顺序：

1. ADS 刷新 SQL 是否执行
2. DWS 是否有结果
3. Doris Sink 是否成功
4. 上游 Flink 作业是否正常

## 8.2 DWS 延迟高

排查顺序：

1. 窗口作业状态
2. Checkpoint 是否失败
3. Backpressure 是否严重
4. Doris 写入是否失败
5. DWD 是否正常产出

## 8.3 DWD 结果异常

排查顺序：

1. DIM 是否正常
2. ODS 事实是否有数据
3. Join 键是否一致
4. 是否存在重复膨胀
5. 事件时间是否异常

## 8.4 ODS 异常

排查顺序：

1. 源头数据是否真的到达
2. MySQL CDC / Kafka 是否正常
3. Flink Source 是否报错
4. Analyze 解析失败率是否升高
5. 分区写入是否异常

## 九、巡检通过标准

若以下条件同时满足，则可判定当前巡检通过：

1. 基础服务全部可用
2. Flink 主链路作业稳定运行
3. ODS / DIM / DWD / DWS / ADS 均正常出数
4. 无 P0 / P1 未闭环告警
5. Dashboard 与 API 消费正常
6. 本次巡检记录已完成归档
