# 服装行业实时湖仓SLA规范

> 项目名称：服装行业实时湖仓构建项目一期  
> 文档定位：一期实时湖仓运行服务等级协议规范  
> 适用范围：`ODS -> DIM -> DWD -> DWS -> ADS` 全链路时效、可用性、告警响应与恢复要求

---

## 一、文档目标

本规范用于定义一期实时湖仓的服务等级目标，目标是：

1. 明确全链路各层时效承诺
2. 明确作业、结果、消费侧的可用性目标
3. 明确告警响应与故障恢复时限
4. 为日常巡检、验收与后续维护提供统一标准

说明：

- 本规范参考实时数仓与湖仓项目常见最佳实践，并结合一期测试环境与当前实现能力制定
- 一期以“可用、稳定、可验收”为主，后续可在生产化阶段继续提高目标值

---

## 二、SLA 分级说明

| 级别 | 说明 | 典型对象 |
|---|---|---|
| 核心链路 SLA | 直接影响两大核心场景的主链路 | ODS 关键事实、DIM、DWD、DWS、ADS |
| 消费侧 SLA | 直接影响 Dashboard 和接口使用 | Superset、API、ADS 结果集 |
| 支撑型 SLA | 支撑主链路运行的基础组件 | Kafka、Flink、Doris、监控链路 |

---

## 三、分层时效目标

## 3.1 ODS 层

| 对象 | 时效目标 | 说明 |
|---|---|---|
| ERP 主数据 CDC | 分钟级 | 以 CDC 同步延迟不超过 5 分钟为目标 |
| ERP 订单 / 库存 CDC | 分钟级 | 核心主链路建议不超过 5 分钟 |
| 行为埋点事件 | 秒级到分钟级 | 视 Topic 和消费情况而定 |
| 物流 API 数据 | 5-15 分钟 | 受第三方接口频率影响 |
| 社媒舆情数据 | 小时级 | 受采集方式和平台限制影响 |

## 3.2 DIM 层

| 对象 | 时效目标 | 说明 |
|---|---|---|
| `ods_m_*_latest` | 分钟级 | 支撑下游补维 |
| `dim_sku_wide` | 分钟级 | 相对 latest 延迟建议不超过 5 分钟 |

## 3.3 DWD 层

| 对象 | 时效目标 | 说明 |
|---|---|---|
| 订单交易明细 | 分钟级 | 建议不超过 5 分钟 |
| 库存流水明细 | 分钟级 | 建议不超过 5 分钟 |
| 物流轨迹明细 | 5-15 分钟 | 受上游 API 时效影响 |
| 行为与社媒明细 | 分钟级到小时级 | 视上游特性而定 |

## 3.4 DWS 层

| 对象 | 时效目标 | 说明 |
|---|---|---|
| 库存 ATP 汇总 | 分钟级 | 关键分钟级主题 |
| 调拨建议汇总 | 分钟级 | 与 ATP 同步要求 |
| 爆款特征汇总 | 小时级 | 以小时粒度输出为主 |
| 爆款排行汇总 | 小时级 | 与特征汇总同步 |

## 3.5 ADS 层

| 对象 | 时效目标 | 说明 |
|---|---|---|
| 库存驾驶舱结果集 | 分钟级 | 支撑运营驾驶舱 |
| 调拨建议接口结果集 | 分钟级 | 支撑接口消费 |
| 爆款趋势 Dashboard 结果集 | 小时级 | 支撑爆款趋势分析 |
| 追单建议接口结果集 | 小时级 | 支撑业务建议输出 |

---

## 四、可用性目标

## 4.1 作业与链路可用性

| 对象 | 建议目标 |
|---|---|
| Flink 核心主作业 | `>= 99%` |
| ODS / DIM / DWD / DWS / ADS 主链路 | `>= 99%` |
| Kafka 消费链路 | `>= 99%` |
| Doris 查询服务 | `>= 99%` |

## 4.2 消费侧可用性

| 对象 | 建议目标 |
|---|---|
| Superset 核心 Dashboard | `>= 99%` |
| 数据服务接口 | `>= 99%` |
| 健康检查接口 | `>= 99%` |

说明：

- 以上为一期建议目标
- 当前测试环境单节点约束下，可接受短时波动，但验收时应保证主链路可用

---

## 五、告警响应与恢复时限

## 5.1 告警分级响应要求

| 级别 | 响应时限 | 处理要求 |
|---|---|---|
| `P1` | 15 分钟内 | 立即介入，优先恢复主链路 |
| `P2` | 30 分钟内 | 当日处理，必要时降级消费 |
| `P3` | 当日内 | 纳入巡检与优化 |
| `P4` | 周期性处理 | 记录即可 |

## 5.2 恢复时限建议

| 场景 | 恢复建议时限 |
|---|---|
| 主作业退出运行态 | 30 分钟内恢复 |
| ADS 无法刷新 | 30 分钟内恢复或降级 |
| Dashboard / API 不可用 | 30 分钟内恢复 |
| 关键结果延迟超 SLA | 1 小时内完成定位与处理 |

## 六、分层监控目标

## 6.1 ODS 层

重点监控：

- CDC 延迟
- Topic 消费积压
- JSON 解析失败率
- ODS 入湖条数

建议要求：

- 核心 ODS 表出数异常应在当日被发现

## 6.2 DIM 层

重点监控：

- latest 到 DIM 写入延迟
- `dim_sku_wide` 重复率
- 关键字段空值率

## 6.3 DWD 层

重点监控：

- 作业状态
- 主键重复
- 维度补齐率
- 时间字段异常

## 6.4 DWS 层

重点监控：

- 窗口作业延迟
- 指标波动
- Doris Sink 成功率

## 6.5 ADS 层

重点监控：

- 结果更新时间
- Dashboard / API 结果可用性
- 物化视图刷新状态

---

## 七、消费侧 SLA 建议

## 7.1 Superset

| 项目 | 建议目标 |
|---|---|
| 数据源连接 | 可用 |
| 核心 Dashboard 可访问 | `>= 99%` |
| 核心 Dashboard 数据刷新 | 满足对应 ADS 时效 |

说明：

- 个别图表能力差异不视为主链路 SLA 失败
- 但应明确记录并纳入后续优化

## 7.2 数据服务接口

| 项目 | 建议目标 |
|---|---|
| 健康检查 `/actuator/health` | 可用 |
| 核心查询接口 | 可用 |
| 分页、排序、字段返回 | 符合设计 |
| 结果时效 | 与 ADS 结果时效保持一致 |

---

## 八、SLA 例外说明

以下情况可作为 SLA 例外处理，但必须记录：

1. 第三方物流 API 上游不可用
2. 社媒采集平台限流或结构变化
3. 测试环境单节点约束导致的非生产级短时波动
4. 发布窗口内的受控维护时间

---

## 九、SLA 验收建议

SLA 验收建议至少确认：

1. 各层时效目标是否明确
2. 核心链路是否具备监控指标
3. 告警响应要求是否明确
4. 恢复时限是否明确
5. 消费侧 SLA 是否明确

---

## 十、SLA 与后续演进建议

一期建议将本规范作为“基线版 SLA”，后续生产化阶段可进一步提升：

1. 将分钟级场景进一步压缩到更低延迟
2. 将更多主题纳入 SLA 体系
3. 将 Fluss / Doris AI 实验结果纳入新的服务能力评估
