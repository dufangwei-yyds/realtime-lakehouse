# 实时湖仓事件流建模实现方案

> 以下为对应【实时湖仓事件流建模实现方案(旧).sql】脚本的完整内容，已原样搬运，便于在仅支持 Markdown 的设备中查看。

```sql
/*
=============================================================
Kafka事件流建模解读
ODS层(Operational Data Store, 操作数据存储层)
ODS层又称"贴源层", 是数据从业务系统进入湖仓的第一站, 其核心目标是保持数据原貌并实现低延迟接入
数据来源与接入: 主要通过CDC(变更数据捕获)技术实时同步业务数据库(如MySQL、Oracle的Binlog)及埋点日志(Kafka消息流)
职责与特点:
原始性: 尽可能保留源系统的数据结构和原始字段, 不做深度转换, 仅进行简单的清洗(如统一时间格式、处理异常字段)
备份与回溯: 实时湖仓通过流式写入支持版本管理, 可用于历史故障溯源和数据重新消费
减少干扰: 承担了业务库的查询压力, 保护生产环境不受分析任务影响
ODS提供"发生了什么"(实时事件流)
核心: 流水存储(原始记录、标准化字段、分区存储), 轻量清洗(关联故障), 延后解析(Schema漂移)

在实时湖仓架构中, 来自Kafka的事件流(用户行为埋点、传感器数据、服务端日志等)是ODS层数据量最大、最无序的部分
与MySQL CDC数据不同, Kafka事实流具有Append-Only(只增不删)、高并发、Schema易变的特点

建模规范: ODS层事件流的"三大原则"
1. 保持原始(Keep Original)
ODS层不应进行任何破坏性的数据处理(如聚合、丢弃字段)
必须完整保留Kafka中的原始JSON内容
2. 标准化字段注入
每张ODS事实表必须注入以下"湖仓元数据字段":
_event_time: 事件发生时间(从日志中提取)
_ingest_time: 进入湖仓的时间
_partition_dt: 分区日期(通常是yyyyMMdd)
_kafka_offset: Kafka偏移量(用于排查数据丢失或重复)
3. 命名规范
格式: ods_[来源]_[主题/业务域]_[事件名]
示例: ods_app_fashion_click(App端服装点击流)

工业级建模方法: Schema与存储设计
针对服装行业高并发场景(如大促抢购), 推荐采用"松散摄入+强类型解析"的方案
1. 存储格式选型: Paimon Append表
由于事实流不需要更新, 应选择Paimon的Append-Only表(无主键), 以获得极致的写入吞吐量
2. 分层解析: JSON宽表化vs Raw存储
方法A(宽表化): 在Flink SQL中将JSON字段打平, 这种方法下游查询快, 但如果Kafka增加字段, ODS任务需重启
方法B(Raw存根): 将整条JSON存在一个STRING字段中, 另提取event_time和user_id作为列
最佳实践: 关键字段(ID、时间)独立成列, 非核心字段统一存入一个MAP或JSON STRING

过来人的"血泪"教训(建模细节)
1. 解决"数据膨胀": 分区策略
教训: 服装企业在大促期间(如双11), 单日点击流可达数十亿, 如果只按天分区, 单个分区文件数过多会压垮HDFS
实践: 建议采用dt(天)+hh(小时)二级分区
2. 应对"Schema漂移"
教训: 前端埋点经常改字段名, 如果ODS强制定义Schema, 任务会频繁挂掉
实践: ODS层建议使用Paimon的Schema Evolution(模式演进)功能
     Flink1.18+配合Paimon支持ALTER TABLE ... ADD COLUMN
3. 关联DWD前的"净土"
教训: 直接把未经清洗的Kafka数据去关联维表, 会导致严重的Join倾斜(比如空ID关联)
实践: 在进入ODS湖表前, 利用Flink进行简单的NULL值过滤或填充默认值(如COALESCE(sku_id, '-1'))
=============================================================
*/

/*
=============================================================
 * 整体说明
 * 1. 业务背景
 * 针对服装行业APP用户行为埋点场景, 构建实时事实层建模体系, 解决埋点数据Schema漂移、迟到数据、重复数据等工业级问题,
 * 遵循【原始存储 + 延迟解析 + 幂等写入】的实时湖仓最佳实践
 *
 * 2. 技术架构
 * Kafka(行为埋点事件流) → Flink SQL(实时清洗/解析) → Paimon(ODS层存储)
 *
 * 3. 前置条件
 * Kafka集群正常运行, 已创建【fashion_app_click】Topic
 * Flink已部署并加载flink-connector-kafka、flink-connector-paimon
 * HDFS集群可用, Paimon Warehouse路径可访问
 * Flink具备Kafka消费权限、Paimon写入权限
=============================================================
*/

/*
=============================================================
步骤1: Kafka行为埋点数据模拟
说明：模拟服装APP用户行为埋点数据, 包含点击/浏览/加购/点赞等行为, ext_info为动态JSON字段(模拟Schema漂移场景)

{"uid":"U9877","item_id":"102","action":"view","event_ts":"2025-12-23 14:05:01","ext_info":"{\"device\":\"Android\",\"page\":\"home\",\"spm\":\"x.y.z\"}"}
{"uid":"U9878","item_id":"103","action":"add_cart","event_ts":"2025-12-24 14:00:01","ext_info":"{\"device\":\"iPhone17\",\"page\":\"category_list\",\"spm\":\"a.b.c\"}"}

{"uid":"U9880","item_id":"105","action":"view","event_ts":"2025-12-24 15:00:01","ext_info":"{\"promo_id\":\"1\",\"client_type\":\"iPhone17\",\"page\":\"category_list\",\"spm\":\"a.b.c\"}"}

# 第四批次: 重复数据(模拟Flink重启重消费)
kafka-console-producer.sh --broker-list localhost:9092 --topic fashion_app_click
{"uid":"U9885","item_id":"110","action":"click","event_ts":"2025-12-24 17:02:01","ext_info":"{\"promo_id\":\"4\",\"client_type\":\"Android\",\"page\":\"home\",\"spm\":\"x.y.z\"}"}
{"uid":"U9885","item_id":"110","action":"click","event_ts":"2025-12-24 17:02:01","ext_info":"{\"promo_id\":\"4\",\"client_type\":\"Android\",\"page\":\"home\",\"spm\":\"x.y.z\"}"}
=============================================================
*/

/*
=============================================================
 * 步骤2: Paimon ODS层建模(Flink SQL)
 * 说明：
 * 构建ODS层事实表三大原则:
 * 保持原始: 存储完整JSON串, 应对 Schema漂移
 * 注入元数据: 补充事件时间、摄入时间、Kafka元数据
 * 分区存储: 按天+小时分区, 解决数据膨胀问题
 =============================================================
*/

/*
 * 步骤2.1: 创建Paimon Catalog及ODS数据库
 * 说明: 关联Paimon湖仓, ODS层存储原始增量事实数据
 */
CREATE CATALOG paimon_catalog WITH (
    'type' = 'paimon',                                            -- 连接器类型: Paimon
    'warehouse' = 'hdfs://192.168.63.128:8020/paimon/erp_dw'  -- Paimon仓库根路径
);
USE CATALOG paimon_catalog;
CREATE DATABASE IF NOT EXISTS ods;                                -- 创建ODS层数据库
USE ods;

/*
 * 步骤2.2: 创建ODS层原始事实表(Append-Only)
 * 核心设计:
 * - append-only: 极致写入吞吐, 适合埋点海量数据
 * - 动态分桶(bucket=-1): 应对大促流量突增
 * - 二级分区: 按天(_partition_dt)+小时(_partition_hh)分区, 提升查询效率
 */
CREATE TABLE ods.ods_kafka_app_click (
    -- 业务核心字段(清洗后)
    user_id STRING,                      -- 用户ID(清洗后, 非空)
    sku_id STRING,                       -- 商品SKU ID(清洗后)
    behavior STRING,                     -- 用户行为(click/view/add_cart/like)
    -- 原始JSON存根: 应对Schema漂移, 保留完整原始数据
    raw_json STRING,
    -- 标准元数据字段(注入)
    _event_time TIMESTAMP(3),            -- 事件发生时间(埋点时间)
    _ingest_time TIMESTAMP(3),           -- 数据入湖时间(Flink处理时间)
    -- Kafka元数据: 用于审计、重放、去重
    _kafka_offset BIGINT,                -- Kafka消息偏移量
    _kafka_partition INT,                -- Kafka分区号
    -- 分区字段(物理分区)
    _partition_dt STRING,                -- 日期分区(yyyyMMdd)
    _partition_hh STRING,                -- 小时分区(HH)
    WATERMARK FOR _event_time AS _event_time - INTERVAL '5' SECOND
) PARTITIONED BY (_partition_dt, _partition_hh)  -- 按日期+小时分区
WITH (
    'connector' = 'paimon',              -- 存储连接器: Paimon
    'write-mode' = 'append-only',        -- 写入模式: 仅追加(高吞吐)
    'bucket' = '-1',                     -- 动态分桶: 自动适配数据量
    'file.format' = 'orc'                -- 文件格式: ORC(压缩率高, 查询快)
);

CREATE TABLE paimon_catalog.ods.ods_kafka_order (
    order_id STRING,
    user_id STRING,
    sku_id STRING,
    order_amount DECIMAL(10,2),
    -- 标准元数据注入
    _event_time TIMESTAMP(3),
    _ingest_time TIMESTAMP(3),
    _kafka_offset BIGINT,
    _kafka_partition INT,
    _partition_dt STRING,
    _partition_hh STRING,
    WATERMARK FOR _event_time AS _event_time - INTERVAL '5' SECOND
) PARTITIONED BY (_partition_dt,_partition_hh)
WITH (
    'connector' = 'paimon',
    'write-mode' = 'append-only',
    'bucket' = '-1',
    'file.format' = 'orc'
);

/*
 * 步骤2.3: 创建ODS去重表(应对重复数据)
 * 核心设计:
 * - 联合主键: _kafka_partition + _kafka_offset + _partition_dt, 保证消息唯一性
 * - deduplicate引擎: 自动去重, 实现Exactly-Once语义
 */
CREATE TABLE ods.kafka_app_click_dedup (
    -- 同基础ODS表字段
    user_id STRING,
    sku_id STRING,
    behavior STRING,
    raw_json STRING,
    _event_time TIMESTAMP(3),
    _ingest_time TIMESTAMP(3),
    _kafka_offset BIGINT,
    _kafka_partition INT,
    _partition_dt STRING,
    _partition_hh STRING,
    WATERMARK FOR _event_time AS _event_time - INTERVAL '5' SECOND,
    -- 核心: 联合主键(同一分区+同一偏移量+同一日期的消息唯一)
    PRIMARY KEY (_kafka_partition, _kafka_offset, _partition_dt, _partition_hh) NOT ENFORCED
) PARTITIONED BY (_partition_dt, _partition_hh)
WITH (
    'connector' = 'paimon',
    'merge-engine' = 'deduplicate',     -- 合并引擎: 主键去重(默认)
    'bucket' = '4',                      -- 主键表建议显式指定分桶数
    'ignore-delete' = 'true'             -- 忽略删除消息, 防止数据丢失
);

/********************************************************************************
 * 步骤3: Flink SQL实时清洗与摄入(ODS层，完整留存原内容)
 * 说明：实现【净土实践】: 过滤空值、标准化字段、注入元数据, 同时处理迟到数据的分区路由
 ********************************************************************************/

/*
 * 步骤3.1: 定义Kafka源表(实时消费埋点数据、订单流数据)
 * 核心配置:
 * - Watermark: 允许数据乱序/迟到1分钟(工业级配置)
 * - 元数据字段: 显式声明Kafka offset/partition
 * - 容错配置: 忽略JSON解析错误, 防止任务崩溃
 */
CREATE TEMPORARY TABLE kafka_app_click_source (
    -- 埋点原始字段
    uid STRING,                          -- 用户ID(原始)
    item_id STRING,                      -- 商品ID(原始)
    action STRING,                       -- 行为类型(原始)
    event_ts TIMESTAMP(3),               -- 事件时间(原始)
    ext_info STRING,                     -- 扩展字段(JSON串)
    -- Kafka元数据字段(VIRTUAL: 虚拟字段, 不存储, 仅读取)
    kafka_offset BIGINT METADATA FROM 'offset' VIRTUAL,
    kafka_partition INT METADATA FROM 'partition' VIRTUAL,
    -- Watermark: 水位线, 允许数据迟到1分钟
    WATERMARK FOR event_ts AS event_ts - INTERVAL '1' MINUTE
) WITH (
    'connector' = 'kafka',                                   -- 连接器: Kafka
    'topic' = 'fashion_app_click',                           -- 消费Topic
    'properties.bootstrap.servers' = 'localhost:9092',       -- Kafka地址
    'properties.group.id' = 'flink_ods_app_click_consumer',  -- 消费者组ID
    'scan.startup.mode' = 'latest-offset',                   -- 启动模式: 从最新偏移量开始
    'properties.auto.offset.reset' = 'latest',               -- 偏移量重置策略
    'format' = 'json',                                       -- 数据格式: JSON
    'json.ignore-parse-errors' = 'true',                     -- 容错: 忽略JSON解析错误
    'json.fail-on-missing-field' = 'false'                   -- 容错: 字段缺失不报错
);

CREATE TEMPORARY TABLE kafka_order_source (
    oid STRING,
    uid STRING,
    item_id STRING,
    amt DECIMAL(10,2),
    order_ts TIMESTAMP(3),
    -- 标准元数据注入
    kafka_offset BIGINT METADATA FROM 'offset' VIRTUAL,
    kafka_partition INT METADATA FROM 'partition' VIRTUAL,
    WATERMARK FOR order_ts AS order_ts - INTERVAL '1' MINUTE
) WITH (
    'connector' = 'kafka',
    'topic' = 'fashion_app_order',
    'properties.bootstrap.servers' = 'localhost:9092',
    'properties.group.id' = 'flink_ods_app_order_consumer',
    'scan.startup.mode' = 'latest-offset',
    'properties.auto.offset.reset' = 'latest',
    'format' = 'json',
    'json.ignore-parse-errors' = 'true',
    'json.fail-on-missing-field' = 'false'
);

/*
 * 步骤3.2: ETL摄入任务(基础ODS表, Append模式)
 * 核心逻辑:
 * - 净土清洗: 过滤空用户ID, 替换空SKU ID
 * - 字段标准化: 行为类型转大写
 * - 分区路由: 基于事件时间生成分区(解决迟到数据分区错误)
 */
SET 'execution.runtime-mode' = 'streaming';      -- 流模式执行
SET 'execution.checkpointing.interval' = '10s';  -- 检查点间隔(必配)

INSERT INTO ods.ods_kafka_app_click
SELECT
    -- 净土清洗: 非空处理, 防止下游Join倾斜
    COALESCE(uid, 'UNKNOWN') as user_id,
    COALESCE(item_id, '-1') as sku_id,
    UPPER(action) as behavior,           -- 标准化: 行为类型转大写
    ext_info as raw_json,                -- 保留原始JSON, 应对Schema漂移
    -- 元数据注入
    event_ts as _event_time,             -- 事件时间(埋点时间)
    CURRENT_TIMESTAMP as _ingest_time,   -- 入湖时间(当前Flink时间)
    kafka_offset as _kafka_offset,       -- Kafka偏移量
    kafka_partition as _kafka_partition, -- Kafka分区
    -- 分区字段: 基于事件时间生成(核心！解决迟到数据分区问题)
    DATE_FORMAT(event_ts, 'yyyyMMdd') as _partition_dt,
    DATE_FORMAT(event_ts, 'HH') as _partition_hh
FROM kafka_app_click_source
WHERE uid IS NOT NULL;  -- 核心清洗: 过滤无用户ID的非法流量

INSERT INTO paimon_catalog.ods.ods_kafka_order
SELECT
    oid as order_id,
    COALESCE(uid,'UNKNOWN') as user_id,
    COALESCE(item_id,'-1') as sku_id,
    amt as order_amount,
    order_ts as _event_time,
    CURRENT_TIMESTAMP as _ingest_time,
    kafka_offset as _kafka_offset,
    kafka_partition as _kafka_partition,
    DATE_FORMAT(order_ts,'yyyyMMdd') as _partition_dt,
    DATE_FORMAT(order_ts,'HH') as _partition_hh
FROM kafka_order_source
WHERE uid IS NOT NULL;

/*
 * 步骤3.3: ETL摄入任务(去重ODS表, Deduplicate模式)
 * 逻辑与基础表一致, 仅写入去重表
 */
INSERT INTO ods.kafka_app_click_dedup
SELECT
    COALESCE(uid, 'UNKNOWN') as user_id,
    COALESCE(item_id, '-1') as sku_id,
    UPPER(action) as behavior,
    ext_info as raw_json,
    event_ts as _event_time,
    CURRENT_TIMESTAMP as _ingest_time,
    kafka_offset as _kafka_offset,
    kafka_partition as _kafka_partition,
    DATE_FORMAT(event_ts, 'yyyyMMdd') as _partition_dt,
    DATE_FORMAT(event_ts, 'HH') as _partition_hh
FROM kafka_app_click_source
WHERE uid IS NOT NULL;

/********************************************************************************
 * 步骤4: ODS层JSON解析(应对Schema漂移)
 * 说明：延迟解析原始JSON, 兼容Schema漂移(字段改名/新增), 实现【零停机适配业务变更】
 ********************************************************************************/

/*
 * 步骤4.1: 创建ODS层解析表(初始Schema)
 * 说明: 解析JSON中的核心字段, 供下游分析使用
 */
CREATE TABLE ods.ods_app_click_analyze (
    user_id STRING,                      -- 用户ID
    sku_id STRING,                       -- 商品SKU ID
    behavior STRING,                     -- 用户行为
    device_type STRING,                  -- 设备类型(解析自raw_json)
    page_id STRING,                      -- 页面ID(解析自raw_json)
    promotion_id STRING,
    _event_time TIMESTAMP(3),            -- 事件时间
    json_parse_status STRING,            -- 解析状态(成功/失败)
    WATERMARK FOR _event_time AS _event_time - INTERVAL '5' SECOND
) WITH (
    'connector' = 'paimon',
    'write-mode' = 'append-only'
);

/*
 * 步骤4.2: 初始解析任务(适配原始Schema)
 */
INSERT INTO ods.ods_app_click_analyze
SELECT
    user_id,
    sku_id,
    behavior,
    JSON_VALUE(raw_json, '$.device') as device_type,  -- 解析原始device字段
    JSON_VALUE(raw_json, '$.page') as page_id,        -- 解析page字段
    _event_time,
    -- 解析状态判断
    CASE WHEN JSON_VALUE(raw_json, '$.device') IS NULL THEN '解析失败' ELSE '解析成功' END as json_parse_status
FROM ods.ods_kafka_app_click;

/*
 * 步骤4.3: Schema漂移适配(零停机升级)
 * 核心逻辑:
 * - 字段改名: 兼容client_type(新)和device(旧)
 * - 字段新增: 解析promo_id字段
 */
-- 步骤4.3.1: 新增字段(Paimon支持Schema Evolution, 无需停机)
ALTER TABLE ods.ods_app_click_analyze ADD COLUMN promotion_id STRING;

-- 步骤4.3.2: 重启解析任务(兼容新旧字段)
INSERT INTO ods.ods_app_click_analyze
SELECT
    user_id,
    sku_id,
    behavior,
    -- 兼容字段改名: 优先解析新字段, 失败则回退到旧字段
    COALESCE(
        JSON_VALUE(raw_json, '$.client_type'),  -- 新字段: client_type
        JSON_VALUE(raw_json, '$.device')        -- 旧字段: device
    ) as device_type,
    JSON_VALUE(raw_json, '$.page') as page_id,
    -- 解析新增字段: promo_id
    JSON_VALUE(raw_json, '$.promo_id') as promotion_id,
    _event_time,
    -- 升级解析状态判断
    CASE
        WHEN JSON_VALUE(raw_json, '$.client_type') IS NOT NULL THEN '解析成功（新字段）'
        WHEN JSON_VALUE(raw_json, '$.device') IS NOT NULL THEN '解析成功（旧字段）'
        ELSE '解析失败'
    END as json_parse_status
```

## ODS核心事实表建模设计与 Flink SQL 实现

> 说明：现有脚本主要覆盖了行为埋点事件流与一个简化订单流。为了支撑服装行业实时湖仓后续 DWD 建设，这里补充 4 类可与 `dim.dim_sku_wide` 关联的核心事实：
>
> - 业务库订单事实
> - 业务库库存流水事实
> - 第三方 API 物流轨迹事实
> - 爬虫社媒事实
>
> 统一原则：
>
> - 遵循 ODS 事件流与 CDC 建模规范
> - 保留原始语义，轻度清洗，补齐元数据
> - 统一沉淀可关联字段：`sku_id`、`order_id`、`biz_time/event_time`

### 一、建模设计说明

| 事实主题     | 来源       | 接入方式             | 与维度表关联键                     | 目标                   |
| ------------ | ---------- | -------------------- | ---------------------------------- | ---------------------- |
| 订单事实     | ERP / OMS  | MySQL CDC            | `sku_id = dim.dim_sku_wide.sku_id` | 支撑销售分析、转化分析 |
| 库存流水事实 | ERP / WMS  | MySQL CDC            | `sku_id = dim.dim_sku_wide.sku_id` | 支撑库存、调拨、ATP    |
| 物流轨迹事实 | 第三方 API | Kafka / API -> Kafka | `sku_id`、`order_id`               | 支撑履约与物流分析     |
| 社媒事实     | 爬虫采集   | Kafka                | `sku_id` 或 `spu_code`             | 支撑爆款识别与舆情分析 |

| `FROZEN`     | 冻结库存 | 因风控、争议或质检冻结       |

### ODS 事实表数据质量校验规则表

#### 1. `ods_order_item_cdc` 数据质量校验

| 校验维度 | 校验字段                                  | 规则                                       | 说明                                                       |
| -------- | ----------------------------------------- | ------------------------------------------ | ---------------------------------------------------------- |
| 主键     | `order_id + line_no`                      | 不可为空，组合唯一                         | 防止订单明细重复或主键缺失                                 |
| 关联键   | `sku_id`                                  | 不可为空                                   | 必须可关联商品维度                                         |
| 时间     | `order_time / _event_time / _ingest_time` | `_event_time <= _ingest_time`              | 保证事件时间不晚于入湖时间                                 |

| 数量     | `qty`                                     | `> 0`                                      | 订单购买数量必须为正                                       |

| 去重     | `order_id + line_no`                      | 同一主键只保留一条最新记录                 | 保证 CDC 幂等写入                                          |

#### 5. 通用 ODS 校验建议 SQL

```sql
-- 0. 订单空主键校验
SELECT *
FROM ods_order_item_cdc
WHERE order_id IS NULL OR line_no IS NULL;

-- 订单主键重复校验
SELECT order_id, line_no, COUNT(*) AS cnt
FROM ods_order_item_cdc
GROUP BY order_id, line_no
HAVING COUNT(*) > 1;

-- 订单时间逆序校验
SELECT *
FROM ods_order_item_cdc
WHERE _event_time > _ingest_time;

-- 订单金额异常校验
SELECT *
FROM ods_order_item_cdc
WHERE sale_price < 0 OR sale_amount < 0 OR qty <= 0;

-- 订单状态非法值校验
SELECT *
FROM ods_order_item_cdc
WHERE order_status NOT IN ('CREATED', 'PAID', 'SHIPPED', 'SIGNED', 'CLOSED', 'REFUNDING', 'REFUNDED');

-- 订单分区错误校验
SELECT *
FROM ods_order_item_cdc
WHERE DATE_FORMAT(_event_time, 'yyyyMMdd') <> _partition_dt;

-- 库存流水空主键/空关联键校验
SELECT *
FROM ods_inventory_txn_cdc
WHERE txn_id IS NULL OR sku_id IS NULL;

-- 库存流水零值校验
SELECT *
FROM ods_inventory_txn_cdc
WHERE qty = 0;

-- 库存流水状态非法校验
SELECT *
FROM ods_inventory_txn_cdc
WHERE stock_status NOT IN ('AVAILABLE', 'LOCKED', 'IN_TRANSIT', 'DEFECTIVE', 'FROZEN');

-- 库存流水业务类型非法校验
SELECT *
FROM ods_inventory_txn_cdc
WHERE biz_type NOT IN ('PUR_IN', 'SALE_OUT', 'RET_IN', 'TRANSFER_OUT', 'TRANSFER_IN', 'ALLOC_LOCK', 'ALLOC_RELEASE', 'ADJUST');

-- 库存流水时间逆序校验
SELECT *
FROM ods_inventory_txn_cdc
WHERE _event_time > _ingest_time;

-- 物流迟到分区校验
SELECT logistics_no, _event_time, _partition_dt, _partition_hh
FROM ods_api_logistics_track
WHERE DATE_FORMAT(_event_time, 'yyyyMMdd') <> _partition_dt
   OR DATE_FORMAT(_event_time, 'HH') <> _partition_hh;

-- 物流空业务键校验
SELECT *
FROM ods_api_logistics_track
WHERE logistics_no IS NULL
   OR (order_id IS NULL AND sku_id IS NULL);

-- 物流重复消息校验
SELECT _kafka_partition, _kafka_offset, _partition_dt, _partition_hh, COUNT(*) AS cnt
FROM ods_api_logistics_track
GROUP BY _kafka_partition, _kafka_offset, _partition_dt, _partition_hh
HAVING COUNT(*) > 1;

-- 社媒互动指标异常校验
SELECT *
FROM ods_social_mention
WHERE like_cnt < 0 OR fav_cnt < 0 OR comment_cnt < 0;

-- 社媒空业务键校验
SELECT *
FROM ods_social_mention
WHERE note_id IS NULL
   OR (sku_id IS NULL AND spu_code IS NULL);

-- 社媒分区错误校验
SELECT *
FROM ods_social_mention
WHERE DATE_FORMAT(_event_time, 'yyyyMMdd') <> _partition_dt
   OR DATE_FORMAT(_event_time, 'HH') <> _partition_hh;

-- 社媒重复消息校验
SELECT _kafka_partition, _kafka_offset, _partition_dt, _partition_hh, COUNT(*) AS cnt
FROM ods_social_mention
GROUP BY _kafka_partition, _kafka_offset, _partition_dt, _partition_hh
HAVING COUNT(*) > 1;

-- 物流延迟解析失败校验
SELECT *
FROM ods_api_logistics_track_analyze
WHERE json_parse_status = '解析失败';

-- 社媒延迟解析失败校验
SELECT *
FROM ods_social_mention_analyze
WHERE json_parse_status = '解析失败';
```

### ODS 每日巡检清单

#### 1. 每天必查

| 巡检项                | 目的                     | 建议 SQL / 检查点                                                            |
| --------------------- | ------------------------ | ---------------------------------------------------------------------------- |
| 订单主键重复          | 防止订单明细膨胀         | `ods_order_item_cdc` 主键重复校验                                            |
| 订单状态非法          | 防止上游状态漂移未同步   | `ods_order_item_cdc` 状态非法值校验                                          |
| 库存流水零值/非法类型 | 防止库存事实失真         | `ods_inventory_txn_cdc` 零值、业务类型、状态校验                             |
| 物流分区正确性        | 保证迟到数据路由正确     | `ods_api_logistics_track` 分区错误校验                                       |
| 社媒互动指标异常      | 保证指标不出现负值       | `ods_social_mention` 互动指标异常校验                                        |
| 解析失败记录          | 判断延迟解析链路是否健康 | `ods_api_logistics_track_analyze`、`ods_social_mention_analyze` 解析失败校验 |

#### 2. 异常时排查

| 场景             | 排查重点                               | 建议 SQL / 检查点                                        |
| ---------------- | -------------------------------------- | -------------------------------------------------------- |
| 订单数据对不上   | 查空主键、时间逆序、分区错误           | `ods_order_item_cdc` 空主键、时间逆序、分区校验          |
| 库存不平         | 查零值、非法状态、重复流水             | `ods_inventory_txn_cdc` 零值、状态、主键唯一性检查       |
| 物流轨迹缺失     | 查空业务键、分区错误、重复消息         | `ods_api_logistics_track` 空业务键、分区、重复校验       |
| 社媒数据异常波动 | 查重复消息、空业务键、原始 JSON        | `ods_social_mention` 重复、空业务键、`raw_json` 留存情况 |
| 解析字段缺失     | 查原始 JSON 是否完整、是否出现字段改名 | `*_analyze` 表解析失败记录 + `raw_json` 抽样排查         |

#### 3. 发布后重点检查

| 检查项                        | 发布后关注点              | 建议 SQL / 检查点                                               |
| ----------------------------- | ------------------------- | --------------------------------------------------------------- |
| 主键与去重是否失效            | 发布后是否出现重复膨胀    | 订单/物流/社媒重复校验                                          |
| 分区是否按事件时间生成        | 迟到数据是否落错分区      | 物流/社媒分区错误校验                                           |
| 状态与枚举是否兼容新版本      | 上游新增状态是否漏配      | 订单状态、库存状态、业务类型非法值校验                          |
| JSON 解析是否兼容 Schema 变更 | 是否出现大面积解析失败    | `ods_api_logistics_track_analyze`、`ods_social_mention_analyze` |
| 关键字段映射是否正确          | 源字段到 ODS 字段是否跑偏 | 抽样核对字段映射关系表 + ODS 明细数据                           |

#### 4. 巡检执行建议

- 每天必查项建议按日调度执行，并沉淀巡检结果表或巡检日报
- 异常时排查项建议和告警联动，作为二线排查手册
- 发布后重点检查建议在上线后 30 分钟、2 小时、次日各执行一轮
- 对返回结果非空的校验 SQL，建议统一定义为“异常待处理”

## 附录3：延迟解析原始 JSON，兼容 Schema 漂移

> 说明：你提醒的这个点是对的。当前新增的物流与社媒事实虽然保留了 `raw_json`，但还缺少“延迟解析层”的显式实现。下面补充推荐的增强方案，把“先原始入湖、后解析适配”的思路落到物流和社媒事实中。

### 一、物流轨迹延迟解析增强

```sql
-- 物流解析结果表
CREATE TABLE ods_api_logistics_track_analyze (
    logistics_no STRING,
    order_id STRING,
    sku_id STRING,
    carrier_code STRING,
    node_code STRING,
    node_name STRING,
    province STRING,
    city STRING,
    station_code STRING,
    courier_name STRING,
    _event_time TIMESTAMP(3),
    json_parse_status STRING
) WITH (
    'connector' = 'paimon',
    'write-mode' = 'append-only'
);

-- 兼容 raw_json中字段 漂移
INSERT INTO ods_api_logistics_track_analyze
SELECT
    logistics_no,
    order_id,
    sku_id,
    carrier_code,
    node_code,
    COALESCE(
        node_name,
        JSON_VALUE(raw_json, '$.node_name'),
        JSON_VALUE(raw_json, '$.node_desc')
    ) AS node_name,
    JSON_VALUE(raw_json, '$.province') AS province,
    JSON_VALUE(raw_json, '$.city') AS city,
    JSON_VALUE(raw_json, '$.station_code') AS station_code,
    COALESCE(
        JSON_VALUE(raw_json, '$.courier'),
        JSON_VALUE(raw_json, '$.courier_name')
    ) AS courier_name,
    _event_time,
    CASE
        WHEN COALESCE(node_name, JSON_VALUE(raw_json, '$.node_name'), JSON_VALUE(raw_json, '$.node_desc')) IS NOT NULL
            THEN '解析成功'
        ELSE '解析失败'
    END AS json_parse_status
FROM ods_api_logistics_track;
```

### 二、社媒事实延迟解析增强

```sql
CREATE TABLE ods_social_mention_analyze (
    platform STRING,
    note_id STRING,
    author_id STRING,
    sku_id STRING,
    spu_code STRING,
    keyword STRING,
    scene_tag STRING,
    creator_level STRING,
    is_video BOOLEAN,
    device_type STRING,
    like_cnt BIGINT,
    fav_cnt BIGINT,
    comment_cnt BIGINT,
    _event_time TIMESTAMP(3),
    json_parse_status STRING
) WITH (
    'connector' = 'paimon',
    'write-mode' = 'append-only'
);

INSERT INTO ods_social_mention_analyze
SELECT
    platform,
    note_id,
    author_id,
    sku_id,
    spu_code,
    keyword,
    COALESCE(
        JSON_VALUE(raw_json, '$.scene'),
        JSON_VALUE(raw_json, '$.scene_tag')
    ) AS scene_tag,
    JSON_VALUE(raw_json, '$.creator_level') AS creator_level,
    CAST(JSON_VALUE(raw_json, '$.is_video') AS BOOLEAN) AS is_video,
    COALESCE(
        JSON_VALUE(raw_json, '$.device'),
        JSON_VALUE(raw_json, '$.client_type')
    ) AS device_type,
    like_cnt,
    fav_cnt,
    comment_cnt,
    _event_time,
    CASE
        WHEN raw_json IS NOT NULL THEN '解析成功'
        ELSE '解析失败'
    END AS json_parse_status
FROM ods_social_mention;
```

### 三、增强价值

- 原始层保留完整 `raw_json`，不因字段漂移而中断主链路
- 解析层可以按需升级，无需重建原始 ODS 入湖逻辑
- 兼容“字段改名、字段新增、旧字段保留一段时间”的真实生产场景
- 非核心字段统一在解析层处理，ODS 主表保持稳定
