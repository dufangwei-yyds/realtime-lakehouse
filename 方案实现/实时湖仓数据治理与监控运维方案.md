## 2. 治理与运维总体目标

### 2.1 治理目标

- 统一：统一命名、统一口径、统一主数据、统一指标解释
- 可追溯：任何结果都能追溯到来源表、计算链路、责任人和版本
- 可控制：数据变更有规范、口径变更有审批、发布变更有记录
- 可演进：一期先治理高价值链路，后续可扩展到更多主题域与应用

### 2.2 运维目标

- 可观测：核心作业、主题表、物化视图、接口结果均可监控
- 可预警：延迟、失败、质量异常、容量风险能够自动告警
- 可恢复：支持故障定位、作业重启、回滚与补数
- 可交接：具备巡检清单、发布手册、回滚手册、验收清单

### 2.3 一期治理运维重点

结合当前项目状态，一期重点聚焦：

1. ODS 数据接入治理
2. DIM 主数据治理
3. DWD / DWS / ADS 主题模型治理
4. 全链路质量规则和告警体系
5. Flink / Paimon / Doris / Superset 的运行保障

## 4. 数据标准治理

### 4.1 命名规范

一期已建立的命名规范继续沿用并固化：

- CDC 源表：`cdc_{table}`
- ODS 维度原始表：`ods_{table}_append`
- ODS 历史表：`ods_{table}_history`
- ODS 最新镜像：`ods_{table}_latest`
- ODS 事件流解析表：`ods_{domain}_{event}_analyze`
- DIM：`dim_{domain}_wide`
- DWD：`dwd_{domain}_detail_rt`
- DWS：`dws_{theme}_{grain}_rt`
- ADS：`ads_{theme}_{consumer}`

### 4.2 通用字段标准

所有实时链路对象建议统一以下语义：


| 字段                               | 统一解释                          |
| -------------------------------- | ----------------------------- |
| `event_time` / `_event_time`     | 业务事件发生时间                      |
| `ingest_time` / `_ingest_time`   | 数据入湖或入层时间                     |
| `partition_dt` / `_partition_dt` | 日期分区                          |
| `partition_hh` / `_partition_hh` | 小时分区                          |
| `json_parse_status`              | JSON 解析状态                     |
| `source_system`                  | 来源系统                          |
| `op_type`                        | 操作类型，如 `INSERT/UPDATE/DELETE` |


### 4.3 状态码与枚举治理

重点统一以下业务编码：

- 订单状态 `order_status`
- 渠道编码 `channel_code`
- 库存业务类型 `biz_type`
- 库存状态 `stock_status`
- 物流节点状态 `node_status`
- 爆款等级 `hot_level`
- 调拨建议动作 `suggest_action`

要求：

- 每个状态字段必须有取值说明
- 状态口径归口到业务负责人确认
- 新增状态码必须更新到数据字典和规则清单

## 5. 主数据治理

### 5.1 一期主数据治理重点

一期主数据治理核心对象为商品域：

- `M_PRODUCT`
- `M_DIM`
- `M_PRODUCT_ALIAS`
- `M_ATTRIBUTESETINSTANCE`
- `M_ATTRIBUTE`
- `dim_sku_wide`

### 5.2 治理目标

- 保证商品主数据跨层语义一致
- 保证 SKU、SPU、品牌、季节、品类等核心维度的唯一解释
- 保证 DIM 宽表作为下游统一补维入口的稳定性

### 5.3 治理要求


| 治理项    | 要求                       |
| ------ | ------------------------ |
| SKU 主键 | `sku_id` 唯一且可追溯到源头主数据    |
| SPU 归属 | 同款 SKU 的 `spu_code` 必须统一 |
| 别名属性   | 颜色、尺码等通过规范透视逻辑统一展开       |
| 扩展属性   | 材质、产地等统一从属性集实例透出         |
| 停用商品   | `is_active` 等状态必须可驱动下游过滤 |


### 5.4 当前建议

- `dim_sku_wide` 作为一期统一商品维度入口继续保留
- 与业务继续确认 ODS / DIM 字段后，再做后续小版本字段补强

## 6. 模型治理

### 6.1 分层边界治理


| 层级  | 应做什么            | 不应做什么         |
| --- | --------------- | ------------- |
| ODS | 贴源接入、标准元数据、延迟解析 | 重度聚合、复杂业务指标固化 |
| DIM | 统一维度与退化宽表       | 复杂事实逻辑        |
| DWD | 明细宽表化、维表关联、幂等处理 | 面向消费端的专题口径    |
| DWS | 主题聚合、公共指标沉淀     | 面向单一页面的展示层逻辑  |
| ADS | 看板型 / 接口型结果集    | 重新发明底层指标定义    |


### 6.2 主键与粒度治理

要求：

- 每张表必须明确主键
- 每张表必须明确粒度说明
- 每张表必须明确服务对象

重点治理点：

- ODS 层主键与分区是否冲突
- DWD 层是否存在重复膨胀
- DWS 层是否存在“粒度不清”
- ADS 层是否存在“一表多责”

### 6.3 变更治理

模型变更建议统一走以下流程：

1. 变更申请
2. 影响分析
3. 业务确认
4. 开发验证
5. 发布窗口执行
6. 发布后观测

## 7. 指标治理

### 7.1 一期核心指标域

围绕两大核心场景，一期重点治理以下指标：

- ATP
- 可用库存
- 锁定库存
- 在途库存
- 订单需求量
- 缺货风险标识
- 调拨优先级
- 销售额
- 订单数
- 浏览数
- 点击数
- 加购数
- 社媒热度
- 爆款综合指数
- 追单建议标识

### 7.2 指标治理要求


| 治理项  | 要求                     |
| ---- | ---------------------- |
| 指标定义 | 明确定义、计算公式、统计口径         |
| 指标粒度 | 明确分钟级、小时级、店仓级、SKU 级等   |
| 指标来源 | 明确来源 DWD / DWS / ADS 表 |
| 责任归属 | 明确业务 owner 和数据 owner   |
| 口径版本 | 变更必须留版本记录              |


### 7.3 指标变更原则

- 能在 DWS 固化的指标，不在 ADS 重复定义
- ADS 只负责消费组织和解释增强
- 排行、等级、建议类结果必须附带可解释字段

## 8. 数据质量治理

### 8.1 质量目标


| 维度  | 目标                    |
| --- | --------------------- |
| 完整性 | 关键字段不缺失               |
| 唯一性 | 主键不重复，消息不膨胀           |
| 及时性 | 满足分钟级或小时级 SLA         |
| 一致性 | 上下游结果可核对              |
| 合法性 | 数值、状态、时间符合业务规则        |
| 稳定性 | 作业重启、迟到数据、Schema 漂移可控 |


### 8.2 分层质量规则

#### ODS

- 主键非空
- 时间字段非空
- Kafka offset / partition 正常
- 分区按事件时间路由
- JSON 解析成功率可观测

#### DIM

- SKU 唯一
- 关键维度字段完整
- 停用状态可正确识别

#### DWD

- 明细主键不重复
- 维度补齐成功率达标
- 事件时间字段可用
- 宽表字段不出现大面积空值

#### DWS

- 公共粒度正确
- 核心指标计算正确
- 聚合结果与 DWD 抽样核对一致

#### ADS

- 看板表与接口表结果一致性可核对
- 解释字段与建议字段完整
- 高频结果集刷新稳定

### 8.3 质量处置策略


| 问题类型  | 处置方式                       |
| ----- | -------------------------- |
| 可自动兜底 | 默认值填充、状态映射、轻度修正            |
| 可隔离问题 | 脏数据隔离表、异常样本记录              |
| 需自动告警 | 延迟超 SLA、重复率异常、条数突变、解析失败率升高 |
| 需人工介入 | 主数据口径变化、业务规则变更、跨层结果明显不一致   |

## 12. 分层监控策略

### 12.1 ODS 监控

重点监控：

- CDC 延迟
- Topic 消费延迟
- JSON 解析失败率
- ODS 入湖条数
- 分区写入正确性
- Paimon Sink 成功率

### 12.2 DIM 监控

重点监控：

- Latest 表到 DIM 宽表写入延迟
- SKU 宽表行数波动
- 关键字段空值率
- 停用商品识别是否生效

### 12.3 DWD 监控

重点监控：

- DWD 作业状态
- Lookup Join 命中率与异常率
- 明细主键重复率
- 宽表字段补齐率
- 事件时间分布异常

### 12.4 DWS 监控

重点监控：

- 窗口作业延迟
- 聚合结果时效
- 指标波动异常
- Doris Sink 写入成功率

### 12.5 ADS 监控

重点监控：

- ADS 刷新完成时间
- Dashboard 结果表更新时间
- API 结果表更新时间
- 物化视图刷新状态
- 接口查询 SLA

## 14. 告警分级与处置

### 14.1 告警分级


| 级别  | 说明      | 示例                          |
| --- | ------- | --------------------------- |
| P1  | 业务主链路中断 | Flink 主作业失败、ADS 无结果         |
| P2  | 结果严重异常  | DWS 延迟超 SLA、Doris MV 长时间不刷新 |
| P3  | 风险预警    | 快照膨胀、消费堆积、小文件增长             |
| P4  | 提示类     | 轻微波动、趋势预警                   |


### 14.2 告警处理原则

- P1：立即拉起处理，需人工介入
- P2：当日处理，必要时降级消费
- P3：纳入巡检和优化计划
- P4：观察并记录

### 14.3 告警路由建议


| 告警类型 | 路由对象         |
| ---- | ------------ |
| 作业失败 | 数据开发 + 运维    |
| 结果延迟 | 数据开发 + 业务接口人 |
| 存储容量 | 运维           |
| 指标异常 | 数据开发 + 业务负责人 |

## 15. 巡检机制

### 15.1 每日巡检

- Flink 主作业是否全部运行
- Checkpoint 是否连续成功
- Kafka 是否存在明显积压
- Paimon 是否存在快照或小文件异常
- Doris FE / BE 是否正常
- ADS 结果是否按时刷新

### 15.2 每周巡检

- 表空间与 HDFS 容量趋势
- Doris 物化视图命中率与刷新状态
- 异常告警数量统计
- 测试环境配置漂移检查

### 15.3 每月巡检

- 治理规则覆盖率复盘
- 数据字典与实际实现一致性复盘
- 指标口径变更复盘
- 历史问题清单关闭情况复盘

## 16. 发布、回滚与补数

### 16.1 发布原则

- 先测试环境验证，再进入验收或生产
- 发布前必须确认：
  - 表结构变更
  - 主键变更
  - 分区变更
  - 指标口径变更

### 16.2 回滚原则


| 场景             | 回滚建议            |
| -------------- | --------------- |
| Flink SQL 逻辑错误 | 回滚到上一个稳定 SQL 版本 |
| DWS / ADS 结果异常 | 回滚结果表刷新逻辑       |
| 主数据错误          | 优先源头修正后重放       |
| 消费侧展示异常        | 先隔离错误结果，再恢复下游消费 |


### 16.3 补数原则

- 补数优先回到最早可控层
- ODS 问题优先从 ODS 重放
- 指标问题优先从 DWD / DWS 重新计算
- 补数必须记录影响范围、时间窗口、责任人

## 17. 权限与安全治理

### 17.1 权限原则

- 环境隔离：DEV / SIT / UAT / PROD 不混用
- 最小权限：开发、测试、运维、BI 分角色授权
- 敏感配置外置：密码、密钥不写入正式文档和代码

### 17.2 当前建议

- Superset、MySQL、Doris 采用独立业务账号
- Flink 发布账号与日常查询账号分离
- 结果表如涉及敏感字段，需预留脱敏能力
