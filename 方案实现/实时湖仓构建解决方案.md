# 服装行业实时湖仓构建解决方案

> 适用对象：服装品牌企业 / 零售企业 / 供应链数字化团队
> 方案目标：从企业级最佳实践视角，输出一套可评审、可实施、可验收、可持续迭代的实时湖仓建设蓝图
> 核心技术栈：`Flink CDC / Kafka / Flink SQL & DataStream API (Flink 1.20.x) / Paimon / Doris 4.x / Superset`
> 扩展方向：`Apache Fluss` 预研验证、`AI 驱动的数据建模/治理/分析重构`

---

## 1. 项目总览

### 1.1 建设背景

服装行业具备明显的“多渠道、多款色码、多组织、多季节、多波次”特征，核心痛点通常集中在以下几个方面：

- ERP、WMS、OMS、POS、会员、埋点、外部物流与社媒数据割裂
- 库存、订单、履约、商品热度等关键数据延迟高，难以支撑实时决策
- 指标口径不统一，跨部门对同一业务对象的理解不一致
- 传统离线数仓无法满足小时级甚至分钟级运营调度需求
- 数据平台偏技术导向，缺少可验收、可运营、可推广的企业级实施方法

### 1.2 建设目标

本项目拟构建一套服装行业实时湖仓平台，实现：

- 打通 ERP、订单、库存、物流、社媒、埋点等多源异构数据
- 基于 `Kafka + Flink + Paimon + Doris` 建设流批一体、明细与指标兼顾的实时湖仓
- 支撑库存、订单履约、爆款识别、经营分析等核心业务场景
- 形成标准化的数据分层、数据治理、指标体系与开发规范
- 通过 `Doris 4.x` 验证 AI 相关能力，通过 `Apache Fluss` 完成新一代实时存储链路预研

### 1.3 项目范围

#### 范围内

- 规划与调研
- 实时湖仓架构设计
- 基础设施与环境建设
- ODS/DWD/DWS/ADS 分层建设
- 主数据治理与指标体系建设
- 2 个核心业务场景落地
- 报表服务与运营看板建设
- 测试、验收、上线、运维与优化机制建设

#### 范围外

- 所有历史数据一次性全面重构
- 所有外围系统主流程改造
- 大模型平台单独建设
- 全量替换企业现有 BI 体系

## 3. 规划与调研

### 3.1 项目组织与治理机制

建议建立四层治理机制：

- 项目委员会：业务负责人、IT 负责人、数据负责人，负责重大决策
- 项目管理办公室 PMO：负责计划、风险、资源、验收推进
- 业务专题组：商品、供应链、零售、电商、会员等域专家参与口径确认
- 数据交付组：架构、数据开发、运维、测试、BI、治理人员负责落地

### 3.2 业务调研五大场景

#### 场景 A：全渠道实时库存可视与智能调拨

- 业务目标：统一总部仓、区域仓、门店、电商前置仓库存，支撑“一盘货”
- 核心价值：降低缺货率、提升履约效率、减少超卖与滞销

#### 场景 B：爆款趋势识别与快速追单

- 业务目标：融合销量、加购、收藏、社媒热度，快速识别潜力爆款
- 核心价值：缩短追单决策周期，提高首单售罄效率

#### 场景 C：订单履约与物流异常实时预警

- 业务目标：跟踪订单从下单、审核、出库、揽收、签收的全流程
- 核心价值：提升履约透明度，降低延迟发货与投诉率

#### 场景 D：门店实时经营效能分析

- 业务目标：按门店、导购、时段、品类实时分析客流、成交、连带率、人效
- 核心价值：支持门店运营和区域督导实时纠偏

#### 场景 E：消费者行为洞察与人货场联动

- 业务目标：融合埋点行为、会员标签、交易数据，识别转化漏斗与兴趣偏好
- 核心价值：支撑精准营销、个性化推荐、活动复盘

### 3.3 核心业务场景筛选

建议一期优先落地以下 2 个场景：

| 入选场景                     | 入选原因                                       | 依赖数据成熟度 | 业务价值 | 技术复杂度 |
| ---------------------------- | ---------------------------------------------- | -------------- | -------- | ---------- |
| 全渠道实时库存可视与智能调拨 | 直接影响销售与履约，跨部门关注度高             | 高             | 高       | 中         |
| 爆款趋势识别与快速追单       | 贴合服装行业快反需求，可体现实时湖仓与 AI 价值 | 中             | 高       | 中高       |

### 3.4 调研输出清单

- 业务流程图：商品、订单、库存、物流、营销、社媒
- 系统关系图：ERP/WMS/OMS/POS/CRM/埋点/第三方平台
- 数据盘点表：系统、表、字段、更新频率、接入方式、责任人
- 指标清单：名称、定义、公式、粒度、口径归属
- 问题清单：缺字段、脏数据、手工环节、系统断点
- 一期与二期范围划分清单

## 4. 方案架构设计

### 4.1 总体架构

```text
业务系统/外部数据源
  ├─ ERP / OMS / WMS / POS / CRM
  ├─ 第三方物流 API
  ├─ 小红书爬虫数据
  └─ App / 小程序 / 官网埋点

接入层
  ├─ Flink CDC
  ├─ Kafka
  ├─ API采集服务 / 爬虫采集服务
  └─ 埋点SDK / 日志采集

计算与湖仓层
  ├─ Flink SQL / DataStream API
  ├─ Paimon ODS / DWD / DWS
  ├─ 维度拉链 / 宽表 / 实时特征
  └─ Fluss 预研链路

服务与消费层
  ├─ Doris 4.x 明细查询 / 聚合分析 / AI实验
  ├─ Superset 可视化看板
  ├─ 指标服务 / 数据 API
  └─ 告警推送 / 运营应用
```

### 4.2 分层架构

| 层级 | 目标                           | 存储建议       | 说明                           |
| ---- | ------------------------------ | -------------- | ------------------------------ |
| ODS  | 原始入湖、保留业务语义         | Paimon         | 接近源系统结构，做轻度标准化   |
| DWD  | 明细清洗、事实沉淀、维度拉齐   | Paimon         | 统一主键、时间、状态、组织口径 |
| DWS  | 主题汇总、实时宽表、指标域模型 | Paimon + Doris | 支撑秒级/分钟级分析            |
| ADS  | 面向场景的数据服务层           | Doris          | 面向报表、API、专题应用        |

### 4.3 技术角色分工

| 组件          | 角色                                                 |
| ------------- | ---------------------------------------------------- |
| Flink CDC     | 从 ERP/OMS/WMS 等关系型系统采集增量变更              |
| Kafka         | 统一解耦消息总线，承载订单、埋点、社媒、API 数据     |
| Flink SQL/API | 实时清洗、维表关联、状态计算、窗口聚合、特征生成     |
| Paimon        | 湖仓主存储，承载主明细、维度、增量快照、流批一体加工 |
| Doris 4.x     | 高并发分析查询、实时主题服务、AI 能力实验载体        |
| Superset      | 经营看板、专题分析、业务自助查询                     |
| Apache Fluss  | 新一代实时分析存储预研，对比 Kafka/Paimon 链路表现   |

### 4.4 关键设计原则

- 主链路优先稳定：`Kafka + Flink + Paimon + Doris` 作为正式生产链路
- 数据湖与分析引擎解耦：Paimon 承接事实沉淀，Doris 承接交互查询
- 先统一主数据，再统一指标
- 先建设高价值场景，再平台化推广
- 所有模型、口径、规则都要文档化并可验收

## 6. 数据接入与数据治理

### 6.1 ODS 层接入范围

一期 ODS 层必须覆盖以下数据：

| 数据域     | 数据对象                                         | 来源           | 接入方式          | 时效目标     |
| ---------- | ------------------------------------------------ | -------------- | ----------------- | ------------ |
| ERP 主数据 | 商品、品牌、季节、品类、组织、门店、仓库、供应商 | ERP            | Flink CDC         | 分钟级       |
| 订单事实   | 销售单、订单明细、退货单、支付单、发货单         | ERP/OMS/POS    | Flink CDC / Kafka | 秒级到分钟级 |
| 库存流水   | 出入库、调拨、锁定、占用、在途、盘点             | ERP/WMS        | Flink CDC         | 分钟级       |
| 物流数据   | 揽收、运输、签收、异常节点、时效状态             | 第三方物流 API | API 拉取 + Kafka  | 5-15 分钟    |
| 社媒数据   | 小红书笔记、互动量、关键词、达人、商品关联       | 爬虫平台       | 爬虫采集 + Kafka  | 小时级       |
| 用户行为   | 浏览、曝光、点击、加购、收藏、下单、支付、分享   | 埋点平台       | SDK/日志 → Kafka  | 秒级         |

### 6.2 接入策略

#### 关系型业务库

- 使用 `Flink CDC` 抽取增量变更
- 按业务域拆分 Source 作业，避免单点超大任务
- 表结构变化纳入变更评审流程

#### 行为日志与事件流

- 埋点事件统一进入 Kafka
- 统一事件模型：事件时间、用户、设备、会话、页面、动作、业务对象
- 支持乱序、水位线和幂等去重

#### API 与爬虫数据

- API 拉取结果先写 Kafka 原始 Topic，再入 ODS
- 小红书爬虫数据必须增加合法合规审查、频率控制、脱敏与来源标识

### 6.3 数据标准化治理

重点治理对象：

- 时间标准：业务时间、事件时间、入湖时间、处理时间统一定义
- 编码标准：SKU、SPU、店仓、组织、渠道、订单号、物流单号统一编码体系
- 主数据标准：商品主数据、门店主数据、组织主数据、供应商主数据统一治理
- 状态标准：订单状态、库存状态、物流状态统一枚举映射
- 命名标准：库、表、字段、Topic、任务、看板统一命名规范

### 6.4 数据质量治理

| 治理维度 | 规则示例                               |
| -------- | -------------------------------------- |
| 完整性   | 主键不能为空、关键业务字段不能为空     |
| 唯一性   | 订单号 + 明细号组合唯一                |
| 一致性   | ERP 商品主数据与 OMS 商品编码映射一致  |
| 时效性   | ODS 入湖延迟不超过 5 分钟              |
| 合法性   | 库存数量不可出现非法负值，时间不可逆序 |
| 稳定性   | 每日字段变更必须有评审记录             |

### 6.5 安全与合规

- 用户标识、手机号、地址等敏感字段脱敏
- 第三方数据采集遵循法务与平台合规要求
- 开发、测试、生产环境权限隔离
- 行级/列级权限在 Doris 与 BI 层同步控制

## 7. 湖仓核心层建设

### 7.1 数据分层实施路径

| 顺序 | 层级 | 建设重点                                         |
| ---- | ---- | ------------------------------------------------ |
| 1    | ODS  | 全量接入、轻度清洗、保留原始语义与湖仓元数据     |
| 2    | DIM  | 主数据统一、SCD2 拉链、Latest 镜像、维度退化宽表 |
| 3    | DWD  | 订单、库存、物流、行为、社媒事实建模，单表即分析 |
| 4    | DWS  | 公共粒度汇总、指标固化、主题宽表与服务层预聚合   |
| 5    | ADS  | 面向库存调拨、爆款识别、看板服务的专题层         |

建议按“`ODS 先稳接入 -> DIM 先做统一维度 -> DWD 打宽明细 -> DWS 固化指标 -> ADS 服务业务`”推进，避免直接从 ODS 跳 DWS 导致口径失控。

### 7.2 核心建模规范与方法论

#### 7.2.1 ODS 事件流与 CDC 建模规范

结合 `实时湖仓事件流建模实现方案.sql`，ODS 层建议明确以下规范：

- 保持原始：不做破坏性聚合，不丢原始 JSON，不在 ODS 固化复杂业务口径
- 注入元数据：每张 ODS 事实表必须补充 `_event_time`、`_ingest_time`、`_partition_dt`、`_kafka_offset`、`_kafka_partition` 等湖仓元字段
- 延后解析：对高漂移、高并发事件流采用“关键字段列化 + 原始 JSON 存根”模式
- 分区存储：埋点与行为流建议按 `dt + hh` 二级分区，避免大促期间单分区文件爆炸
- Schema 演进：优先采用 Paimon 模式演进能力，降低前端埋点变更对任务稳定性的影响
- 轻度净化：进入 ODS 前只做基础空值处理、时间标准化、非法字段兜底，保证后续关联不被脏数据放大

#### 7.2.2 DIM 维度建模规范

结合 `实时湖仓维度建模实现方案.sql`，维度层建议采用“四层维度建模法”：

| 维度层次        | 作用                        | 适用场景                 |
| --------------- | --------------------------- | ------------------------ |
| Append 表       | 保留原始维度变更轨迹        | CDC 增量沉淀、审计追溯   |
| SCD2 History 表 | 保存历史版本，支持时间回溯  | 价格、季节、组织归属变化 |
| Latest 镜像表   | 存储最新状态，支持高效 Join | DWD Lookup Join          |
| 退化宽表        | 物理打平常用维度字段        | 降低 DWD/DWS Join 开销   |

维度层关键规则：

- 维度不只解决“是什么”，还要解决“在当时是什么”
- 高变维度必须有历史表，禁止只有 Latest 没有 History
- 常用关联维度要做物理退化宽表，避免下游重复多表 Join
- 维度主键、业务主键、版本起止时间必须统一命名与语义
- 所有维度都要区分高基数维与低基数维，分别规划存储与缓存策略

#### 7.2.3 DWD 明细建模规范

结合 `实时湖仓DWD建模实现方案.sql`，DWD 层是整个实时湖仓的“腰部”，必须坚持以下原则：

- 单表即分析：事实流和常用维度做宽表化，尽量让下游查询不再频繁关联维表
- 事件时间优先：所有统计与关联以 `event_time` 为准，而不是处理时间
- 标准口径统一：订单状态、金额精度、渠道、组织、商品编码统一映射
- 按业务过程建模：如下单、支付、退款、出库、签收等分开建模，不混成“大而全万能表”
- 幂等去重：即使 ODS 已去重，DWD 仍要通过主键表或去重规则再次兜底
- 事实关联事实必须显式限定时间区间，如点击后 30 分钟内下单才算转化

#### 7.2.4 DWS 汇总建模规范

结合 `实时湖仓DWS建模实现方案.sql`，DWS 层重点不是“继续堆表”，而是沉淀公共粒度和统一指标：

- 先定义公共粒度，再做聚合，如 `[小时 + 组织 + SKU]`、`[天 + 渠道 + 品类]`
- 所有核心指标在 DWS 固化口径，避免每张报表各算各的
- 区分加和性指标与非加和性指标，分别选择 SUM、Bitmap、HLL 等聚合策略
- DWS 覆盖 80% 通用查询，复杂探索性分析则由 Doris 外表 + 物化视图兜底

#### 7.2.5 ADS 应用建模规范

结合当前建设目标与后续 `实时湖仓ADS建模实现方案.md` 的落地方向，ADS 层不是“再做一层汇总”，而是把 DWS 的公共能力组织成可直接服务业务的主题数据集、接口结果集和 BI 消费模型：

- 面向场景建模：ADS 必须直接对应业务消费对象，如库存驾驶舱、调拨建议、爆款榜单、追单建议，而不是按技术组件命名
- 面向消费端组织字段：字段设计要服务 Superset 看板、接口返回、告警推送，优先保留筛选字段、排序字段、展示字段、钻取字段
- 一张 ADS 只服务一类稳定场景：避免把看板字段、接口字段、明细排查字段全部混在一张“万能 ADS 表”中
- 指标不重复发明：ADS 原则上不重新定义核心业务口径，而是承接 DWS 固化指标，在必要时补充业务规则标签和结果解释字段
- 查询体验优先：ADS 模型要优先保证秒级查询、可分页、可筛选、可排序，必要时通过 Doris 聚合模型、物化视图、分区裁剪进一步加速
- 服务化输出：同一主题应同时考虑“BI 看板视图”和“接口服务结果集”两类消费方式，避免后期再二次加工
- 可解释性：所有推荐类、预警类、排行类 ADS 结果都要带上触发原因、规则分值、更新时间，避免业务方只看到结论看不到依据

### 7.3 分层建设方法与落地策略

#### 7.3.1 ODS 建设方法

- ERP/WMS/OMS 等业务库通过 `Flink CDC -> Paimon ODS`
- Kafka 行为流通过 `Kafka -> Flink SQL -> Paimon Append ODS`
- 高并发行为表采用 Append-Only + 动态分桶
- 有重复风险的消息流增加 Paimon 主键去重表

#### 7.3.2 DIM 建设方法

- 商品、价格、季节、门店、组织等维度优先建设
- 基于 CDC 构建 Append 表，再派生 SCD2 History 与 Latest 镜像
- 高频 Join 维度做退化宽表，例如 `dim_sku_wide`
- Lookup Join 场景启用缓存，并控制 TTL 和缓存行数

#### 7.3.3 DWD 建设方法

- 订单、库存、行为、物流等事实在 DWD 完成清洗、统一、打宽
- 事实流与事实流通过 Interval Join / Window Join 处理业务链路
- 事实流与维度通过 Lookup Join 处理主数据补全
- DWD 明细优先落 Paimon，保证“可重算、可回溯、低成本”

#### 7.3.4 DWS 建设方法

- 固定高频指标通过 Flink 预聚合或 Doris 聚合表落地
- 灵活分析主题优先走 Doris 读取 Paimon DWD 外表 + Async MV 自动加速
- DWS 既可以落 Paimon，也可以把热点主题落 Doris 形成服务层

#### 7.3.5 ADS 建设方法

- ADS 优先落 Doris，直接面向 Superset BI 看板、数据服务接口和经营专题应用
- ADS 输入以 DWS 为主、DWD 为辅：稳定指标来自 DWS，必要钻取字段或解释字段可补充引用 DWD 明细
- 围绕一期两大核心场景优先建设专题 ADS：
  - 库存主题：库存驾驶舱、缺货预警、调拨建议、店仓健康度
  - 爆款主题：爆款排行、趋势识别、追单建议、热度归因
- 对高频访问 ADS 结果集，优先采用 Doris 内表 + 异步物化视图，降低看板高并发对底层明细的冲击
- 对接口服务场景，ADS 要同步设计分页键、排序键、过滤条件和更新时间字段，保证前端/服务端可直接消费
- Superset 已完成搭建，方案层只需强调“ADS 通过 Doris 直连 Superset 提供数据集与图表支撑”，不再展开部署过程
- ADS 建设完成后，应同步输出对应的数据服务清单、接口口径说明和 BI 数据字典，保证业务交付闭环

### 7.4 建模过程中的血泪教训与规避原则

结合现有实现脚本，建议在方案中明确以下“反模式”：

| 血泪教训                               | 典型后果                    | 规避方式                                   |
| -------------------------------------- | --------------------------- | ------------------------------------------ |
| ODS 过早做强解析                       | 埋点一改字段就挂任务        | 采用“关键字段列化 + raw_json 存根”         |
| 行为流只按天分区                       | 大促期间单分区文件数爆炸    | 使用 `dt + hh` 二级分区                    |
| 未注入 Kafka offset 等元数据           | 无法排查重复与丢数          | ODS 强制补齐湖仓元数据                     |
| DWD 直接拿脏数据 Join 维表             | Join 倾斜、空关联、状态膨胀 | 先做空值过滤和默认值兜底                   |
| DWD 只用处理时间                       | 迟到数据统计错误            | 统一使用事件时间和水位线                   |
| Lookup Join 不做缓存                   | 维表查询打爆存储            | 启用缓存并设置 TTL                         |
| DWD 不做幂等兜底                       | 任务重启后明细膨胀          | 主键表 / deduplicate 再兜底                |
| 试图把所有逻辑塞进 DWS                 | 口径混乱、难维护            | 明细逻辑放 DWD，指标固化放 DWS             |
| 直接拿湖上大明细做复杂 distinct        | 查询性能差                  | 用 Doris 物化视图做预计算                  |
| ADS 直接透传 DWD 明细给看板            | 查询慢、字段乱、口径飘      | 通过 DWS/ADS 主题模型收口，按场景组织字段  |
| 一个 ADS 表同时服务看板、接口、排查    | 结构失控、变更牵一发动全身  | 拆分看板型 ADS、接口型 ADS、排查型明细入口 |
| 排行/推荐类 ADS 只有结果没有依据       | 业务不信任、难运营          | 补充规则标签、分值来源、更新时间、触发原因 |
| 接口直接复用 BI 查询 SQL               | 性能不稳定、难治理          | 为接口单独设计结果集和分页排序键           |
| Superset 数据集直接连 DWD/DWS 明细大表 | 高并发下拖垮查询引擎        | 让 Superset 优先消费 Doris ADS 主题表或 MV |

### 7.5 维度建模设计

核心维度建议：

- 商品维：SPU、SKU、款色码、品牌、系列、季节、品类、上市波段
- 组织维：集团、大区、直营网店、加盟店、仓库、前置仓
- 渠道维：电商平台、门店、直播、分销
- 时间维：日、周、月、季度、促销周期
- 用户维：会员等级、渠道来源、消费层级、标签
- 供应链维：供应商、工厂、产线、物流商

关键要求：

- 主数据维表优先落 Paimon，支持 SCD2 历史追踪
- 统一 surrogate key / business key 使用规则
- 所有维度字段必须有中文释义与枚举说明
- 高频维度必须同时具备 Latest 镜像和退化宽表能力
- 维度宽表要服务于 DWD 单表分析，而不是仅为了“维度完整”而堆字段

### 7.6 事实建模设计

核心事实建议：

- 订单事实：下单、支付、发货、签收、退货
- 库存流水事实：入库、出库、调拨、锁定、释放、盘点、在途
- 物流轨迹事实：节点事件、异常事件、时效事件
- 行为事实：曝光、点击、浏览、加购、收藏、支付转化
- 社媒事实：笔记发布、点赞、收藏、评论、关键词热度

事实层落地要求：

- 每张事实表必须说明业务粒度、主键、事件时间、状态流转逻辑
- 事实表必须明确是否 append-only、upsert 或 deduplicate
- 订单与行为类事实必须区分业务时间和入湖时间
- 明细宽表字段控制在“够分析、不失控”的范围内，避免无限膨胀

### 7.7 两大核心场景建设

#### 场景一：全渠道实时库存可视与智能调拨

##### 业务目标

- 实时识别缺货风险、超卖风险、滞销风险

`ERP/WMS库存流水 + OMS订单占用 + 门店库存 + 在途库存 + 物流节点`
`-> Kafka / Flink`
`-> Paimon DWD 库存事实`
`-> DWS 实时库存宽表`
`-> Doris ADS 库存调拨主题`

##### 关键指标

- ATP 可承诺库存
- 缺货率

- 调拨响应时长

##### 业务目标

- 实时识别高潜力商品

- 输出追单建议与风险提示

##### 核心数据链路

`订单成交 + 加购收藏 + 页面行为 + 小红书热度 + 库存余量`

##### 核心指标

- 浏览转化率
- 加购率 / 收藏率
- 社媒热度指数

##### 交付成果

- 追单建议分析报表

### 7.8 Paimon 建设要点

- ODS 使用 Append/Primary Key 表结合，按业务对象选择写入模式
- 维表采用主键表 + SCD2 历史策略
- 大事实表按业务日期、渠道、组织等维度规划分区
- 制定 Compact、Snapshot Retention、Bucket 策略
- 明细沉淀优先在 Paimon，面向交互查询再同步至 Doris
- 行为流 Append 表优先使用动态分桶，高并发主键表显式设置 bucket
- Paimon 承担“数据基石”角色，确保湖上数据持久、低成本、可重算

### 7.9 Doris 4.x 建设要点

- ADS 层主题模型落 Doris，支撑秒级查询
- 对复杂明细与汇总混合查询设计宽表与星型模型并存
- 验证 Doris 4.x AI 相关能力，服务自然语言分析、智能问答、SQL 助手等探索场景

### 7.10 Doris 物化视图应用设计

结合 `实时湖仓DWS建模实现方案.sql` 与现有最佳实践，Doris 物化视图不应只作为“性能补丁”，而应成为湖仓查询加速的标准能力。

#### 物化视图定位

- 对接 Paimon DWD/DWS 外表，承接高频聚合与去重指标加速
- 通过透明改写，让业务查询原表时自动命中物化视图
- 减少重复 ETL，避免为每个报表单独造一张汇总表

#### 推荐应用场景

| 场景            | 物化视图类型        | 说明                                    |
| --------------- | ------------------- | --------------------------------------- |
| 季节销售趋势    | 异步物化视图        | 按天 + 季节 + 品类预聚合销售额          |
| 库存调拨驾驶舱  | 异步物化视图        | 按组织 + SKU + 小时聚合 ATP、缺货预警数 |
| 爆款识别        | 异步物化视图        | 按小时预聚合销量、浏览、加购、社媒热度  |
| UV/下单用户去重 | Bitmap/HLL 物化视图 | 加速去重类指标计算                      |
| 大屏战报        | 预聚合表 + MV       | 对固定看板做多层加速                    |

#### 建设策略

- 主明细保留在 Paimon，Doris 通过 Catalog 挂载外表
- 高频固定查询走 Doris 内部聚合表或异步物化视图
- 中高频灵活查询优先采用 Async MV，降低实时 ETL 复杂度
- 去重指标优先用 Bitmap/HLL 预计算，避免反复扫描明细
- 通过 `REFRESH AUTO` 或定时刷新保障准实时性

#### 建议落地模型

| 模型类型      | 示例                             | 适用特点                   |
| ------------- | -------------------------------- | -------------------------- |
| Aggregate 表  | `dws_sales_org_sku_1h`           | 高频固定报表，极致查询性能 |
| Async MV      | `mv_season_category_sales_daily` | 外表透明改写，维护成本低   |
| Bitmap/HLL MV | `mv_hot_sku_uv_1h`               | UV/去重类指标性能提升明显  |
| 宽表加速 MV   | `mv_inventory_dispatch_rt`       | 面向库存和履约驾驶舱       |

#### 设计注意事项

- 物化视图的粒度要围绕公共查询路径，而不是围绕单一临时报表
- 明细层口径不稳定前，不建议过早铺太多 MV
- 对跨天、跨小时聚合要明确刷新策略与迟到数据补偿策略
- 必须定期检查 MV 命中率、刷新耗时、存储占用，避免“视图失控”

| 告警机制 | 延迟超阈值、数据质量异常、查询失败、资源异常 |
| 发布管理 | SQL/作业/模型版本管理、灰度发布、回滚策略    |
| 成本优化 | Topic 保留、Compact 策略、冷热分层、查询优化 |
| 月度复盘 | SLA、业务价值、需求迭代、故障分析            |

### 10.3 迭代路线图

#### 一期

- 打通核心数据源
- 建成 2 个业务场景
- 建立指标体系、开发规范、监控治理机制

#### 二期

- 扩展订单履约、门店经营、会员增长等主题
- 建设更多实时服务接口
- 推动数据产品化与资产目录化

#### 三期

- 引入 AI 驱动的数据开发与分析能力
- 评估 Fluss 是否转入部分生产链路
- 建设语义层和自助分析助手

## 11. Apache Fluss 预研与实验测试方案

### 11.1 预研目标

Apache Fluss 在本项目中不建议直接替代主生产链路，而建议作为专项 PoC 组件进行对比验证，重点关注：

- 实时写入吞吐能力
- 低延迟读取能力
- 与 Flink 的集成效率
- 对实时明细表/实时特征表的适配度
- 运维复杂度与资源成本

### 11.2 建议验证场景

| 场景           | 验证内容                                |
| -------------- | --------------------------------------- |
| 行为流实时特征 | 对比 Kafka + Paimon 与 Fluss 的链路时延 |
| 爆款指数特征表 | 验证多维实时特征的读写表现              |
| 实时查询接口   | 评估面向热点明细的读取性能              |

### 11.3 验证方式

- 建立旁路实验链路，不影响正式生产链路
- 选择单一主题先跑小规模 PoC
- 统计吞吐、延迟、恢复时间、资源成本、稳定性
- 形成对比报告，为二期技术演进提供决策依据

### 11.4 输出物

- Fluss PoC 部署文档
- 测试数据集与作业脚本
- 性能对比报告
- 是否引入二期架构的评审结论

## 12. 基于 AI 能力重构实时湖仓的探索

### 12.1 总体思路

AI 不应被孤立为“报表问答”能力，而应重构实时湖仓的三个核心环节：

- 智能建模
- 智能治理
- 智能分析与运营闭环

### 12.2 AI 落地方向

#### 方向一：智能建模助手

- 自动解析源表结构、生成 ODS/DWD 建模草案
- 生成 Flink SQL 模板、字段映射建议、维度归类建议
- 识别同义字段、重复表、潜在主外键关系

#### 方向二：智能数据治理

- 自动识别异常波动、缺失字段、漂移指标
- 基于规则 + 模型做数据质量根因分析
- 为字段补充释义、标签、血缘说明

#### 方向三：智能经营分析

- 面向 Doris 4.x 验证自然语言问数能力
- 构建“爆款原因解释”“库存异常解释”“履约异常解释”能力
- 支持运营人员通过问答方式获取趋势洞察和建议

#### 方向四：智能调度与运维

- 对 Flink 作业异常自动归因
- 对 Kafka 积压、状态膨胀、写入失败给出优化建议
- 自动生成巡检日报与发布风险提示

### 12.3 AI 建设原则

- 先辅助，再自动化
- 先规则增强，再模型增强
- 高风险动作必须有人审
- 所有 AI 输出都要可追溯、可解释、可回滚

## 14. 风险与控制措施

| 风险             | 表现                     | 应对措施                             |
| ---------------- | ------------------------ | ------------------------------------ |
| 主数据不统一     | 商品/店仓编码打不通      | 优先建设 MDM 映射与标准口径          |
| 源系统字段质量差 | 脏数据、缺字段、时间不准 | 接入前做字段评估，建立补录与兜底规则 |
| 业务口径反复变化 | 指标持续返工             | 建立指标评审机制，版本化管理         |
| 流式链路不稳定   | 积压、延迟、状态膨胀     | 分域拆作业、加强监控、压测先行       |
| 外部数据合规风险 | 爬虫、外部接口使用不规范 | 引入法务评审、访问频控、脱敏         |
| AI 期望过高      | 过早追求全自动           | 明确 AI 为增强能力，分阶段落地       |

## 附录 D：实时湖仓开发规范（建议版）

### D.1 命名规范

- Kafka Topic：`rt_{domain}_{object}_{event}`
- ODS 表：`ods_{source}_{table}`
- DWD 表：`dwd_{subject}_{grain}_rt`
- DWS 表：`dws_{subject}_{metric}_rt`
- ADS 表：`ads_{scenario}_{service}`
- Flink 作业：`job_{domain}_{purpose}_{env}`

### D.2 开发规范

- 一张源表对应一份接入说明，不允许“口口相传”
- 所有字段映射必须沉淀在文档与版本库中
- 状态类字段必须维护标准码表
- 维表 Join 必须明确主键、时态、迟到处理策略
- 事实表必须明确去重规则、幂等策略、时间语义

### D.3 发布规范

- SQL、配置、脚本统一纳入 Git 管理
- 生产发布必须经过评审、测试、回滚演练
- 重大字段变更必须更新数据字典与影响分析

### D.4 运维规范

- 所有核心作业必须接入监控
- 所有告警必须有责任人和处理流程
- 每月进行容量与成本复盘

## 15. 结论与建议

本方案建议以“核心业务场景牵引 + 标准化湖仓底座 + 渐进式 AI 增强”的方式推进服装行业实时湖仓建设：

- 一期聚焦“实时库存调拨”和“爆款追单”两大场景，优先打通价值链路
- 技术底座以 `Flink CDC / Kafka / Flink / Paimon / Doris / Superset` 为主生产方案
- `Apache Fluss` 以旁路 PoC 方式验证，为后续演进预留选项
- `Doris 4.x + AI` 作为智能分析与运营助手的实验平台，逐步推动数据平台从“报表平台”走向“决策平台”

如果按企业级项目标准实施，建议把本方案进一步拆分为：

- 《业务调研与蓝图设计说明书》
- 《数据模型与指标口径说明书》
- 《实时链路实施方案与部署手册》
- 《测试验收方案》
- 《运维巡检与迭代优化方案》

这样既适合管理层评审，也便于研发、测试、运维和业务团队分工执行。
