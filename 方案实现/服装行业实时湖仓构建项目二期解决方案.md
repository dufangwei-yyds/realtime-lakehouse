# 服装行业实时湖仓构建项目二期解决方案

> 适用对象：服装品牌企业 / 零售企业 / 供应链数字化团队
> 方案目标：在一期基础上，深化实时湖仓能力，引入AI增强，扩展业务场景，构建企业级数据智能平台
> 核心技术栈：`Flink CDC / Kafka / Flink SQL & DataStream API (Flink 1.21.x) / Paimon / Doris 4.x / Superset / Apache Fluss / AI 能力`
> 扩展方向：`AI 驱动的数据建模/治理/分析 / 实时特征平台 / 全渠道会员营销 / 供应链智能决策`

---

## 1. 项目总览

### 1.1 一期成果回顾

一期项目已成功构建了服装行业实时湖仓的基础框架，完成了以下核心成果：

- 搭建了基于 `Kafka + Flink + Paimon + Doris` 的实时湖仓架构
- 实现了全渠道实时库存可视与智能调拨、爆款趋势识别与快速追单两大核心场景
- 建立了标准化的数据分层、数据治理、指标体系与开发规范
- 验证了 Doris 4.x AI 相关能力和 Apache Fluss 预研链路

### 1.2 二期建设背景

随着业务的快速发展和数字化转型的深入，服装企业面临新的挑战与机遇：

- 全渠道融合加速，需要更实时、更全面的数据视图
- 消费者行为更加复杂，需要更智能的分析和预测能力
- 供应链响应速度要求提高，需要更精准的决策支持
- 数据孤岛问题仍然存在，需要更完善的治理体系
- AI 技术在数据领域的应用潜力巨大，需要系统性落地

### 1.3 二期建设目标

本项目拟在一期基础上，构建更加完善、智能的服装行业实时湖仓平台，实现：

- 扩展订单履约、门店经营、会员增长等核心业务场景
- 引入 AI 驱动的数据建模、治理与分析能力
- 推进 Apache Fluss 在特定场景的生产化应用
- 构建实时特征平台，支撑更复杂的业务决策
- 完善数据治理体系，提升数据质量和可运维性
- 形成企业级数据智能平台，赋能业务全链路

### 1.4 项目范围

#### 范围内

- 一期核心场景的深化与优化
- 新增 3 个核心业务场景的建设
- AI 能力在数据湖仓中的系统性落地
- Apache Fluss 在特定场景的生产化应用
- 实时特征平台建设
- 数据治理体系完善
- 数据服务接口扩展
- 监控运维体系升级

#### 范围外

- 企业级大模型平台的独立建设
- 所有历史数据的全面重构
- 外围业务系统的主流程改造
- 全量替换企业现有 BI 体系

## 4. 方案架构设计

### 4.1 总体架构

```text
业务系统/外部数据源
  ├─ ERP / OMS / WMS / POS / CRM
  ├─ 第三方物流 API
  ├─ 小红书爬虫数据
  ├─ App / 小程序 / 官网埋点
  └─ 会员系统

接入层
  ├─ Flink CDC
  ├─ Kafka
  ├─ API采集服务 / 爬虫采集服务
  └─ 埋点SDK / 日志采集

计算与湖仓层
  ├─ Flink SQL / DataStream API
  ├─ Paimon ODS / DWD / DWS
  ├─ 维度拉链 / 宽表 / 实时特征
  ├─ Apache Fluss（生产化应用）
  └─ AI 特征平台

服务与消费层
  ├─ Doris 4.x 明细查询 / 聚合分析 / AI能力
  ├─ Superset 可视化看板
  ├─ 指标服务 / 数据 API
  ├─ 告警推送 / 运营应用
  └─ AI 分析助手
```

### 4.2 分层架构

| 层级 | 目标 | 存储建议 | 说明 |
| ---- | ------------------------------ | -------------- | ------------------------------ |
| ODS | 原始入湖、保留业务语义 | Paimon | 接近源系统结构，做轻度标准化 |
| DIM | 主数据统一、维度管理 | Paimon | SCD2 拉链、Latest 镜像、维度退化宽表 |
| DWD | 明细清洗、事实沉淀、维度拉齐 | Paimon | 统一主键、时间、状态、组织口径 |
| DWS | 主题汇总、实时宽表、指标域模型 | Paimon + Doris | 支撑秒级/分钟级分析 |
| ADS | 面向场景的数据服务层 | Doris | 面向报表、API、专题应用 |
| **特征层** | 实时特征计算与存储 | **Fluss + Paimon** | 支撑AI模型和复杂决策 |

### 4.3 技术角色分工

| 组件 | 角色 |
| ------------- | ---------------------------------------------------- |
| Flink CDC | 从 ERP/OMS/WMS 等关系型系统采集增量变更 |
| Kafka | 统一解耦消息总线，承载订单、埋点、社媒、API 数据 |
| Flink SQL/API | 实时清洗、维表关联、状态计算、窗口聚合、特征生成 |
| Paimon | 湖仓主存储，承载主明细、维度、增量快照、流批一体加工 |
| **Apache Fluss** | **实时特征存储、热点明细加速、低延迟查询** |
| Doris 4.x | 高并发分析查询、实时主题服务、AI 能力落地载体 |
| Superset | 经营看板、专题分析、业务自助查询 |
| **AI 模块** | **智能建模、智能治理、智能分析、智能运维** |

### 4.4 关键设计原则

- 主链路稳定优先：`Kafka + Flink + Paimon + Doris` 作为正式生产链路
- 创新能力有序引入：`Apache Fluss` 和 `AI 能力` 分场景试点后推广
- 数据湖与分析引擎解耦：Paimon 承接事实沉淀，Doris 承接交互查询，Fluss 承接实时特征
- 先统一主数据，再统一指标，最后智能化
- 先建设高价值场景，再平台化推广
- 所有模型、口径、规则都要文档化并可验收

## 6. 数据接入与数据治理

### 6.1 ODS 层接入范围扩展

在一期基础上，二期 ODS 层新增以下数据：

| 数据域 | 数据对象 | 来源 | 接入方式 | 时效目标 |
| ---------- | ------------------------------------------------ | -------------- | ----------------- | ------------ |
| 会员数据 | 会员基本信息、会员等级、积分、消费记录 | CRM | Flink CDC | 分钟级 |
| 门店运营 | 门店客流、导购业绩、时段销售、试穿记录 | POS/门店系统 | API 拉取 + Kafka | 15 分钟 |
| 营销数据 | 活动配置、优惠券、促销效果、渠道投放 | 营销系统 | API 拉取 + Kafka | 小时级 |
| 供应链数据 | 采购订单、生产计划、供应商交付、质检记录 | SCM | Flink CDC | 小时级 |

### 6.2 接入策略

#### 关系型业务库

- 使用 `Flink CDC` 抽取增量变更
- 按业务域拆分 Source 作业，避免单点超大任务
- 表结构变化纳入变更评审流程
- 支持 Schema 演进管理

#### 行为日志与事件流

- 埋点事件统一进入 Kafka
- 统一事件模型：事件时间、用户、设备、会话、页面、动作、业务对象
- 支持乱序、水位线和幂等去重
- 增加事件类型标准化处理

#### API 与爬虫数据

- API 拉取结果先写 Kafka 原始 Topic，再入 ODS
- 小红书爬虫数据必须增加合法合规审查、频率控制、脱敏与来源标识
- 新增门店运营数据的标准化处理

### 6.3 数据标准化治理

在一期基础上，二期重点治理对象：

- 时间标准：业务时间、事件时间、入湖时间、处理时间统一定义
- 编码标准：SKU、SPU、店仓、组织、渠道、订单号、物流单号统一编码体系
- 主数据标准：商品主数据、门店主数据、组织主数据、供应商主数据、会员主数据统一治理
- 状态标准：订单状态、库存状态、物流状态、会员状态统一枚举映射
- 命名标准：库、表、字段、Topic、任务、看板统一命名规范
- **特征标准**：特征名称、计算逻辑、更新频率、生命周期统一规范

### 6.4 数据质量治理

| 治理维度 | 规则示例 |
| -------- | -------------------------------------- |
| 完整性 | 主键不能为空、关键业务字段不能为空 |
| 唯一性 | 订单号 + 明细号组合唯一 |
| 一致性 | ERP 商品主数据与 OMS 商品编码映射一致 |
| 时效性 | ODS 入湖延迟不超过 5 分钟，关键链路不超过 1 分钟 |
| 合法性 | 库存数量不可出现非法负值，时间不可逆序 |
| 稳定性 | 每日字段变更必须有评审记录 |
| **特征质量** | **特征值范围、更新频率、计算准确性** |

### 6.5 安全与合规

- 用户标识、手机号、地址等敏感字段脱敏
- 第三方数据采集遵循法务与平台合规要求
- 开发、测试、生产环境权限隔离
- 行级/列级权限在 Doris 与 BI 层同步控制
- **AI 模型数据使用合规性审查**

## 7. 湖仓核心层建设

### 7.1 数据分层实施路径

| 顺序 | 层级 | 建设重点 |
| ---- | ---- | ------------------------------------------------ |
| 1 | ODS | 全量接入、轻度清洗、保留原始语义与湖仓元数据 |
| 2 | DIM | 主数据统一、SCD2 拉链、Latest 镜像、维度退化宽表 |
| 3 | DWD | 订单、库存、物流、行为、社媒、会员事实建模，单表即分析 |
| 4 | DWS | 公共粒度汇总、指标固化、主题宽表与服务层预聚合 |
| 5 | **特征层** | **实时特征计算、特征存储、特征服务** |
| 6 | ADS | 面向库存调拨、爆款识别、订单履约、门店运营、会员营销的专题层 |

### 7.2 核心建模规范与方法论

#### 7.2.1 ODS 事件流与 CDC 建模规范

- 保持原始：不做破坏性聚合，不丢原始 JSON，不在 ODS 固化复杂业务口径
- 注入元数据：每张 ODS 事实表必须补充 `_event_time`、`_ingest_time`、`_partition_dt`、`_kafka_offset`、`_kafka_partition` 等湖仓元字段
- 延后解析：对高漂移、高并发事件流采用“关键字段列化 + 原始 JSON 存根”模式
- 分区存储：埋点与行为流建议按 `dt + hh` 二级分区，避免大促期间单分区文件爆炸
- Schema 演进：优先采用 Paimon 模式演进能力，降低前端埋点变更对任务稳定性的影响
- 轻度净化：进入 ODS 前只做基础空值处理、时间标准化、非法字段兜底，保证后续关联不被脏数据放大

#### 7.2.2 DIM 维度建模规范

采用“四层维度建模法”：

| 维度层次 | 作用 | 适用场景 |
| --------------- | --------------------------- | ------------------------ |
| Append 表 | 保留原始维度变更轨迹 | CDC 增量沉淀、审计追溯 |
| SCD2 History 表 | 保存历史版本，支持时间回溯 | 价格、季节、组织归属变化 |
| Latest 镜像表 | 存储最新状态，支持高效 Join | DWD Lookup Join |
| 退化宽表 | 物理打平常用维度字段 | 降低 DWD/DWS Join 开销 |

维度层关键规则：

- 维度不只解决“是什么”，还要解决“在当时是什么”
- 高变维度必须有历史表，禁止只有 Latest 没有 History
- 常用关联维度要做物理退化宽表，避免下游重复多表 Join
- 维度主键、业务主键、版本起止时间必须统一命名与语义
- 所有维度都要区分高基数维与低基数维，分别规划存储与缓存策略

#### 7.2.3 DWD 明细建模规范

DWD 层是整个实时湖仓的“腰部”，必须坚持以下原则：

- 单表即分析：事实流和常用维度做宽表化，尽量让下游查询不再频繁关联维表
- 事件时间优先：所有统计与关联以 `event_time` 为准，而不是处理时间
- 标准口径统一：订单状态、金额精度、渠道、组织、商品编码统一映射
- 按业务过程建模：如下单、支付、退款、出库、签收等分开建模，不混成“大而全万能表”
- 幂等去重：即使 ODS 已去重，DWD 仍要通过主键表或去重规则再次兜底
- 事实关联事实必须显式限定时间区间，如点击后 30 分钟内下单才算转化

#### 7.2.4 DWS 汇总建模规范

DWS 层重点不是“继续堆表”，而是沉淀公共粒度和统一指标：

- 先定义公共粒度，再做聚合，如 `[小时 + 组织 + SKU]`、`[天 + 渠道 + 品类]`、`[天 + 门店 + 导购]`
- 所有核心指标在 DWS 固化口径，避免每张报表各算各的
- 区分加和性指标与非加和性指标，分别选择 SUM、Bitmap、HLL 等聚合策略
- DWS 覆盖 80% 通用查询，复杂探索性分析则由 Doris 外表 + 物化视图兜底

#### 7.2.5 **特征层建模规范**

新增特征层，支撑 AI 模型和复杂业务决策：

- 特征分类：用户特征、商品特征、门店特征、渠道特征、营销特征
- 特征计算：实时特征（分钟级更新）、准实时特征（小时级更新）、离线特征（天级更新）
- 特征存储：热点特征存储在 Fluss，历史特征存储在 Paimon
- 特征服务：提供统一的特征查询接口，支持在线推理和批量分析

#### 7.2.6 ADS 应用建模规范

ADS 层不是“再做一层汇总”，而是把 DWS 和特征层的能力组织成可直接服务业务的主题数据集、接口结果集和 BI 消费模型：

- 面向场景建模：ADS 必须直接对应业务消费对象，如库存驾驶舱、调拨建议、爆款榜单、订单履约看板、门店效能分析、会员营销等
- 面向消费端组织字段：字段设计要服务 Superset 看板、接口返回、告警推送，优先保留筛选字段、排序字段、展示字段、钻取字段
- 一张 ADS 只服务一类稳定场景：避免把看板字段、接口字段、明细排查字段全部混在一张“万能 ADS 表”中
- 指标不重复发明：ADS 原则上不重新定义核心业务口径，而是承接 DWS 固化指标，在必要时补充业务规则标签和结果解释字段
- 查询体验优先：ADS 模型要优先保证秒级查询、可分页、可筛选、可排序，必要时通过 Doris 聚合模型、物化视图、分区裁剪进一步加速
- 服务化输出：同一主题应同时考虑“BI 看板视图”和“接口服务结果集”两类消费方式，避免后期再二次加工
- 可解释性：所有推荐类、预警类、排行类 ADS 结果都要带上触发原因、规则分值、更新时间，避免业务方只看到结论看不到依据

### 7.3 分层建设方法与落地策略

#### 7.3.1 ODS 建设方法

- ERP/WMS/OMS/CRM 等业务库通过 `Flink CDC -> Paimon ODS`
- Kafka 行为流通过 `Kafka -> Flink SQL -> Paimon Append ODS`
- 高并发行为表采用 Append-Only + 动态分桶
- 有重复风险的消息流增加 Paimon 主键去重表
- 新增会员、门店运营、营销数据的标准化接入

#### 7.3.2 DIM 建设方法

- 商品、价格、季节、门店、组织、会员等维度优先建设
- 基于 CDC 构建 Append 表，再派生 SCD2 History 与 Latest 镜像
- 高频 Join 维度做退化宽表，例如 `dim_sku_wide`、`dim_member_wide`
- Lookup Join 场景启用缓存，并控制 TTL 和缓存行数

#### 7.3.3 DWD 建设方法

- 订单、库存、行为、物流、会员等事实在 DWD 完成清洗、统一、打宽
- 事实流与事实流通过 Interval Join / Window Join 处理业务链路
- 事实流与维度通过 Lookup Join 处理主数据补全
- DWD 明细优先落 Paimon，保证“可重算、可回溯、低成本”

#### 7.3.4 DWS 建设方法

- 固定高频指标通过 Flink 预聚合或 Doris 聚合表落地
- 灵活分析主题优先走 Doris 读取 Paimon DWD 外表 + Async MV 自动加速
- DWS 既可以落 Paimon，也可以把热点主题落 Doris 形成服务层
- 新增门店运营、会员营销等主题的汇总模型

#### 7.3.5 **特征层建设方法**

- 实时特征通过 Flink 计算写入 Fluss，支撑低延迟查询
- 准实时和离线特征通过 Flink 或 Spark 计算写入 Paimon
- 构建特征服务接口，支持在线推理和批量分析
- 建立特征血缘和版本管理机制

#### 7.3.6 ADS 建设方法

- ADS 优先落 Doris，直接面向 Superset BI 看板、数据服务接口和经营专题应用
- ADS 输入以 DWS 和特征层为主、DWD 为辅：稳定指标来自 DWS，必要钻取字段或解释字段可补充引用 DWD 明细
- 围绕二期三大核心场景优先建设专题 ADS：
  - 订单履约主题：履约进度、异常预警、时效分析
  - 门店运营主题：销售效能、人效分析、库存健康度
  - 会员营销主题：会员画像、消费分析、个性化推荐
- 对高频访问 ADS 结果集，优先采用 Doris 内表 + 异步物化视图，降低看板高并发对底层明细的冲击
- 对接口服务场景，ADS 要同步设计分页键、排序键、过滤条件和更新时间字段，保证前端/服务端可直接消费

### 7.4 建模过程中的最佳实践与规避原则

| 最佳实践 | 说明 | 落地方式 |
| -------- | -------------------------------------- | ------------------------------------------ |
| ODS 采用“关键字段列化 + raw_json 存根” | 避免埋点一改字段就挂任务 | 统一 ODS 表结构设计 |
| 行为流使用 `dt + hh` 二级分区 | 避免大促期间单分区文件数爆炸 | 统一分区策略 |
| 注入 Kafka offset 等元数据 | 便于排查重复与丢数 | ODS 强制补齐湖仓元数据 |
| DWD 先做空值过滤和默认值兜底 | 避免 Join 倾斜、空关联、状态膨胀 | 统一清洗规则 |
| 统一使用事件时间和水位线 | 避免迟到数据统计错误 | 统一时间处理策略 |
| Lookup Join 启用缓存 | 避免维表查询打爆存储 | 统一缓存配置 |
| DWD 做幂等兜底 | 避免任务重启后明细膨胀 | 主键表 / deduplicate 策略 |
| 明细逻辑放 DWD，指标固化放 DWS | 避免口径混乱、难维护 | 统一分层职责 |
| 用 Doris 物化视图做预计算 | 避免直接拿湖上大明细做复杂 distinct | 统一查询加速策略 |
| 通过 DWS/ADS 主题模型收口 | 避免查询慢、字段乱、口径飘 | 统一服务层设计 |
| 拆分看板型 ADS、接口型 ADS、排查型明细入口 | 避免结构失控、变更牵一发动全身 | 统一 ADS 设计原则 |
| 补充规则标签、分值来源、更新时间、触发原因 | 避免业务不信任、难运营 | 统一结果可解释性设计 |
| 为接口单独设计结果集和分页排序键 | 避免性能不稳定、难治理 | 统一接口设计规范 |
| 让 Superset 优先消费 Doris ADS 主题表或 MV | 避免高并发下拖垮查询引擎 | 统一 BI 消费策略 |

### 7.5 维度建模设计

核心维度建议：

- 商品维：SPU、SKU、款色码、品牌、系列、季节、品类、上市波段
- 组织维：集团、大区、直营网店、加盟店、仓库、前置仓
- 渠道维：电商平台、门店、直播、分销
- 时间维：日、周、月、季度、促销周期
- **用户维**：会员等级、渠道来源、消费层级、标签、生命周期
- 供应链维：供应商、工厂、产线、物流商

关键要求：

- 主数据维表优先落 Paimon，支持 SCD2 历史追踪
- 统一 surrogate key / business key 使用规则
- 所有维度字段必须有中文释义与枚举说明
- 高频维度必须同时具备 Latest 镜像和退化宽表能力
- 维度宽表要服务于 DWD 单表分析，而不是仅为了“维度完整”而堆字段

### 7.6 事实建模设计

核心事实建议：

- 订单事实：下单、支付、发货、签收、退货
- 库存流水事实：入库、出库、调拨、锁定、释放、盘点、在途
- 物流轨迹事实：节点事件、异常事件、时效事件
- 行为事实：曝光、点击、浏览、加购、收藏、支付转化
- 社媒事实：笔记发布、点赞、收藏、评论、关键词热度
- **会员事实**：注册、登录、等级变更、积分变动、权益使用
- **门店运营事实**：客流、销售、试穿、导购业绩、时段表现

事实层落地要求：

- 每张事实表必须说明业务粒度、主键、事件时间、状态流转逻辑
- 事实表必须明确是否 append-only、upsert 或 deduplicate
- 订单与行为类事实必须区分业务时间和入湖时间
- 明细宽表字段控制在“够分析、不失控”的范围内，避免无限膨胀

### 7.8 Paimon 建设要点

- ODS 使用 Append/Primary Key 表结合，按业务对象选择写入模式
- 维表采用主键表 + SCD2 历史策略
- 大事实表按业务日期、渠道、组织等维度规划分区
- 制定 Compact、Snapshot Retention、Bucket 策略
- 明细沉淀优先在 Paimon，面向交互查询再同步至 Doris
- 行为流 Append 表优先使用动态分桶，高并发主键表显式设置 bucket
- Paimon 承担“数据基石”角色，确保湖上数据持久、低成本、可重算

### 7.9 **Fluss 建设要点**

- 在热点明细和实时特征场景下使用 Fluss
- 与 Flink 集成，实现低延迟读写
- 优先应用于：实时特征存储、热点商品数据、高频查询场景
- 制定数据生命周期管理策略
- 与 Paimon 形成互补，构建多层次存储架构

### 7.10 Doris 4.x 建设要点

- ADS 层主题模型落 Doris，支撑秒级查询
- 对复杂明细与汇总混合查询设计宽表与星型模型并存
- 充分利用 Doris 4.x AI 相关能力，服务自然语言分析、智能问答、SQL 助手等场景
- 构建基于 Doris 的实时服务接口，支撑高并发查询

### 7.11 Doris 物化视图应用设计

Doris 物化视图作为湖仓查询加速的标准能力：

#### 物化视图定位

- 对接 Paimon DWD/DWS 外表，承接高频聚合与去重指标加速
- 通过透明改写，让业务查询原表时自动命中物化视图
- 减少重复 ETL，避免为每个报表单独造一张汇总表

#### 推荐应用场景

| 场景 | 物化视图类型 | 说明 |
| --------------- | ------------------- | --------------------------------------- |
| 季节销售趋势 | 异步物化视图 | 按天 + 季节 + 品类预聚合销售额 |
| 库存调拨驾驶舱 | 异步物化视图 | 按组织 + SKU + 小时聚合 ATP、缺货预警数 |
| 爆款识别 | 异步物化视图 | 按小时预聚合销量、浏览、加购、社媒热度 |
| 门店运营分析 | 异步物化视图 | 按天 + 门店 + 导购聚合销售、客流、人效 |
| UV/下单用户去重 | Bitmap/HLL 物化视图 | 加速去重类指标计算 |
| 大屏战报 | 预聚合表 + MV | 对固定看板做多层加速 |

#### 建设策略

- 主明细保留在 Paimon，Doris 通过 Catalog 挂载外表
- 高频固定查询走 Doris 内部聚合表或异步物化视图
- 中高频灵活查询优先采用 Async MV，降低实时 ETL 复杂度
- 去重指标优先用 Bitmap/HLL 预计算，避免反复扫描明细
- 通过 `REFRESH AUTO` 或定时刷新保障准实时性

## 8. AI 能力落地

### 8.1 总体思路

AI 能力在二期的落地不是孤立的功能，而是重构实时湖仓的三个核心环节：

- 智能建模
- 智能治理
- 智能分析与运营闭环

### 8.2 AI 落地方向

#### 方向一：智能建模助手

- 自动解析源表结构、生成 ODS/DWD 建模草案
- 生成 Flink SQL 模板、字段映射建议、维度归类建议
- 识别同义字段、重复表、潜在主外键关系
- 基于历史数据模式，推荐最优分区和存储策略

#### 方向二：智能数据治理

- 自动识别异常波动、缺失字段、漂移指标
- 基于规则 + 模型做数据质量根因分析
- 为字段补充释义、标签、血缘说明
- 预测数据质量问题并提供修复建议

#### 方向三：智能经营分析

- 面向 Doris 4.x 验证自然语言问数能力
- 构建“爆款原因解释”“库存异常解释”“履约异常解释”能力
- 支持运营人员通过问答方式获取趋势洞察和建议
- 自动生成分析报告和洞察摘要

#### 方向四：智能调度与运维

- 对 Flink 作业异常自动归因
- 对 Kafka 积压、状态膨胀、写入失败给出优化建议
- 自动生成巡检日报与发布风险提示
- 预测系统负载并提供扩容建议

#### 方向五：智能推荐与预测

- 基于历史销售数据和实时行为数据，预测商品销量
- 为库存调拨提供智能建议
- 为会员提供个性化商品推荐
- 预测门店销售趋势和客流高峰

### 8.3 AI 建设原则

- 先辅助，再自动化
- 先规则增强，再模型增强
- 高风险动作必须有人审
- 所有 AI 输出都要可追溯、可解释、可回滚
- 数据安全和隐私保护优先

### 8.4 AI 技术栈

- 特征工程：基于 Flink 和 Paimon 的实时特征计算
- 模型训练：基于 Python 生态的机器学习框架
- 模型部署：容器化部署，支持在线推理
- 模型管理：版本控制、A/B 测试、模型监控

## 12. 风险与控制措施

| 风险 | 表现 | 应对措施 |
| ---------------- | ------------------------ | ------------------------------------ |
| 主数据不统一 | 商品/店仓/会员编码打不通 | 优先建设 MDM 映射与标准口径 |
| 源系统字段质量差 | 脏数据、缺字段、时间不准 | 接入前做字段评估，建立补录与兜底规则 |
| 业务口径反复变化 | 指标持续返工 | 建立指标评审机制，版本化管理 |
| 流式链路不稳定 | 积压、延迟、状态膨胀 | 分域拆作业、加强监控、压测先行 |
| 外部数据合规风险 | 爬虫、外部接口使用不规范 | 引入法务评审、访问频控、脱敏 |
| **AI 期望过高** | **过早追求全自动** | **明确 AI 为增强能力，分阶段落地** |
| **Fluss 集成复杂度** | **技术栈复杂度增加** | **先试点后推广，建立完善的监控和回滚机制** |
| **特征平台建设挑战** | **特征管理复杂度高** | **建立特征标准和生命周期管理机制** |
