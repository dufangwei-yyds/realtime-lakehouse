# 技术选型报告

> 项目名称：服装行业实时湖仓构建项目一期  
> 文档定位：一期实时湖仓技术选型与技术路线决策报告  
> 适用范围：项目评审、结项归档、后续二期技术演进参考

---

## 一、选型目标

本次技术选型围绕服装行业实时湖仓一期建设展开，目标是：

1. 选择一条可落地、可联调、可运维的实时湖仓主链路
2. 支撑两大核心场景：
  - 全渠道实时库存可视与智能调拨
  - 爆款趋势识别与快速追单
3. 同时满足明细沉淀、实时计算、交互查询、看板消费和接口服务
4. 为后续 Fluss 预研和 Doris AI 功能实验保留扩展空间

---

## 二、选型原则

一期技术选型遵循以下原则：

1. 稳定优先

优先选择生态成熟、与当前团队能力匹配、可快速落地的组件。

1. 分层解耦

接入、计算、存储、查询、消费尽量职责清晰，避免单一组件承担过多角色。

1. 实时与可维护并重

不仅要跑得快，也要便于排障、监控、回滚和持续维护。

1. 明细与服务并重

既要能沉淀 Paimon 明细，也要能通过 Doris 对外提供交互式服务。

1. 预留演进空间

一期以主生产链路稳定为目标，二期再逐步扩展 Fluss 和 Doris AI 等创新能力。

---

## 三、一期技术路线结论

一期最终采用的核心技术栈为：

- `Flink CDC`
- `Kafka`
- `Flink SQL / Flink DataStream API (Flink 1.20.x)`
- `Paimon`
- `Doris 4.x`
- `Superset`

推荐架构定位如下：


| 组件                   | 角色定位                                   |
| -------------------- | -------------------------------------- |
| Flink CDC            | 关系型业务库增量接入                             |
| Kafka                | 统一消息总线和事件缓冲层                           |
| Flink SQL            | 主体实时建模、窗口聚合、分层计算                       |
| Flink DataStream API | 处理 SCD2 等需要更强状态控制的专项链路                 |
| Paimon               | 湖仓主存储，承载 ODS / DIM / DWD / DWS 明细与主题沉淀 |
| Doris 4.x            | ADS 查询服务、Dashboard、接口结果、物化视图加速         |
| Superset             | BI 看板消费层                               |

## 四、核心组件选型说明

## 4.1 Flink CDC

### 选型原因

- 与 MySQL 等关系型数据库集成成熟
- 适合主数据、订单、库存等关系型业务库增量接入
- 能与 Flink 主链路天然协同

### 在一期中的主要作用

- 接入 ERP 主数据
- 接入订单主表和订单明细
- 接入库存流水

### 选型结论

适合作为一期关系型业务库增量接入标准方案。

## 4.2 Kafka

### 选型原因

- 生态成熟，适合作为事件流标准总线
- 能承接行为埋点、物流 API、社媒爬虫等异步事件
- 便于与 Flink 解耦，利于削峰和重放

### 在一期中的主要作用

- 行为埋点数据接入
- 物流轨迹事件接入
- 社媒舆情事件接入
- API / 爬虫采集结果缓冲

### 选型结论

适合作为一期统一事件总线。

## 4.3 Flink SQL / DataStream API

### 选型原因

- Flink SQL 适合主体分层建模，开发效率高
- DataStream API 适合复杂状态和 SCD2 专项处理
- 能同时支撑 CDC、流式 Join、窗口聚合、水位处理

### 在一期中的主要作用

- ODS 标准化接入
- DIM 宽表建设
- DWD 明细打宽
- DWS 主题聚合
- ODS 维度 SCD2 专项处理

### 选型结论

采用“Flink SQL 为主、DataStream API 为辅”的组合路线。

## 4.4 Paimon

### 选型原因

- 适合承载实时湖仓主存储
- 支持主键表、Append 表、快照、流批一体
- 与 Flink 集成自然，适合 ODS / DWD / DWS 分层沉淀

### 在一期中的主要作用

- ODS 维度 append / history / latest
- ODS 事件流事实与 analyze 解析表
- DIM 宽表
- DWD 明细表
- DWS 汇总结果沉淀

### 选型结论

适合作为一期实时湖仓的主存储底座。

## 4.5 Doris 4.x

### 选型原因

- 查询性能好，适合 Dashboard 和接口高频查询
- 支持物化视图，适合高频榜单和驾驶舱加速
- 具备 AI Function / Text Search / Vector Search 等可探索空间

### 在一期中的主要作用

- 承载 ADS 结果集
- 承载 Dashboard 和 API 查询服务
- 验证异步物化视图加速能力

### 选型结论

适合作为一期结果服务和交互查询层。

## 4.6 Superset

### 选型原因

- 开源、轻量、与 Doris 集成方便
- 足以承载一期库存和爆款主题看板
- 有利于快速完成 OOTB 验证

### 在一期中的主要作用

- 库存驾驶舱
- 爆款趋势 Dashboard

### 选型结论

适合作为一期 BI 看板消费层。

## 五、为什么采用“Paimon + Doris”双引擎协同

一期没有选择让单一组件同时承担“明细沉淀 + 高并发查询 + 结果服务”的全部职责，而是采用：

- `Paimon` 负责湖仓明细与分层沉淀
- `Doris` 负责 ADS、交互式查询和消费服务

这样设计的原因：

1. Paimon 更适合承接分层事实沉淀和可重算基础
2. Doris 更适合承接 BI / API 高并发查询
3. 双引擎职责清晰，更利于运维和性能优化
4. 与一期的 ODS、DWD、DWS、ADS 分层思路天然匹配

## 六、替代方案对比说明

## 6.1 与纯 Kafka + Flink + Doris 的对比


| 方案                             | 优点           | 局限              |
| ------------------------------ | ------------ | --------------- |
| 纯 Kafka + Flink + Doris        | 链路简单         | 缺少湖仓层明细沉淀与可重算基础 |
| Kafka + Flink + Paimon + Doris | 分层清晰、明细沉淀能力强 | 架构复杂度略高         |


选型结论：

- 一期更需要“明细沉淀 + 交互查询”并重，因此采用 `Paimon + Doris` 更合适。

## 6.2 与纯离线湖仓方案的对比


| 方案     | 优点          | 局限               |
| ------ | ----------- | ---------------- |
| 纯离线方案  | 稳定、团队熟悉     | 难以满足分钟级实时分析和运营调度 |
| 实时湖仓方案 | 时效高、能支撑实时场景 | 构建和运维复杂度更高       |


选型结论：

- 一期核心场景明确要求实时能力，因此不采用纯离线路线。

## 6.3 与单纯使用 SQL 引擎的对比


| 方案                      | 优点      | 局限                |
| ----------------------- | ------- | ----------------- |
| 全部依赖 SQL 引擎             | 研发门槛低   | SCD2、复杂状态、事件流处理受限 |
| SQL + DataStream API 混合 | 灵活，覆盖更全 | 团队需要同时掌握两种范式      |


选型结论：

- 一期需要处理 ODS 维度 SCD2 等复杂场景，因此保留 DataStream API 是必要的。

## 七、一期技术选型与业务场景的匹配关系


| 场景             | 关键数据        | 技术支撑                                   |
| -------------- | ----------- | -------------------------------------- |
| 全渠道实时库存可视与智能调拨 | 订单、库存、维度主数据 | Flink CDC + Flink SQL + Paimon + Doris |
| 爆款趋势识别与快速追单    | 行为、社媒、订单、库存 | Kafka + Flink SQL + Paimon + Doris     |


说明：

- 库存场景更依赖关系型业务库事实和维度
- 爆款场景更依赖事件流、行为和社媒信号融合
- 当前技术栈同时覆盖两类需求

## 八、一期选型实践结果总结

结合当前项目实际联调和验证结果，可以得出以下结论：

1. `Flink CDC + Kafka + Flink + Paimon + Doris + Superset` 这条主链路已被一期实践验证可行
2. ODS、DIM、DWD、DWS、ADS 全链路已打通并验证通过
3. Superset 和数据服务接口均已完成消费侧验证
4. 当前技术栈已能够支撑一期两大核心场景

## 九、当前约束与风险说明

当前技术选型在一期测试环境下也存在一些现实约束：

1. 测试环境规模有限

当前 Doris 单 `BE`、Kafka 单 Broker，更接近验证环境而非正式生产环境。

1. Superset 图表能力有限

当前版本不支持 `Mixed Time-series` 等个别图表能力，需在图表设计时做适配。

1. 实时链路运维复杂度高

相比离线数仓，实时湖仓对监控、巡检、回滚和补数要求更高。

1. Fluss 和 Doris AI 仍处于后续预研阶段

当前尚不纳入一期正式生产主链路。

## 十、Fluss 与 Doris AI 的后续定位

## 10.1 Apache Fluss

当前定位：

- 作为二阶段轻量预研方向
- 不替代一期主生产链路
- 重点评估其在实时分析与 AI 场景中的潜力

建议验证方向：

- 实时特征层
- 热点明细层
- 与 Flink 集成方式
- 吞吐与低延迟表现

## 10.2 Doris AI

当前定位：

- 作为 Doris 4.x 增强能力方向
- 先做轻量实验，不直接承诺生产级落地

建议验证方向：

- AI Function
- Text Search
- Vector Search
- 与商品、社媒、热点分析的结合可能性

---

## 十一、技术选型结论

综合业务目标、数据成熟度、团队可落地能力、一期联调结果和后续演进空间，一期技术选型结论如下：

1. 一期主生产链路继续采用：
  `Flink CDC / Kafka / Flink SQL & DataStream API / Paimon / Doris 4.x / Superset`
2. 一期不建议引入过多替代链路，避免分散主线精力
3. 二期可在当前主链路稳定基础上，逐步推进：
  - `Apache Fluss` 预研
  - `Doris AI` 相关功能实验
