## 一、调研目标

本次业务调研围绕服装行业实时湖仓一期建设展开，目标是：

1. 梳理服装行业核心业务过程与数据对象
2. 识别影响实时经营决策的关键场景
3. 明确一期优先建设范围与边界
4. 为 ODS、DIM、DWD、DWS、ADS 分层建设提供业务输入
5. 为指标体系、数据服务接口和 BI 看板设计提供依据

## 三、行业业务特征总结

服装行业相较一般零售业务，具有以下鲜明特征：

1. 商品粒度复杂

商品通常按 `SPU -> SKU -> 款色码` 管理，且同时受品牌、年份、季节、波段、系列、品类、颜色、尺码等多维属性影响。

1. 库存流转链路长

库存不仅存在于总部仓、区域仓、门店、电商前置仓，还会经历调拨、锁定、在途、退货、盘点等多种状态变化。

1. 快反经营要求高

爆款识别、快速追单、缺货预警、临时调拨等业务对时效要求高，传统 T+1 离线分析难以满足需求。

1. 渠道与组织复杂

线下门店、电商、直营、加盟、区域组织、仓库组织并存，导致指标口径和组织归属容易不一致。

1. 外部信号价值高

除 ERP 交易和库存数据外，社媒热度、用户行为、物流节点等外部信号对经营判断影响显著。

## 四、核心业务对象梳理

### 4.1 商品对象

商品域是一期的核心主数据域，主要对象包括：

- 商品主数据
- 商品维度属性
- 商品别名信息
- 属性集实例
- 属性明细

核心管理维度包括：

- SKU
- SPU / 款号
- 品牌
- 商品年份
- 季节
- 波段
- 系列
- 品类
- 颜色
- 尺码

### 4.2 订单对象

订单域是交易与经营分析的核心事实域，主要对象包括：

- 订单主表
- 订单明细
- 支付时间
- 订单状态
- 渠道归属
- 门店归属

### 4.3 库存对象

库存域是供应链管理的核心事实域，主要对象包括：

- 库存流水
- 库存状态
- 业务单号
- 店仓归属
- 调拨过程
- 在途过程

### 4.4 物流对象

物流域主要关注订单履约和节点透明度，主要对象包括：

- 物流单号
- 物流商
- 物流节点
- 节点状态
- 节点时间
- 快递员 / 站点信息

### 4.5 用户与社媒对象

行为和社媒域主要服务于爆款识别和趋势分析，主要对象包括：

- 用户浏览、点击、加购、收藏
- 订单转化行为
- 社媒笔记
- 作者与达人属性
- 点赞、收藏、评论等互动指标

## 五、核心业务过程梳理

## 5.1 商品主数据维护过程

业务过程说明：

1. ERP 中维护商品主数据
2. 维护商品属性、维度、别名、属性集实例
3. 商品状态、季节、品类、价格等可能发生变更
4. 下游需要既能拿到“当前最新状态”，也能追溯“历史时点状态”

数据建设启示：

- 维度域必须支持 `append + history + latest`
- 必须支持 SCD2 历史追溯
- 必须为 DWD 提供统一的退化宽表入口

## 5.2 订单交易过程

业务过程说明：

1. 用户下单
2. 订单进入不同状态流转
3. 订单与渠道、门店、商品、支付时间建立关联
4. 下游既关注订单明细，也关注分钟级、小时级聚合结果

数据建设启示：

- ODS 需保留订单原始事实
- DWD 需统一订单明细粒度与维度补齐
- DWS 需沉淀订单需求量、销售额等主题指标

## 5.3 库存变动与调拨过程

业务过程说明：

1. 商品因入库、出库、调拨、盘点等业务发生库存变化
2. 库存同时存在可用、锁定、在途等状态
3. 调拨决策依赖实时 ATP、库存健康度、需求强弱

数据建设启示：

- 库存流水必须进入 ODS
- DWD 需沉淀标准化库存流水明细
- DWS 需沉淀 ATP、调拨建议等公共主题
- ADS 需面向看板和接口输出解释性结果

## 5.4 物流履约过程

业务过程说明：

1. 订单发货后进入物流过程
2. 物流轨迹按节点不断更新
3. 履约链路关注延迟、异常、签收等状态

数据建设启示：

- 第三方物流 API 数据需进入 ODS 原始表和解析表
- DWD 需沉淀物流节点级明细
- 后续可扩展物流异常实时预警主题

## 5.5 爆款形成与追单过程

业务过程说明：

1. 商品在销售、点击、加购、收藏、社媒互动等多方面出现明显上升趋势
2. 商品企划和供应链需要快速判断是否追单
3. 追单决策同时受热度、销量、库存支撑能力影响

数据建设启示：

- 行为数据、社媒数据与订单数据必须统一事件建模
- DWD 需沉淀爆款归一化事件明细
- DWS 需沉淀爆款特征与排行
- ADS 需输出追单建议与触发原因

## 六、五大业务场景调研结果

### 6.1 场景一：全渠道实时库存可视与智能调拨

业务目标：

- 统一查看店仓库存
- 判断可售库存与风险库存
- 识别缺货风险并给出调拨优先级

核心诉求：

- 店仓级 SKU 级实时 ATP
- 缺货风险识别
- 调拨优先级排序
- Dashboard 和接口双消费

### 6.2 场景二：爆款趋势识别与快速追单

业务目标：

- 实时识别热度上升商品
- 结合库存和销售趋势给出追单建议

核心诉求：

- 统一爆款综合分值
- 支撑小时级趋势观测
- 输出排行榜、等级、触发原因

### 6.3 场景三：订单履约与物流异常预警

业务目标：

- 追踪订单履约路径
- 识别物流延迟和异常节点

核心诉求：

- 节点级物流可视化
- 异常事件识别
- 客服和履约团队可快速定位问题

### 6.4 场景四：门店实时经营效能分析

业务目标：

- 实时分析门店经营表现和人效情况

核心诉求：

- 门店、区域、品类维度经营分析
- 更偏零售运营视角

### 6.5 场景五：消费者行为洞察与人货场联动

业务目标：

- 分析用户行为转化漏斗
- 支撑精准营销和选品分析

核心诉求：

- 统一事件模型
- 统一人货场行为口径

## 七、一期场景筛选结论

综合业务价值、数据成熟度、实现复杂度和跨部门关注度，一期优先选择以下两个核心场景：


| 场景             | 业务价值 | 数据成熟度 | 技术复杂度 | 一期结论   |
| -------------- | ---- | ----- | ----- | ------ |
| 全渠道实时库存可视与智能调拨 | 高    | 高     | 中     | 入选     |
| 爆款趋势识别与快速追单    | 高    | 中     | 中高    | 入选     |
| 订单履约与物流异常预警    | 中高   | 中     | 中     | 二期优先候选 |
| 门店实时经营效能分析     | 中    | 中     | 中     | 后续扩展   |
| 消费者行为洞察与人货场联动  | 中高   | 中     | 中高    | 后续扩展   |


筛选原因：

- 两大入选场景业务价值最直接
- 对实时湖仓价值体现最明显
- 已具备较好的数据基础与联调可行性
- 能较好覆盖交易、库存、行为、社媒等多源数据融合能力

## 八、业务痛点与建设对应关系


| 业务痛点            | 对应建设方向                          |
| --------------- | ------------------------------- |
| 库存视图分散，难以统一查看   | 统一 ODS 库存接入、DWD 库存明细、DWS ATP 汇总 |
| 调拨决策依赖人工经验      | 构建 DWS 调拨建议与 ADS 接口结果           |
| 爆款识别依赖人工经验和滞后数据 | 融合订单、行为、社媒构建爆款特征主题              |
| 指标口径不统一         | 建立业务指标体系和数据字典                   |
| 多系统数据割裂         | 建立 CDC + 事件流 + 湖仓统一分层模型         |
| 数据消费方式单一        | 同时建设 Dashboard 与 API 结果集        |

## 九、一期建设边界确认

一期范围内：

- 主数据域统一
- 订单、库存、物流、社媒、行为等关键源接入
- 两大核心场景闭环建设
- Superset 和 API 两类消费能力
- 数据治理、监控运维和验收体系建设

一期范围外：

- 所有主题一次性全面建设
- 所有历史数据大规模补齐与全面治理
- Fluss 正式生产切换
- Doris AI 生产级大规模落地

## 十、业务调研输出结论

本次业务调研确认了以下关键结论：

1. 商品、订单、库存、行为、社媒是一期实时湖仓建设的核心对象
2. 全渠道实时库存可视与智能调拨、爆款趋势识别与快速追单是一期最具价值的两个核心场景
3. 实时湖仓不仅要解决“数据汇集”，更要解决“统一口径、统一主数据、统一消费”
4. 业务消费形态需要同时覆盖 Dashboard 和 API
5. 后续二期可在当前基础上继续扩展物流异常、门店经营、消费者洞察等主题

## 十一、对后续建设的输入建议

### 11.1 对模型建设的输入

- DIM 层优先保障商品维度统一
- DWD 层优先保障订单与库存明细可统一分析
- DWS 层优先固化 ATP、调拨建议、爆款特征与排行
- ADS 层优先面向 Dashboard 与 API 输出结果

### 11.2 对指标建设的输入

- 一期优先补齐库存类、订单类、热度类、建议类指标
- 指标必须同步明确粒度、公式、来源层级和责任归属

### 11.3 对治理运维的输入

- 重点关注主链路时效性和结果一致性
- 重点关注 ODS 事件流与 ODS 维度 SCD2 的稳定性
- 重点关注 DWS / ADS 结果刷新与消费 SLA
