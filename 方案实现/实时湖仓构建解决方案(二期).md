# 服装行业实时湖仓构建项目二期解决方案

> 适用对象：服装品牌企业 / 零售企业 / 供应链数字化团队（二期延续）
> 方案目标：在一期已建成的主链路基础上，以"稳中求进、局部增强、场景驱动"为原则，扩展业务场景覆盖、推进技术创新试点，推动数据平台从"数据服务层"向"智能决策增强平台"演进
> 核心技术栈：`Flink CDC / Kafka / Flink SQL & DataStream API (Flink 1.20.x) / Paimon 1.0.x / Doris 4.x + AI / Apache Fluss 0.8.0 (试点) / Superset`
> 一期基座：一期主链路保持不变，二期在已有成果上做增量演进
> 扩展方向：`Apache Fluss` 局部试点验证、`Doris AI` 能力增强落地、多源数据接入补齐、新增三大核心场景建设

***

#### 1.1.3 二期建设的必要性

在一期成果基础上，二期的建设驱动来自以下四个方面：

**（1）业务场景延伸需求**

一期聚焦了"库存可视"和"爆款识别"两大场景，但以下高价值场景尚未覆盖：

- 订单履约与物流异常实时预警：从下单到签收的全链路追踪能力缺失
- 门店实时经营效能分析：按门店/导购/时段/品类的实时经营洞察不足
- 消费者行为深度洞察与人货场联动：埋点数据的深度挖掘和价值释放不充分

**（2）数据接入能力补齐**

一期接入范围主要集中在 MySQL CDC 和基础 Kafka 验证，以下数据源尚待完善：

- 埋点数据：缺少规范的埋点数据模型设计和完整接入链路
- Nginx 服务日志：API 网关日志未纳入湖仓
- 第三方 API 数据：物流轨迹、天气数据等接入不够完善
- 社媒爬虫数据：小红书等社媒数据需要更规范化的接入流程

**（3）技术创新价值释放**

一期已完成两项关键预研/实验：

- `Fluss` 三阶段预研证明其在"实时状态层"和"行为流富化层"具备试点价值
- `Doris AI` 五轮实验证明其 AI Functions / Text Search / Vector Search 能力具备生产结合基础

这些预研结论需要在二期中从"实验室"走向"小范围试点"，验证真实业务价值。

**（4）平台能力升级需求**

从"能用的数据平台"向"好用的决策增强平台"演进，需要：

- 更强的实时状态表达能力
- 更强的知识检索和语义检索能力
- 更强的文本分析与运营辅助能力
- 更完善的调度编排和元数据管理能力

### 1.2 二期建设目标

#### 1.2.1 总体目标

在一期稳定运行的主链路基础上，围绕 **"增量演进、局部增强、场景驱动"** 三条主线，完成以下建设：

1. **扩展业务场景覆盖**：新增 3 大核心业务场景，使实时湖仓支撑的场景从 2 个扩展到 5 个
2. **推进技术创新试点**：`Fluss` 小范围试点 + `Doris AI` 能力增强落地
3. **补齐数据接入体系**：建立更完整的多源异构数据接入能力
4. **深化治理与平台化**：推动治理精细化、调度自动化、元数据统一化
5. **推动智能化增强**：逐步引入 AI 能力，提升平台的决策辅助价值

#### 1.2.2 目标分解

| 维度    | 一期状态                        | 二期目标                                  |
| ----- | --------------------------- | ------------------------------------- |
| 业务场景  | 2 个核心场景（库存、爆款）              | 5 个核心场景（+物流预警、门店经营、消费者洞察）             |
| 数据接入  | MySQL CDC + 基础 Kafka        | 全量 CDC + 完整埋点 + 日志 + API + 社媒         |
| 存储引擎  | Paimon（主）+ Doris（ADS）       | Paimon（主）+ Doris（ADS + AI）+ Fluss（试点） |
| AI 能力 | 预研阶段（Doris AI 实验、Fluss PoC） | Doris AI 小范围落地 + Fluss 局部试点           |
| 治理体系  | 基础治理框架                      | 精细化规则 + 自动化巡检 + 元数据管理                 |
| 调度能力  | 手动/Flink 原生调度               | DolphinScheduler 编排（可选）               |
| 消费形态  | Superset 看板 + REST API      | 看板 + API + AI 增强 + 智能问答（探索）           |

### 1.3 项目范围

#### 范围内

- 新增 3 个核心业务场景的分层建模与端到端链路建设
- `Fluss` 在 1-2 个局部场景的小范围旁路试点
- `Doris AI` 在社媒舆情、商品知识、客服文本领域的能力增强落地
- 多源数据接入体系补齐（埋点、日志、API、爬虫规范化）
- 数据治理体系的深化与自动化
- 平台调度编排能力的评估与可选引入
- 测试、验收、上线、迭代优化

#### 范围外

- 不替换一期主生产链路（Paimon 主存储地位不变）
- 不进行大规模系统重构或全量数据迁移
- 不单独建设独立的大模型平台
- 不全面替换现有 BI 工具体系
- 不涉及外围系统的主流程改造

### 3.2 场景 C：订单履约与物流异常实时预警

#### 3.2.1 业务目标

- 跟踪订单从下单、审核、支付、出库、揽收、运输、派送到签收的全生命周期
- 实时识别履约异常节点（超时未发货、揽收超时、运输滞留、疑似丢件）
- 提升履约透明度，降低延迟发货率与客诉率
- 支撑客服团队主动干预和运营团队履约效率优化

#### 3.2.2 核心数据链路

```
ERP/OMS 订单状态变更 ──CDC──▶ Kafka (rt_trade_order_status_changed)
                                │
第三方物流 API ──定时拉取──▶ Kafka (rt_logistics_track_update)
                                │
                          ┌─────▼─────┐
                          │  Flink SQL  │
                          │  状态机计算  │
                          └─────┬─────┘
                                │
                    ┌───────────▼───────────┐
                    ▼                       ▼
              Paimon DWD              Paimon DWS
          订单履约事实表          履约时效汇总表
          物流轨迹事实表          异常事件聚合表
                    │                       │
                    └───────────┬───────────┘
                                │
                         ┌──────▼──────┐
                         │   Doris ADS  │
                         │ 履约驾驶舱主题 │
                         │ 异常预警结果集 │
                         └──────┬──────┘
                                │
                    ┌───────────▼───────────┐
                    ▼                       ▼
              Superset 看板           ADS API / 告警推送
          (履约全程监控大屏)     (异常事件订阅通知)
```

#### 3.2.3 关键指标

| 指标名称    | 定义                   | 粒度       | 更新频率 |
| ------- | -------------------- | -------- | ---- |
| 发货及时率   | 承诺时效内发货订单数 / 应发货订单总数 | 渠道/仓/日   | 实时   |
| 平均履约时长  | 从下单到签收的平均耗时          | 渠道/物流商/日 | 实时   |
| 物流异常率   | 发生异常节点订单数 / 发货订单总数   | 物流商/日    | 实时   |
| 超时未发货单量 | 超过承诺时间仍未发货的订单数       | 渠道/仓/SKU | 分钟级  |
| 揽收超时单量  | 出库后 N 小时内未揽收的订单数     | 物流商/仓    | 分钟级  |
| 运输滞留单量  | 在途超过预期时长的订单数         | 物流线路/日   | 小时级  |
| 签收异常率   | 签收异常（拒收/破损/丢失）占比     | 物流商/周    | 日级   |

#### 3.2.4 交付成果

- 订单履约全链路状态追踪宽表（DWD 层）
- 物流轨迹事件明细表（DWD 层）
- 履约时效汇总表（DWS 层）
- 异常事件聚合与预警表（DWS/ADS 层）
- 履约驾驶舱 Superset 看板（含地图可视化）
- 异常事件 API 接口 + 企业微信/钉钉告警推送

### 3.3 场景 D：门店实时经营效能分析

#### 3.3.1 业务目标

- 按门店、导购、时段、品类维度实时分析客流与成交
- 计算连带率、客单价、人效、坪效、试销率等核心经营指标
- 支持区域督导实时纠偏和门店店长即时调整经营策略
- 为门店排班、陈列调整、促销节奏提供数据依据

#### 3.3.2 核心数据链路

```
POS 销售数据 ──CDC──▶ Kafka (rt_trade_pos_sale)

会员/积分数据 ──CDC──▶ Kafka (rt_crm_member_point)

                       ┌─────▼─────┐

                       │  维表关联    │
                       │  指标计算    │
                       └─────┬─────┘
                             │
                 ┌───────────▼───────────┐
                 ▼                       ▼
           Paimon DWD              Paimon DWS
       门店销售明细事实表       门店经营汇总表
       导购绩效明细事实表      时段效能聚合表
                 │                       │
                 └───────────┬───────────┘
                             │
                      ┌──────▼──────┐
                      │   Doris ADS  │
                      │ 门店经营主题 │
                      │ 导购绩效主题 │
                      │ 区域对比主题 │
                      └──────┬──────┘
                             │
                 ┌───────────▼───────────┐
                 ▼                       ▼
           Superset 看板           ADS API
       (门店经营实时大屏)      (导购排行/区域对比)
```

#### 3.3.3 关键指标

| 指标名称   | 定义                  | 粒度       | 更新频率 |
| ------ | ------------------- | -------- | ---- |

## 5. Fluss 试点专项方案

### 5.1 试点定位与边界

基于一期已完成的三阶段预研结论，二期中 Fluss 的定位如下：

| 维度       | 结论                                |
| -------- | --------------------------------- |
| **定位**   | Paimon 的补充和完善方向，不是替代方案            |
| **范围**   | 1-2 个局部场景的旁路试点                    |
| **原则**   | 可回退、可量化、不影响主链路 SLA                |
| **版本**   | Fluss 0.8.0（与 Flink 1.20.x 已验证兼容） |
| **风险等级** | 低风险（旁路运行，随时可停）                    |

### 5.2 试点场景选择

根据预研最终结论和二期试点建议，二期选择以下两个方向进行试点：

#### 5.2.1 试点场景一：实时库存状态层（最优先）

**为什么选这个场景**

- 与一期"全渠道实时库存可视"场景高度贴合
- `PrimaryKey Table` 在此场景中语义表达最自然
- 结果最容易解释和量化评估
- 最适合做小范围可回退试点

**试点内容**

| 项目   | 内容                                           |
| ---- | -------------------------------------------- |
| 数据范围 | 选取部分 `sku_id + store_id` 组合的库存状态             |
| 存储方式 | 使用 `Fluss PrimaryKey Table` 承接实时库存状态更新       |
| 数据来源 | 复用一期的库存流水 CDC 数据，双写入 Fluss                   |
| 对照基准 | 当前 Paimon latest 镜像表 / DWD 库存事实表 / DWS ATP 表 |
| 评估维度 | 建模复杂度、SQL 复杂度、状态更新清晰度、查询体验、资源开销              |

**技术方案要点**

```sql
-- Fluss PrimaryKey Table 设计（试点）
CREATE TABLE fluss_inventory_state (
    sku_id STRING,
    store_id STRING,
    available_qty DECIMAL(12, 2),
    locked_qty DECIMAL(12, 2),
    in_transit_qty DECIMAL(12, 2),
    atp_qty DECIMAL(12, 2),
    -- 继承湖仓元数据规范
    _event_time TIMESTAMP(3),
    _ingest_time TIMESTAMP(3),
    _partition_dt STRING,
    PRIMARY KEY (sku_id, store_id) NOT ENFORCED
) WITH (
    'connector' = 'fluss',
    'table.type' = 'PRIMARY_KEY'
);
```

**评价指标**

| 评价维度    | 指标项                  | 目标值               |
| ------- | -------------------- | ----------------- |
| 建模复杂度   | DDL 行数、主键设计直观性       | 比 Paimon 方式更简洁或相当 |
| SQL 复杂度 | 写入/读取/Join 的 SQL 复杂度 | 降低或不增加            |
| 资源开销    | Flink State 使用量变化    | 减少或不显著增加          |
| 查询体验    | 状态读取延迟、结果可解释性        | ≤ 当前 Paimon 方式延迟  |
| 运维成本    | 监控、排障、插件依赖复杂度        | 可接受范围内            |

#### 5.2.2 试点场景二：行为流富化层（第二优先）

**为什么选这个场景**

- `Lookup Join` 已在三阶段预研中验证通过
- 与二期"消费者行为洞察"场景天然相关
- 可为后续实时特征层和 AI 前置数据打基础

**试点内容**

| 项目   | 内容                                                   |
| ---- | ---------------------------------------------------- |
| 数据范围 | 选择 1 条核心行为流（如 APP 商品点击流）                             |
| 存储方式 | `Fluss Log Table` 存储行为事件 + `PrimaryKey Table` 存储商品状态 |
| 关联方式 | `Fluss Lookup Join` 实现实时富化                           |
| 对照基准 | 当前 Kafka → Flink Lookup Join(Paimon DIM) 方式          |
| 评估维度 | 是否减少 Flink State 依赖、链路简洁性、维护成本                       |

**技术方案要点**

```sql
-- Fluss Log Table 存储行为事件（试点）
CREATE TABLE fluss_behavior_log (
    user_id STRING,
    sku_id STRING,
    behavior STRING,       -- CLICK/VIEW/ADD_CART/ORDER/PAY
    session_id STRING,
    page_id STRING,
    _event_time TIMESTAMP(3),
    _partition_dt STRING,
    _partition_hh STRING
) WITH (
    'connector' = 'fluss',
    'table.type' = 'LOG'
);

-- Fluss PrimaryKey Table 存储商品状态（供 Lookup Join 使用）
CREATE TABLE fluss_sku_state (
    sku_id STRING,
    sku_name STRING,
    brand_name STRING,
    category_l1 STRING,
    category_l2 STRING,
    season_code STRING,
    price DECIMAL(18, 2),
    _event_time TIMESTAMP(3),
    PRIMARY KEY (sku_id) NOT ENFORCED
) WITH (
    'connector' = 'luss',
    'table.type' = 'PRIMARY_KEY'
);

-- 通过 Flink Lookup Join 进行实时富化
INSERT INTO dwd_app_behavior_enriched_rt
SELECT
    b.user_id,
    b.sku_id,
    b.behavior,
    s.sku_name,
    s.brand_name,
    s.category_l1,
    s.season_code,
    b._event_time,
    b._partition_dt
FROM fluss_behavior_log b
LEFT JOIN fluss_sku_state FOR SYSTEM_TIME AS OF b._event_time AS s
ON b.sku_id = s.sku_id;
```

### 5.3 试点路线

```
第一阶段：单场景试点（第 1-3 周）
  ├── 选择"实时库存状态层"作为首个切入点
  ├── 完成 Fluss 环境搭建与基线数据准备
  ├── 实现 PK Table 写入与读取
  └── 输出：试点方案 + 操作记录 + 初步结论

第二阶段：并行对照（第 4-6 周）
  ├── Fluss 链路与 Paimon 主链路并行运行
  ├── 相同数据双写两路，定期抽样比对
  ├── 收集时延、复杂度、资源、稳定性数据
  └── 输出：对比测试报告 + 量化评估结果

第三阶段：扩大 or 收敛（第 7-8 周）
  ├── 根据评估结论决定：
  │   ✅ 收益明显 → 扩大到第二个场景（行为流富化层）
  │   ⚠️ 收益有限 → 维持现状，记录经验，暂不扩大
  │   ❌ 问题较多 → 回退，总结原因
  └── 输出：最终试点结论 + 二期 Fluss 定位建议
```

### 5.4 回退策略

| 触发条件                      | 回退动作                     | 影响        |
| ------------------------- | ------------------------ | --------- |
| Fluss 写入延迟持续 > 阈值         | 切换回纯 Paimon 方式           | 无影响（双路并行） |
| Fluss Lookup Join 出现数据不一致 | 停止 Fluss 链路，使用 Paimon 基线 | 无影响       |
| Fluss 环境/插件出现稳定性问题        | 停止 Fluss 试点作业            | 仅影响试点本身   |
| 团队认为运维成本过高                | 结束试点，保留经验文档              | 无影响       |

CREATE TABLE ads_xhs_sentiment_analysis (
    note_id VARCHAR(64) NOT NULL COMMENT '笔记ID',
    note_title VARCHAR(500) COMMENT '笔记标题',
    note_content TEXT COMMENT '笔记原始内容',
    sentiment_label VARCHAR(32) COMMENT '情感标签: POSITIVE/NEGATIVE/NEUTRAL',
    sentiment_score DECIMAL(5,4) COMMENT '情感得分: 0~1',
    sentiment_summary VARCHAR(1000) COMMENT 'AI生成的内容摘要',
    keywords VARCHAR(500) COMMENT '提取的关键词',
    hot_level VARCHAR(16) COMMENT '热度级别: HIGH/MEDIUM/LOW',
    related_sku_id VARCHAR(64) COMMENT '关联商品SKUs',
    analysis_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '分析时间',
    partition_dt VARCHAR(8) COMMENT '分区日期',
    PRIMARY KEY (note_id, partition_dt)
)
ENGINE=OLAP
DUPLICATE KEY(note_id)
COMMENT '社媒情感分析AI增强结果'
DISTRIBUTED BY HASH(note_id) BUCKETS 4
PROPERTIES ("replication_num" = "1");

-- 2. 使用 Doris AI Function 做批量情感分析
-- （通过 SQL 调用内置 AI Function 或 UDF）
INSERT INTO ads_xhs_sentiment_analysis
SELECT
    note_id,
    note_title,
    note_content,
    ai_sentiment(note_content) AS sentiment_label,        -- AI Function
    ai_sentiment_score(note_content) AS sentiment_score,  -- AI Function
    ai_summarize(note_content) AS sentiment_summary,       -- AI Function
    ai_extract_keywords(note_content, 5) AS keywords,      -- AI Function
    CASE
        WHEN like_cnt + collect_cnt > 10000 THEN 'HIGH'
        WHEN like_cnt + collect_cnt > 1000 THEN 'MEDIUM'
        ELSE 'LOW'
    END AS hot_level,
    related_sku_id,
    CURRENT_TIMESTAMP AS analysis_time,
    DATE_FORMAT(CURRENT_DATE(), '%Y%m%d') AS partition_dt
FROM ods_xhs_note_stats
WHERE partition_dt = DATE_FORMAT(DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY), '%Y%m%d');

   埋点数据 ODS 层接入表（遵循 02-naming.md 命名规范）
   表名：ods_kafka_behavior_analyze
   写入模式：append-only（纯事件追加）
   分区策略：dt + hh 二级分区（遵循 05-watermark-partition.md）
   ============================================================ */

CREATE TABLE IF NOT EXISTS ods_kafka_behavior_analyze (
    -- 业务核心字段（来自埋点 SDK）
    event_id STRING COMMENT '全局唯一事件ID',
    event_name STRING COMMENT '事件名称(标准化枚举)',
    device_id STRING COMMENT '设备标识',
    user_id STRING COMMENT '用户ID(登录后有)',
    session_id STRING COMMENT '会话ID',
    platform STRING COMMENT '平台: IOS/ANDROID/MINIAPP/WEB',
    page_name STRING COMMENT '页面名称',
    ref_page STRING COMMENT '来源页面',
    target_object_id STRING COMMENT '目标对象ID(SKU/SPU等)',
    target_object_type STRING COMMENT '目标对象类型',
    ext_info STRING COMMENT '扩展信息原始JSON',

    -- 湖仓元数据字段（遵循 04-field.md 规范）
    _event_time TIMESTAMP(3) COMMENT '业务事件时间',
    _ingest_time TIMESTAMP(3) COMMENT '入湖时间',
    _partition_dt STRING COMMENT '日期分区(yyyyMMdd)',
    _partition_hh STRING COMMENT '小时分区(HH)',

    -- Kafka 元数据（Kafka 接入必填）
    _kafka_offset BIGINT COMMENT 'Kafka偏移量',
    _kafka_partition INT COMMENT 'Kafka分区号',

    WATERMARK FOR _event_time AS _event_time - INTERVAL '5' MINUTE
) PARTITIONED BY (_partition_dt, _partition_hh)
WITH (
    'connector' = 'paimon',
    'write-mode' = 'append-only',
    'bucket' = '-1',           /* 动态分桶（遵循 08-paimon.md） */
    'file.format' = 'orc',

### 8.7 二期建模血泪教训规避清单

| 经验教训                        | 典型后果               | 二期规避方式                                    |
| --------------------------- | ------------------ | ----------------------------------------- |
| 埋点 Schema 漂移导致任务频繁修改        | 每次埋点改版都要改 ODS 作业   | 采用"关键字段列化 + raw\_json 存根"模式，延后解析          |
| 用户行为数据量爆炸导致 Paimon 小文件过多    | 查询性能下降，Compact 压力大 | Append 表使用动态分桶(bucket=-1)，合理设置 Compact 策略 |

| 转化漏斗多步骤关联导致 Join 链过长        | 延迟高，排查困难           | 用 DWD 层预聚合每步漏斗数据，ADS 层直接算比率               |

| AI Function 调用耗时影响查询性能      | 看板加载慢              | AI 结果异步预计算写入 ADS 表，看板直接查预计算结果             |

## 12. 风险与控制措施

### 12.1 二期特有风险

| 风险                 | 表现                     | 概率 | 影响 | 应对措施                          |
| ------------------ | ---------------------- | -- | -- | ----------------------------- |
| **埋点数据质量差**        | 埋点不规范、缺字段、采样不一致        | 高  | 高  | 提前制定埋点规范、与前端/增长团队对齐、ODS 层加强容错 |
| **Fluss 试点不如预期**   | 复杂度未降低或有稳定性问题          | 中  | 低  | 已有回退预案、旁路运行不影响主链路             |
| **Doris AI 准确率不足** | 情感分析/分类结果偏差较大          | 中  | 中  | 设置信度阈值、人工审核机制、逐步调优            |
| **行为数据量爆炸**        | 埋点全量接入后存储和计算压力激增       | 高  | 高  | 分级接入（先核心事件再全量）、动态分桶、冷热分层      |
| **物流 API 不稳定**     | 第三方物流接口限频/宕机/格式变化      | 中  | 中  | 重试 + 降级 + 缓存 + 多源备份           |
| **新场景口径反复变更**      | 物流/门店/消费者指标口径不断调整      | 中  | 中  | 建立指标评审机制、版本化管理、变更走流程          |
| **二期范围蔓延**         | 需求不断增加超出原定范围           | 高  | 高  | 严格变更控制、分优先级、超出部分放入三期          |
| **团队能力不足**         | Fluss/Doris AI 新技术学习成本 | 低  | 中  | 预研已有基础、试点先行、文档齐全、逐步上手         |

### 12.2 风险应对通用原则

1. **所有试点必须可回退**：Fluss 和 Doris AI 都设计了明确的回退路径
2. **所有新增必须不影响一期**：新作业/新表/新规则均独立于一期链路
3. **所有变更必须可审计**：Git 管理 + 版本发布 + 变更记录
4. **所有指标必须经业务确认**：新场景指标在开发前冻结口径
5. **所有 AI 结果必须有人审**：高风险 AI 输出不直接驱动业务决策

## 14. 三期展望（前瞻）

二期完成后，三期可以考虑以下方向：

### 14.1 平台级能力升级

- **语义层建设**：面向业务的统一语义层，支持自然语言问数
- **DataHub / Atlas 元数据平台**：统一血缘管理、资产目录、数据地图
- **DolphinScheduler 全面接入**：全链路调度编排与依赖管理

### 14.2 AI 能力深度整合

- **智能问答助手**：基于 Doris AI + LLM 的自然语言数据查询
- **智能归因分析**：自动识别指标波动的根因并给出解释
- **推荐系统集成**：将向量检索能力对接到推荐引擎

### 14.3 Fluss 评估扩大

- 如果二期 Fluss 试点结论积极，三期可评估：
  - 将 Fluss 从"试点"升级为"正式补充组件"
  - 扩大 Fluss 在实时状态层和行为流富化层的应用范围
  - 评估 `Delta Join` / `Partial Update` 在更复杂场景的价值

### 14.4 数据产品化

- **数据服务市场**：将 ADS API 封装为标准数据服务
- **自助分析平台**：基于 Superset + 语义层的自助取数
- **数据驱动的运营工具箱**：将看板、告警、推荐打包为运营产品

## 15. 结论与建议

### 15.1 总结

本二期方案的核心主张可以概括为：

1. **不推翻，只增强**：在一期稳定的主链路上做增量演进，不进行大规模重构
2. **先场景，后技术**：三大新业务场景是二期的主要价值载体，Fluss 和 Doris AI 是增强手段
3. **小步试点，可回退**：所有技术创新都通过旁路试点验证，有明确的量化评价指标和回退预案
4. **治理先行，规范延续**：二期全部开发严格遵循一期已建立的 13 份规范文件体系
5. **从"数据服务"走向"决策增强"**：二期开始引入 AI 能力，但坚持"先辅助后自动"

### 15.2 核心预期成果

二期完成后，服装行业实时湖仓项目将实现：

- **业务场景**：从 2 个扩展到 5 个，覆盖供应链/商品/零售/会员/增长全链条
- **数据接入**：从 5 类扩展到 9+ 类，埋点和日志体系规范化
- **技术能力**：Fluss 有明确试点结论，Doris AI 有落地增强功能
- **平台成熟度**：治理更深、监控更全、文档更完整、运维更智能
- **演进基础**：为三期的语义层、智能问答、平台化打下基础

### 15.3 建议下一步行动

如果本方案通过评审，建议立即启动以下动作：

1. **成立二期项目组**：明确角色分工和 RACI
2. **锁定二期范围**：按优先级确认首批建设的功能和试点场景
3. **启动需求调研**：针对三大新场景做详细的业务需求访谈
4. **准备试点环境**：Fluss 和 Doris AI 的环境提前准备
5. **制定埋点规范**：与前端/增长团队对齐埋点数据模型
