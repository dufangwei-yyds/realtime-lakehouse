# 实时湖仓开发规范

> 项目名称：服装行业实时湖仓构建项目一期  
> 文档定位：一期实时湖仓研发交付与工程实施规范  
> 适用范围：`ODS -> DIM -> DWD -> DWS -> ADS` 分层建设、Flink 作业开发、数据服务开发、发布交付与文档同步

---

## 一、文档目标

本规范用于统一一期实时湖仓的开发标准，目标是：

1. 统一分层建设与对象命名
2. 统一字段、主键、时间、水位、分区等核心技术约束
3. 统一 Flink SQL / DataStream API 开发方式
4. 统一发布、验证、回滚与文档同步要求

---

## 二、开发原则

一期开发必须遵循以下原则：

1. 先统一标准，再做具体实现
2. 先保证主链路正确，再做性能优化
3. 先在 DWS 固化公共指标，再在 ADS 组织消费
4. 文档、实现、验证三者保持一致
5. 所有变更都必须具备回滚点

---

## 三、命名规范

## 3.1 分层对象命名


| 对象类型        | 命名规范                           | 示例                          |
| ----------- | ------------------------------ | --------------------------- |
| CDC 源表      | `cdc_{table}`                  | `cdc_sale_order`            |
| ODS 维度追加表   | `ods_{table}_append`           | `ods_m_product_append`      |
| ODS 维度历史表   | `ods_{table}_history`          | `ods_m_product_history`     |
| ODS 维度最新镜像表 | `ods_{table}_latest`           | `ods_m_product_latest`      |
| ODS 事件解析表   | `ods_{domain}_{event}_analyze` | `ods_app_click_analyze`     |
| DIM 宽表      | `dim_{domain}_wide`            | `dim_sku_wide`              |
| DWD 明细表     | `dwd_{domain}_detail_rt`       | `dwd_order_trade_detail_rt` |
| DWS 汇总表     | `dws_{theme}_{grain}_rt`       | `dws_inventory_atp_rt`      |
| ADS 结果表     | `ads_{theme}_{consumer}`       | `ads_hot_sku_dashboard`     |


## 3.2 Kafka Topic 命名

建议统一：

- `rt_{domain}_{object}_{event}`

示例：

- `rt_trade_order_created`
- `rt_behavior_app_click`
- `rt_social_xhs_note`

## 3.3 作业命名

建议统一：

- `job_{domain}_{purpose}_{env}`

示例：

- `job_product_scd2_prod`
- `job_inventory_dws_sit`

## 四、分层开发规范

## 4.1 ODS 层

必须做到：

- 贴源接入
- 保留关键原始语义
- 补齐元数据字段
- 延迟解析高漂移 JSON
- 不做重度业务聚合

不得做：

- 复杂业务指标固化
- 过早宽表化
- 直接服务消费侧展示

## 4.2 DIM 层

必须做到：

- 统一维度主键与维度口径
- 维度域支持 `append / history / latest`
- 下游高频 Join 的维度退化为宽表

## 4.3 DWD 层

必须做到：

- 以事实过程为核心建模
- 统一主键、粒度和事件时间
- 维度补齐和明细宽表化
- 主键幂等与重复兜底

## 4.4 DWS 层

必须做到：

- 先定义公共粒度，再定义指标
- 核心指标统一在 DWS 固化
- 窗口、水位、延迟策略明确

## 4.5 ADS 层

必须做到：

- 面向 Dashboard / API 分开设计
- 具备解释字段和建议字段
- 不重复定义底层公共指标

## 五、字段规范

## 5.1 通用字段规范


| 字段                  | 含义     |
| ------------------- | ------ |
| `event_time`        | 业务事件时间 |
| `ingest_time`       | 入层时间   |
| `partition_dt`      | 日期分区   |
| `partition_hh`      | 小时分区   |
| `json_parse_status` | 解析状态   |


要求：

- 同语义字段命名尽量统一
- 分层时不得随意改语义
- 新增字段必须同步到数据字典

## 5.2 字段类型规范

- 金额统一使用 `DECIMAL`
- 数量统一使用 `DECIMAL`
- 时间统一使用 `TIMESTAMP(3)`
- 状态字段统一使用 `STRING`
- 主键字段保持跨层一致的类型

---

## 六、主键、粒度与幂等规范

## 6.1 主键规范

- 每张表必须明确主键
- 主键字段必须在文档中可查
- Flink / Paimon 主键表必须与业务主键语义一致

## 6.2 粒度规范

- 每张表必须明确行粒度
- DWD 不允许多业务过程混在一张“万能宽表”
- DWS 不允许粒度不清

## 6.3 幂等规范

- ODS 事实表要利用 Kafka 元数据或业务主键兜底去重
- DWD 仍应对重复膨胀做二次防护
- 结果写入 Doris 时必须关注重复写入风险

---

## 七、事件时间、水位与分区规范

## 7.1 事件时间规范

- 统计与窗口优先使用 `event_time`
- 不得默认使用处理时间代替业务事件时间

## 7.2 水位规范

- 所有用于窗口计算的 DWD 正式表应在建表时定义 `WATERMARK`
- 临时 `_wm` 表只允许作为过渡方案，不建议长期保留

## 7.3 分区规范

建议：

- 行为流、社媒流按 `dt + hh` 二级分区
- 核心事实表至少有 `partition_dt`
- 分区字段必须与事件时间语义一致

---

## 八、Flink SQL / DataStream API 开发规范

## 8.1 Flink SQL 规范

- `CREATE TABLE` 必须补齐必要字段注释和关键属性
- Doris Sink 参数必须显式写清 `fenodes`
- 窗口作业优先使用单语句 `WITH ... SELECT ...`，避免 Planner 冲突
- 所有复杂 `JOIN` 需明确 Join 键与时间语义

## 8.2 DataStream API 规范

适用场景：

- ODS 维度 SCD2 专项处理
- 复杂状态控制
- 复杂定制指标输出

要求：

- 自定义业务指标必须接入监控
- 状态逻辑必须可解释
- 关键 ProcessFunction 逻辑需有注释和专项说明

---

## 九、Doris 开发规范

## 9.1 结果表规范

- ADS 表优先面向消费对象设计
- 高频结果建议考虑物化视图参考项
- 单 `BE` 测试环境下，物化视图建议显式指定 `"replication_num" = "1"`

## 9.2 外部表与物化视图规范

- 外部表优先用于读取 Paimon 分层结果
- 高并发主题可叠加异步物化视图
- 物化视图必须明确刷新策略

---

## 十、质量与验证规范

开发完成后至少需完成：

1. 字段级核查
2. 主键重复检查
3. 时间字段检查
4. 分区检查
5. 抽样结果比对
6. 消费侧验证

如涉及异常场景，还需补充：

- Schema 漂移测试
- 迟到数据测试
- 重复数据测试

---

## 十一、发布与回滚规范

## 11.1 发布前要求

- 变更范围明确
- 测试环境验证通过
- 回滚方案已准备
- 验证 SQL 已准备

## 11.2 发布后要求

- 30 分钟内完成重点验证
- 结果异常必须优先评估回滚
- 发布记录必须归档

## 11.3 回滚要求

- 回滚优先回到上一个稳定版本
- 禁止在故障未定位清楚时叠加发布

---

## 十二、文档同步规范

以下内容发生变化时，必须同步更新文档：

- 新增表或字段
- 主键变更
- 分区变更
- 指标口径变更
- Dashboard / API 返回字段变更

至少同步到：

- 数据字典
- 分层实现方案
- 指标体系清单
- 接入数据源清单
- 验收材料

---

## 十三、代码与资产管理规范

- SQL、配置、脚本统一纳入版本管理
- 发布版本必须可追溯
- 环境差异配置必须显式记录
- 示例代码与生产逻辑要明确区分

---

## 十四、验收建议

开发规范验收建议至少确认：

1. 命名是否统一
2. 分层边界是否清晰
3. 主键 / 粒度 / 水位 / 分区是否明确
4. 发布与回滚要求是否可执行
5. 文档同步机制是否已落实
